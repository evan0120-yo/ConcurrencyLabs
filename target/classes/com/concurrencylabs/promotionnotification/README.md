# 大促通知推播 · promotionnotification

> 一句話定位：大促要對百萬用戶發推播 / 簡訊 / 站內信，行銷通知可延遲可丟，但要跟交易通知隔離。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：任務補償型（大量 fan-out / 非同步）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 大量發送是可非同步、可分批、可重試的背景工作，且要控供應商限流。
- **與 `notificationcenter` 的差別** → 這題聚焦「電商促銷」的行銷群發；`notificationcenter` 是平台級多渠道通知中樞。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
大促要對百萬用戶群發
   [活動] ──> [通知] 分渠道大量發送（推播 / 簡訊 / 站內信）
   ├─ 行銷通知 → 可延遲、可丟、可分批
   └─ 但要跟「交易通知」隔離（訂單 / 付款通知優先）

────────── 一次群發百萬則，別把交易通知塞住 ──────────
```

### 分段解說

- 行銷群發量體巨大，但每一則的「時效」與「必達」要求低。
- 危險在於：行銷洪流若跟交易通知共用管道，會把「付款成功」這種重要通知擠掉。
- 供應商（簡訊 / 推播商）本身有速率上限，得配合。

### 痛點分析

- **能不能延遲 / 丟？** 行銷通知可以。
- **要不要隔離？** 要，交易通知優先，不能被行銷淹沒。
- **外部限流** → 供應商有速率限制，發送要配速。
- **真正的痛**：巨量低優先的行銷通知，不能拖垮少量高優先的交易通知，隔離與配速是關鍵。
