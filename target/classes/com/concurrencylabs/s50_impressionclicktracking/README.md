# 曝光 / 點擊事件追蹤 · s50_impressionclicktracking

> 一句話定位：海量的曝光與點擊事件持續回報，收得下、可抽樣、可延遲入湖，少數關鍵事件不能丟。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：AdTech / 資料流
- 壓力模型：寫爆型 / ingestion（核心）
- 數據角色：ingestion 寫爆 →（下游）分析 / 計費

---

## Block 2：為什麼被列上這些

- **寫爆型 / ingestion** → 曝光 / 點擊是持續的海量寫入事件流。
- **數據角色** → 這些事件是計費、報表、優化的原料，屬資料管線起點。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
每個廣告被看到 / 被點
   曝光 曝光 點擊 曝光 …（每秒數十萬筆）
   │
   ▼
[事件追蹤] 全部先收下
   ├─ 一般曝光 → 量大、可抽樣、可延遲入湖
   └─ 計費相關點擊 → 關鍵，不能丟（要準確計費）

────────── 大促 / 熱門檔期，事件量翻好幾倍 ──────────
```

### 分段解說

- 曝光量極大但單筆價值低，可抽樣。
- 但「牽涉計費的點擊」是關鍵資料，不能丟。
- 收集端要先扛住寫爆，加工留給下游。

### 痛點分析

- **能不能丟 / 抽樣？** 一般曝光可以，計費事件不行。
- **讀爆還寫爆？** 純寫爆。
- **能不能延遲？** 入湖分析可延遲，計費要準。
- **真正的痛**：跟 IoT 心跳同構——海量低價值事件裡混著少數「計費關鍵」事件，要分流對待，不能一視同仁。
