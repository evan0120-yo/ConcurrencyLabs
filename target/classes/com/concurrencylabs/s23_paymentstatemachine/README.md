# 交易狀態機系統 · s23_paymentstatemachine

> 一句話定位：一筆交易在建立→付款→成功/失敗/退款之間流轉，靠冪等 key 與狀態機保證不錯亂。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：支付 / 金融
- 壓力模型：錢包帳本型（真錢）、任務補償型（callback / 查單 / 補償）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 交易關乎真錢，每個狀態轉移都要正確、可追溯。
- **任務補償型** → 付款結果靠第三方 callback、主動查單、逾時補償收斂，天生要冪等。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一筆交易的一生
   建立訂單 ──> 待付款 ──> (付款中) ──> 成功 / 失敗
                                   └─> 退款

   收斂靠三種力量：
   ├─ 第三方 callback（可能遲到 / 重送 / 遺失）
   ├─ 主動查單（我去問第三方現在到底怎樣）
   └─ 逾時補償（太久沒結果就走補救）

────────── 同一筆交易的多個訊息可能亂序到達 ──────────
```

### 分段解說

- 這題的本體是「狀態機」：合法的轉移只有幾條，其他都要擋。
- 各種訊息（callback / 查單 / 逾時）可能亂序、重複，狀態機要能容忍。
- 核心轉移要最小、最穩，周邊功能全部解耦出去。

### 痛點分析

- **能不能重複 / 亂序？** 訊息會，但狀態不能被帶錯。
- **一致性** → 涉及真錢，狀態必須正確。
- **收斂機制** → callback 不可靠，要有查單與逾時兜底。
- **真正的痛**：用一個穩固的狀態機，去消化「不可靠、會亂序重複」的外部訊息，且全程不錯帳。
