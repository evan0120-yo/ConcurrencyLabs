# Checkout 下單鏈路 · s16_checkout

> 一句話定位：按下結帳一次要串算價、扣庫存、建單、付款、通知，核心不能錯、周邊可降級。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：搶資源型（扣庫存）、任務補償型（周邊非同步 / 補償）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 下單要扣庫存，直接踩到 `s15_inventorydeduction` 的爭搶。
- **任務補償型** → 積分、推薦、通知等周邊步驟應非同步、可失敗補償，不該卡住核心。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
按「結帳」，一次要串多個步驟
   使用者 ──> [Checkout]
      │
      ├─ 核心：算價 + 扣庫存 + 建單 + 付款   ← 錯一個都不行
      └─ 周邊：積分 + 推薦 + 通知           ← 可慢、可降級

────────── 大促時跟秒殺連動一起爆 ──────────

使用者想要：下單順、別轉圈圈
營運想要：核心一定成，周邊掛了不能擋下單
```

### 分段解說

- Checkout 是「編排者」，把好幾個系統串成一條龍。
- 關鍵是把「核心」跟「周邊」切開：周邊掛掉不能連累下單。
- 任一步失敗，要嘛回滾、要嘛補償，不能留下半殘訂單。

### 痛點分析

- **核心 vs 周邊** → 必須切開，周邊降級不影響成單。
- **一致性** → 錢與庫存與訂單要對得起來。
- **平均還是尖峰？** 跟大促 / 秒殺連動，會一起爆。
- **真正的痛**：一條鏈路串多個系統，如何在任一環節出錯時仍保住「核心成單」且不留髒資料。
