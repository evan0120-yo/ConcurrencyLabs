# 地理位置查詢 / Nearby Search · s71_nearbysearch

> 一句話定位：查「我附近的 X」（店家 / 司機 / 人），熱門地區查詢集中，可用近似換速度。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：地理 / 派單
- 壓力模型：讀爆型（核心）、熱點型（熱門地區）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 「查附近」是高頻讀，且要做地理範圍計算。
- **熱點型** → 市中心 / 熱門商圈的查詢遠多於偏遠地區。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者：「我附近 3 公里有哪些餐廳 / 司機？」
   │
   ▼
[Nearby Search] 用地理索引找範圍內的目標 ──> 依距離排序
   ├─ 熱門地區（市中心）→ 查詢集中 → 可快取熱門格
   └─ 目標會移動（司機）→ 位置一直更新

────────── 尖峰時段、熱門商圈，大家都在查附近 ──────────

使用者想要：夠快、夠準（附近的沒漏、太遠的別出現）
```

### 分段解說

- 地理查詢要把「二維位置」變成可快速查的索引（如網格 / geohash）。
- 熱門地區查詢集中，可快取那些格子的結果。
- 若目標會移動（司機），位置更新頻繁，是動態的。

### 痛點分析

- **讀爆還寫爆？** 讀爆（查附近），司機類另有位置寫入。
- **有沒有熱點？** 熱門商圈是熱點。
- **能不能近似？** 「附近」本就模糊，可近似換速度。
- **真正的痛**：把地理範圍查詢做快，且熱門地區的查詢集中要靠分格快取來扛。
