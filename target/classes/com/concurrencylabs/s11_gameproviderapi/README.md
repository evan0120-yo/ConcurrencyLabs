# 第三方遊戲商 API 串接 · s11_gameproviderapi

> 一句話定位：平台呼叫外部遊戲商 API，怕它慢、怕它掛，一家壞掉不能拖垮整個平台。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：平台防爆型（隔離 / 熔斷）、任務補償型（重試 / fallback）

---

## Block 2：為什麼被列上這些

- **平台防爆型** → 命脈被「別人家的系統」掐住，重點是把風險隔離、不讓它蔓延。
- **任務補償型** → 呼叫失敗要重試、要有 fallback，屬於整合可靠性問題。
- **博弈 domain** → 聚合多家遊戲商是博弈平台的常態。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
玩家玩某遊戲商的遊戲
   │
   ▼
[平台] ──呼叫──> [外部遊戲商 API]
   ├─ (回應正常) ──> 繼續
   ├─ (很慢)     ──> 拖住平台資源
   └─ (掛掉)     ──> 若不隔離，會拖垮整個平台

────────── 平台命脈被別人家的系統掐住 ──────────

營運想要：某遊戲商掛掉，只影響那一家，不能全站陪葬
```

### 分段解說

- 外部系統不可控：它的慢與掛你都無法預防。
- 所以策略是「隔離」：把每家遊戲商的風險關在自己的圈裡。
- 這題偏「防禦工程」，業務邏輯反而簡單。

### 痛點分析

- **不可控依賴** → 對方慢 / 掛你無法避免，只能防。
- **隔離** → 一家壞掉不能拖垮全站（資源不要共用到死）。
- **重試 / fallback** → 失敗要有退路，但重試不能雪崩。
- **真正的痛**：如何在「依賴一堆外部系統」的前提下，讓任何一家的故障都被關在局部。
