# Marketplace 商家庫存同步 · merchantinventorysync

> 一句話定位：平台要同步成千上萬商家的商品 / 庫存變更，大商家不能拖垮小商家的同步。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：社群 / Feed / 搜尋 / Marketplace
- 壓力模型：任務補償型（非同步同步 + 失敗補償）、多租戶型（商家隔離）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 商家庫存 / 商品變更用非同步同步，失敗要能補償重試。
- **多租戶型** → 每個商家是一個租戶，大商家不能拖垮小商家。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
商家在自己系統改了價格 / 庫存
   商家 A ─┐
   商家 B ─┼──> [同步] 非同步把變更拉進平台
   大商家 C┘        │
                    ├─ (成功) ──> 平台商品更新
                    └─ (失敗) ──> 補償重試

────────── 大商家一次改 10 萬個 SKU ──────────

小商家想要：我改一個庫存，別因為大商家在洗全站就卡住我
```

### 分段解說

- 同步是非同步的：商家改完，平台在背後拉。
- 大商家的批量變更會佔滿同步通道，拖累小商家。
- 失敗要能補償，不能因為某次同步失敗就資料錯亂。

### 痛點分析

- **能不能非同步？** 可以，同步天生是背景工作。
- **要不要隔離？** 要，大商家獨立通道，別拖垮小商家。
- **失敗補償** → 同步失敗要能重試補平。
- **真正的痛**：跟多租戶「吵鬧鄰居」同構——大商家的批量同步不能佔滿通道，害小商家的變更遲遲同步不上。
