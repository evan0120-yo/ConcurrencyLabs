# Feature Store / Online ML Serving · s75_featurestoreonlineserving

> 一句話定位：線上模型推論時即時取用戶 / 內容特徵，要極低延遲、特徵缺失要 fallback、線上限時回應。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：ML / 資料平台
- 壓力模型：讀爆型（核心）、低延遲型（線上推論限時）
- 數據角色：特徵讀（online / nearline / offline 分層）

---

## Block 2：為什麼被列上這些

- **讀爆型 + 低延遲** → 每次推論都要即時取一批特徵，高頻且有時限。
- **特徵讀角色** → 特徵分 online / nearline / offline 三層，取用要就快取層拿。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者一個請求要模型即時判斷（推薦 / 風控 / 排序）
   │
   ▼
[Feature Store] 即時取這個 user / item 的特徵
   ├─ online 特徵（即時行為）→ 要最新、要快
   ├─ nearline / offline 特徵（近期 / 歷史統計）→ 預先算好備著
   └─ 某特徵剛好缺 → 不能卡住 → 用預設值 fallback，模型照樣回應

────────── 推論有硬時限，特徵拿太慢 = 整個推論超時 ──────────

工程想要：特徵取用極快、缺了也要有退路、線上一定要在時限內回
```

### 分段解說

- 模型推論的前置是「取特徵」，取得慢整個推論就超時。
- 特徵分層：即時的要快取、歷史的預先算好。
- 特徵可能缺（新用戶 / 資料延遲），要能 fallback 不卡死。

### 痛點分析

- **讀爆還寫爆？** 讀爆（每次推論取一批）。
- **延遲要求** → 線上推論有硬時限。
- **缺特徵怎辦？** 要 fallback，不能因缺一個特徵就掛。
- **真正的痛**：線上推論的硬時限逼特徵取用必須極快，且要能容忍「特徵缺失」用 fallback 續命，不讓單一特徵拖垮整個回應。
