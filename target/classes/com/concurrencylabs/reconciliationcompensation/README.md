# 對帳與補償平台 · reconciliationcompensation

> 一句話定位：把自家帳與第三方帳逐筆比對，找出差異單，走補償流程補平，主交易不等它。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：支付 / 金融
- 壓力模型：任務補償型（核心）
- 數據角色：批次比對 / 分析讀

---

## Block 2：為什麼被列上這些

- **任務補償型** → 對帳是批次拉檔、找差異、走補償狀態機的典型後台工作。
- **批次分析讀** → 需要大量讀取雙方帳目做比對。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
每天 / 每小時
   [我方帳] ──┐
              ├──> [對帳] 逐筆比對
   [第三方帳]─┘        │
                       ├─ (一致) ──> 打勾
                       └─ (差異) ──> 差異單 ──> 補償流程補平

────────── 主交易絕不等對帳，對帳在背後慢慢跑 ──────────

營運想要：帳最終要平，但別讓對帳拖慢即時交易
```

### 分段解說

- 對帳是「事後」的：交易當下不等它。
- 找出差異只是第一步，補償把差異收斂才是目的。
- 補償本身是個狀態機（待處理 → 處理中 → 已補平 / 需人工）。

### 痛點分析

- **能不能延遲？** 可以，對帳天生是事後批次。
- **要不要隔離？** 要，主交易不能被對帳拖累。
- **一致性** → 最終要對得平，差異要有交代。
- **真正的痛**：在雙方帳都可能出錯 / 遲到的情況下，穩定地找出差異並把它補平，且完全不影響即時交易。
