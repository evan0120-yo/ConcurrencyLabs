# 第三方 API 重試補償 · s35_thirdpartyretrycompensation

> 一句話定位：呼叫外部 API 失敗時安全重試，用退避與死信避免雪崩式重送把對方（跟自己）打垮。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：任務 / 排程 / 第三方整合
- 壓力模型：任務補償型（核心）、平台防爆型（防重送雪崩）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 失敗重試、退避、死信、補償是它的本體。
- **平台防爆型** → 重試若不節制會造成雪崩式重送，反而把系統打垮。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
呼叫外部 API 失敗了
   │
   ▼
[重試補償]
   ├─ 立刻重試？ ──> 錯！對方正在掛，你狂送只會更糟
   ├─ 指數退避重試 ──> 等 1s、2s、4s… 再試
   └─ 試到上限仍失敗 ──> 進死信，之後人工 / 補償處理

────────── 對方大當機，你有一萬筆要重送 ──────────

危險：一萬筆同時重送 = 雪崩，把對方跟自己一起弄死
```

### 分段解說

- 重試是好意，但「馬上、全部一起重試」是災難。
- 退避是為了給對方喘息，也保護自己。
- 試不成的要落到死信，不能無限重送。

### 痛點分析

- **失敗怎麼重試？** 要退避，不能狂送。
- **會不會雪崩？** 會，這是最大風險。
- **兜底** → 死信 + 人工 / 補償。
- **真正的痛**：重試本意是提高成功率，但沒節制的重試會演變成把對方與自己一起打垮的雪崩。
