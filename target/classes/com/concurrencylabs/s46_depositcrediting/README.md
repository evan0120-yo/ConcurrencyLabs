# 充值入帳系統 · s46_depositcrediting

> 一句話定位：掃鏈上到帳事件，確認後把對應資產入帳到使用者，入帳必須冪等、不可重複加。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：交易所 / Web3
- 壓力模型：錢包帳本型（入帳真資產）、任務補償型（掃鏈 / 確認 / 冪等）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 入帳是往使用者帳戶加真資產，錯 / 重複都是資損。
- **任務補償型** → 靠掃鏈事件觸發，事件會重放，需冪等與確認機制。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者往充值地址轉入 1 BTC
   │
   ▼
[掃鏈] 偵測到這筆轉入 ──> 等足夠確認數 ──> 入帳
   ├─ (確認足夠) ──> 帳戶 +1 BTC（同一筆只能加一次）
   └─ (確認不足) ──> 繼續等，先不入帳

────────── 掃鏈事件可能重放、鏈可能分叉回滾 ──────────

使用者想要：到帳快
平台想要：不能因為分叉 / 重放而重複入帳
```

### 分段解說

- 入帳前要等「足夠確認數」，避免鏈回滾造成假到帳。
- 掃鏈事件會重放，同一筆轉入不能被入帳兩次。
- 這是真資產，錯帳直接等於虧錢。

### 痛點分析

- **能不能重複？** 絕不能，冪等是命脈。
- **確認機制** → 要等足夠確認避免回滾。
- **能不能延遲？** 到帳可稍延遲（等確認），但不可錯。
- **真正的痛**：在「事件會重放、鏈會回滾」的不確定環境下，保證每筆充值只入帳一次、且金額正確。
