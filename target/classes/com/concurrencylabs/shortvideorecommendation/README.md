# 短影音首頁推薦流 · shortvideorecommendation

> 一句話定位：使用者一開 App 就要有一串「可能想看」的影片，推薦要即時給、但重運算得預算好。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：社群 / Feed / 搜尋 / Marketplace
- 壓力模型：讀爆型（每次刷都要一批推薦）
- 數據角色：分析 / 特徵讀（召回 → 排序）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 每次下滑都要生成 / 取一批推薦，讀壓力極高。
- **特徵讀角色** → 推薦要讀使用者與內容特徵，屬線上特徵存取。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者一開 App / 一直下滑
   │
   ▼
[推薦流] 要即時給下一批「可能想看」的影片
   ├─ 召回：從海量影片粗選幾百支
   ├─ 粗排：快速篩到幾十支
   └─ 精排：細算排出最終順序
   *重運算不能臨場全做，否則刷不動*

────────── 每個使用者、每次下滑都要一批 ──────────
```

### 分段解說

- 推薦是「漏斗」：召回 → 粗排 → 精排，一層層縮小。
- 越後面越精細越貴，不能全部臨場算。
- 常把離線 / 近線算好的特徵備著，線上只做輕量排序。

### 痛點分析

- **讀爆還寫爆？** 讀爆（每次刷都要）。
- **能不能預算？** 重運算要離線 / 近線先算好。
- **延遲要求** → 要即時給，慢了使用者就滑走。
- **真正的痛**：推薦品質要靠重運算，但刷的頻率極高，逼你把重活挪到離線，線上只留輕量排序。
