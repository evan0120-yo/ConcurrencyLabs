# 背景任務平台 · s33_backgroundjob

> 一句話定位：把耗時工作丟到背景非同步跑，worker 可水平擴展，支援限流、暫停、重跑。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：任務 / 排程 / 第三方整合
- 壓力模型：任務補償型（核心）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 這是所有「非同步、可重試、可調度」工作的通用底座，是任務補償型的原型。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
某操作會觸發耗時工作（寄信 / 產檔 / 同步）
   請求 ──> 丟一個 job ──> [背景任務平台] ──> worker 撿起來跑
                                              ├─ (成功) ──> 標記完成
                                              └─ (失敗) ──> 重試 / 進死信

────────── 任務突然湧入一大批 ──────────

營運想要：能水平加 worker、能限流、能暫停某類任務、能重跑
```

### 分段解說

- 核心價值是「解耦」：請求不等耗時工作做完。
- worker 要能隨負載水平擴展。
- 平台要可控：限流、暫停、重跑是營運的基本需求。

### 痛點分析

- **能不能非同步？** 這題本身就是為非同步而生。
- **失敗怎麼辦？** 要重試、要有死信兜底。
- **可控性** → 限流 / 暫停 / 重跑是必要能力。
- **真正的痛**：作為通用底座，要在「大量任務湧入」時仍可控（不失控雪崩）、可觀測、可人工介入。
