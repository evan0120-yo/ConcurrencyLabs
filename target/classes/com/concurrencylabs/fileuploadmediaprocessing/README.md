# 檔案上傳 / 圖片影片處理 · fileuploadmediaprocessing

> 一句話定位：使用者上傳大檔，後端接收後要掃毒、縮圖、轉碼，重活分級非同步處理。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：企業資料
- 壓力模型：任務補償型（轉碼 pipeline）、寫爆型（大檔傳輸）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 掃毒 / 縮圖 / 轉碼是耗時的後處理，應非同步、可重試、可分級。
- **寫爆型** → 大檔上傳本身是重 IO / 頻寬壓力。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者上傳一支 2GB 影片
   │
   ├─ 走後端中轉？ ──> 佔頻寬、佔連線、易失敗
   └─ 直傳儲存 + 後端拿到通知
             │
             ▼
   [後處理 pipeline] 掃毒 ──> 縮圖 ──> 多解析度轉碼
       ├─ (小圖) ──> 快、優先
       └─ (4K 轉碼) ──> 慢、可排隊分級

────────── 一堆人同時上傳大檔 ──────────
```

### 分段解說

- 上傳與後處理是兩件事：先安全收下，再慢慢加工。
- 後處理有輕有重（縮圖 vs 4K 轉碼），要分級排隊。
- 大檔上傳容易斷，要能續傳、要保護頻寬。

### 痛點分析

- **能不能非同步？** 後處理可以，使用者不必等轉碼完。
- **輕重分級** → 縮圖優先、重轉碼排隊。
- **大檔風險** → 上傳易失敗，要能分段 / 續傳。
- **真正的痛**：把「即時要的（縮圖）」跟「慢工（轉碼）」分級，別讓重活塞住輕活，也別讓大檔傳輸拖垮系統。
