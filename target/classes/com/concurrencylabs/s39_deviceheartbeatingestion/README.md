# IoT 裝置心跳與資料上報 · s39_deviceheartbeatingestion

> 一句話定位：成千上萬台裝置持續往後端回報「我還活著 + 我的數據」，後端要全收下、不塞爆、還要即時抓到異常。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：IoT
- 壓力模型：寫爆型 / ingestion（收洪水資料）
- 數據角色：ingestion 寫爆 →（下游）冷熱分層 / 入湖

---

## Block 2：為什麼被列上這些

- **IoT domain** → 上報方是「裝置」不是「人」，量體由裝置數 × 回報頻率決定，跟人為操作無關。
- **寫爆型 / ingestion** → 流量幾乎全是寫入、讀取極少；壓力來自「同時湧入的寫」，不是熱 key 爭搶。
- **數據角色（ingestion）** → 這是資料進系統的第一站，它決定下游 metrics、資料湖能拿到什麼。
- **冷熱分層（延伸）** → 心跳資料絕大多數是「正常、無聊」的，量大但單筆價值低，天生要分冷熱。

> 這就是你說的「同時符合數據上報、也符合 IoT」的典型：domain 掛在 IoT，但壓力模型跟 `s61_logingestion`、`s54_eventpipeline` 是同一家人。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
【裝置視角 — 平常】
溫度感測器 / 手環 / 車機 …（想像 50 萬台）
   │  每 10 秒回報：我在線、溫度 26.4、電量 80%
   ▼
[後端上報入口] ──(全部收下)──> 【平常：一條平穩的資料細流】

────────────── 問題出在尖峰與異常 ──────────────

【裝置視角 — 尖峰】
整點所有裝置同時醒來回報 / 斷網恢復後補傳堆積
   │
   ├─ 裝置想要：傳出去就好，不在乎後端何時處理（射後不理）
   ├─ 營運想要：某台裝置「掉線 / 數值異常」要「馬上」知道
   └─ 系統面對：瞬間 50 萬筆同時砸進來
```

### 分段解說

- 平常：資料是「一條穩定細流」，每台裝置乖乖每 10 秒一筆。
- 尖峰：整點時間對齊、斷網重連補傳，會讓平穩細流瞬間變成洪峰。
- 關鍵不對稱：裝置端是「射後不理」，營運端卻要「即時抓到異常」——大量正常資料裡的少數異常才是重點。

### 痛點分析

- **能不能丟？** 正常心跳丟幾筆沒差；但「掉線 / 異常」事件不能丟。
- **能不能延遲？** 正常資料入庫延遲幾分鐘可接受；異常要秒級。
- **平均還是尖峰？** 平均不高，但整點對齊 + 斷網補傳造成尖峰洪水。
- **讀爆還寫爆？** 幾乎純寫爆，讀很少。
- **真正的痛**：把「大量無聊資料」跟「少數關鍵異常」混在同一條洪流裡；若一視同仁，異常會被淹沒或延遲，等於裝置壞了沒人知道。
