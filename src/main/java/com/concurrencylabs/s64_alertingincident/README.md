# Alerting / Incident 通知 · s64_alertingincident

> 一句話定位：依規則偵測異常發告警，大事故時要去重 / 聚合 / 抑制，避免通知風暴淹沒值班人。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Observability
- 壓力模型：任務補償型（規則評估 + 通知）、平台防爆型（防通知風暴）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 規則評估、告警派送、通知重送是持續的後台工作。
- **平台防爆型** → 大事故會引發告警風暴，要去重 / 聚合 / 抑制，否則反而癱瘓應變。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
指標 / 事件觸發告警規則
   │
   ▼
[告警] 該通知誰？
   ├─ 正常：一條異常 → 一則通知
   └─ 大事故：一個機房掛 → 5000 條告警同時炸
        ├─ 若原樣全發 ──> 值班人手機被轟炸，反而看不到重點
        └─ 去重 + 聚合 + 抑制 ──> 收斂成「機房 X 故障」一則

────────── 半夜大當機，告警瘋狂湧出 ──────────

值班人想要：告訴我「根因」，別把 5000 條症狀全丟我臉上
```

### 分段解說

- 告警的價值在「可行動」，不在「量多」。
- 大事故時一個根因會炸出海量子告警，全發等於沒發。
- 去重、聚合、抑制把噪音收斂成少數有意義的告警。

### 痛點分析

- **會不會風暴？** 會，大事故必伴告警風暴。
- **能不能聚合 / 抑制？** 要，把症狀收斂成根因。
- **關鍵不能漏** → 收斂但不能把真正重要的壓掉。
- **真正的痛**：最需要告警的大事故時刻，也是告警最容易變成「風暴淹沒重點」的時刻，去重聚合抑制是關鍵。
