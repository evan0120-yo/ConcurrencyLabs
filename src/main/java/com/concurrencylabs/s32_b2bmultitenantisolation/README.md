# B2B 多租戶 SaaS 隔離 · s32_b2bmultitenantisolation

> 一句話定位：一套系統服務多家企業客戶，資料與資源要隔離，不能讓某個大客戶拖垮其他人。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：會員 / Gateway / 多租戶
- 壓力模型：多租戶型（核心）、平台防爆型（hot tenant 限流）

---

## Block 2：為什麼被列上這些

- **多租戶型** → 一套系統多租戶共用，隔離是這題的本質。
- **平台防爆型** → 大客戶（hot tenant）流量暴衝時要限流，避免鄰居受害。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一套系統服務很多家企業
   A 公司 ─┐
   B 公司 ─┼──> [同一套 SaaS]
   C 公司 ─┘        │
                    ├─ 資料隔離：A 絕不能看到 B 的資料
                    └─ 資源隔離：C 流量暴衝，不能拖垮 A / B

────────── 某大客戶搞活動，流量是別人 100 倍 ──────────

小客戶想要：別因為隔壁大戶忙就被拖累（吵鬧鄰居問題）
```

### 分段解說

- 多租戶的兩條底線：資料不能串、資源不能互相拖累。
- 「吵鬧鄰居」是核心痛點：一個大戶不能吃光共用資源。
- 大到一定程度的租戶，往往要獨立資源池 / 獨立庫。

### 痛點分析

- **資料隔離** → 跨租戶絕不可見，是安全紅線。
- **資源隔離** → hot tenant 要限流 / quota，保護鄰居。
- **分級** → 大租戶可能要獨立 DB / cell。
- **真正的痛**：共用一套系統省成本，卻要同時保證「資料不串」與「大戶不吵到小戶」。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s32 資料隔離是紅線、資源隔離防吵鬧鄰居,分級隔離 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：共用一套系統,每筆資料帶 tenantId 邏輯隔離
├─ 大方塊：tenantId 資料隔離(強制過濾) · 共用資源池
└─ 撐不住的訊號 → 某大戶流量暴衝拖垮鄰居(吵鬧鄰居)

【P2 · per-tenant quota + hot tenant 限流】約 10,000 QPS
├─ 定義：每租戶 quota/限流,hot tenant 暴衝時限流降級保鄰居
├─ 大方塊（綁一起）
│   ├─ [Redis] per-tenant quota/限流
│   ├─ hot tenant 偵測 + 限流/降級
│   └─ 資料隔離強制(跨租戶一律拒)
├─ 為何綁：資源隔離 = quota + hot 限流 + 資料隔離紅線,一起做
└─ 撐不住的訊號 → 超大租戶共用池已裝不下

【P3 · 大租戶獨立資源池 / 獨立庫 cell】約 50,000+ QPS
├─ 定義：大租戶分級到獨立資源池/獨立 DB/cell,小戶留共用池
├─ 大方塊：租戶分級路由 · 大戶獨立 cell/庫 · 共用池保小戶
└─ 為何綁：分級隔離 = 分級路由 + 獨立部署,一起做
```
