# 大型 Notification Center · s73_notificationcenter

> 一句話定位：平台級多渠道通知中樞，交易通知優先必達，行銷通知可延遲可丟，供應商要配速。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：通知（平台級）
- 壓力模型：任務補償型（核心）、平台防爆型（防通知風暴 / 供應商限流）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 大量 fan-out、分渠道、重試投遞是它的本體。
- **平台防爆型** → 供應商有速率上限，且要防行銷洪流擠掉交易通知。
- **與 `s22_promotionnotification` 的差別** → 那個聚焦電商促銷群發；這個是「全平台各業務共用」的通知中樞。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
全平台各種通知都匯到這裡
   付款成功 / 提幣完成 / 促銷 / 系統公告 ──> [Notification Center]
   │
   ├─ 交易通知（付款 / 安全）→ 高優先、必達、要快
   └─ 行銷通知（促銷 / 推薦）→ 低優先、可延遲、可丟

   分渠道送：App 推播 / 簡訊 / Email / 站內信
   └─ 每個渠道供應商都有速率上限

────────── 大促群發百萬則，同時又有付款通知要發 ──────────

平台想要：付款通知絕不能被行銷洪流塞住
```

### 分段解說

- 這是「全平台通知的匯流排」，各業務都往這送。
- 核心是「優先級」：交易必達、行銷可丟。
- 各渠道供應商有速率限制，要配速、要重試。

### 痛點分析

- **能不能延遲 / 丟？** 行銷可以，交易不行。
- **要不要隔離 / 分優先級？** 要，交易與行銷分道。
- **外部限流** → 供應商速率上限要遵守。
- **真正的痛**：作為全平台匯流排，要用「優先級 + 渠道配速」保證少量高優先的交易通知，永遠不被巨量低優先的行銷通知淹沒。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s73 平台匯流排、優先級隔離+渠道配速,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：各業務通知匯入,分渠道送
├─ 大方塊：通知匯入 · 分渠道投遞 · 重試
└─ 撐不住的訊號 → 大促群發百萬則、付款通知被行銷洪流塞住

【P2 · 優先級隔離 + 渠道配速 + 去重重試】約 10,000 QPS
├─ 定義：交易/行銷優先級分道,供應商令牌桶配速,去重重試
├─ 大方塊（綁一起）
│   ├─ 交易高優先必達 / 行銷低優先可丟(分佇列)
│   ├─ 供應商速率配速
│   └─ 去重 + 分級重試
├─ 為何綁：交易不被行銷淹 = 優先級隔離 + 配速 + 去重,一起做
└─ 撐不住的訊號 → 通知量再放大、多渠道多供應商並行

【P3 · 任務分片 + 渠道獨立鏈路 + 交易/行銷資源隔離】約 50,000+ QPS
├─ 定義：投遞任務分片並行,各渠道獨立鏈路,交易/行銷資源徹底隔離
├─ 大方塊：任務分片 · 渠道獨立鏈路 · 交易/行銷資源隔離
└─ 為何綁：規模化匯流排 = 分片 + 渠道獨立 + 資源隔離,一起做
```
