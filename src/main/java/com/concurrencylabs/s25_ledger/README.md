# Ledger 帳本系統 · s25_ledger

> 一句話定位：所有資金流動的權威帳本，只增不改，餘額是結果、流水是真相，不可被非核心流量拖垮。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：支付 / 金融
- 壓力模型：錢包帳本型（核心）、熱點型（熱帳戶）
- 數據角色：append-only 歷史（對帳唯一真相）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 帳本是整個金流的正確性根基，要求冪等、可追溯。
- **熱點型** → 商戶主帳戶 / 平台帳戶會被高頻記帳，變成熱 accountId。
- **append-only 歷史** → 流水只增不改，是對帳與稽核的唯一依據。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
每一筆資金流動都記進帳本
   付款 / 退款 / 手續費 / 結算 ──> [Ledger] append 一筆流水
   ▼
   餘額 = 流水累加的結果（可快照）
   流水 = 不可竄改的歷史真相

────────── 月底對帳 / 稽核 / 糾紛 ──────────

問：「這個餘額是怎麼來的？」
答：把流水攤開，每一筆都對得起來
```

### 分段解說

- 帳本跟一般業務資料不同：它不能被改、只能被追加。
- 餘額是「算出來的結果」，真正的權威是流水。
- 它是被許多系統依賴的地基，不能被行銷 / 報表這類非核心流量拖垮。

### 痛點分析

- **能不能改 / 丟？** 不能改、不能丟、不能重複記。
- **有沒有熱點？** 平台 / 商戶主帳戶是熱點。
- **要不要隔離？** 要，帳本不可被非核心流量拖垮。
- **真正的痛**：作為全金流的正確性根基，要同時滿足不可竄改、可對帳、又扛得住熱帳戶的高頻記帳。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s25 是全金流正確性根基,不可竄改+可對帳+扛熱帳戶,壓力最高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：append-only 流水 + 餘額,bizNo 冪等記帳
├─ 大方塊：append 流水(只增不改) · 餘額同交易更新 · bizNo 去重
└─ 撐不住的訊號 → 平台/商戶主帳戶熱點、記帳與對帳讀互搶

【P2 · 餘額快照 + 記帳/對帳讀分離】約 10,000 QPS
├─ 定義：餘額改快照由流水推導,對帳/報表讀走副本與記帳隔離
├─ 大方塊（綁一起）
│   ├─ 餘額快照(免每筆算全量)
│   ├─ 記帳(寫)與對帳/報表(讀)分流
│   └─ 讀副本(可稍舊,對帳靠流水重算)
├─ 為何綁：快照 + 讀寫分離,一起保「記帳不被非核心讀拖垮」
└─ 撐不住的訊號 → 單一熱帳戶仍是寫熱點

【P3 · 熱帳戶分片 + 分桶記帳】約 50,000 QPS
├─ 定義：熱帳戶(平台/商戶主帳)分片/分桶承接高頻記帳,讀時聚合
├─ 大方塊：熱帳戶分桶 · 跨桶記帳 · 讀聚合餘額
└─ 為何綁：一分桶,冪等鍵/餘額聚合/對帳都收斂到桶

【P4 · 帳本獨立 + 冷熱分離歸檔】約 100,000+ QPS
├─ 定義：帳本整包獨立部署與業務隔離,歷史流水冷熱分離歸檔
├─ 大方塊：帳本獨立叢集 · 近期熱/歷史冷歸檔 · 稽核走冷區
└─ 為何綁：金流地基隔離 = 獨立部署 + 冷熱分離,一起做
```
