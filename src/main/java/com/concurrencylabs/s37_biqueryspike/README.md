# BI / 後台查詢爆炸 · s37_biqueryspike

> 一句話定位：內部後台跑各種聚合 / 篩選 / 報表查詢，重讀取會把支撐線上的資料庫拖垮。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：企業資料
- 壓力模型：讀爆型（重聚合查詢）
- 數據角色：分析 / 查詢讀（OLAP 傾向）

---

## Block 2：為什麼被列上這些

- **讀爆型 / 分析讀** → BI 查詢是重聚合、掃大量列的讀，型態跟線上交易（OLTP）完全不同。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
營運 / PM 在後台點「這個月各區銷售趨勢」
   │
   ▼
[BI 查詢] 掃大量訂單、GROUP BY、算聚合
   ├─ 查得慢 → 使用者一直等
   └─ 若打線上主庫 → 把即時交易一起拖慢

────────── 開會前大家同時拉報表 ──────────

營運想要：報表要能查，但別影響前台使用者下單
```

### 分段解說

- BI 查詢是「重讀取」：掃很多資料、做聚合。
- 這種查詢跟線上交易搶同一個庫，會互相傷害。
- 常見解法方向是把分析查詢跟線上交易的資料源分開。

### 痛點分析

- **讀爆還寫爆？** 讀爆，且是重聚合。
- **OLTP vs OLAP** → 兩種型態不該擠同一個庫。
- **能不能延遲？** 報表通常可接受稍舊的資料。
- **真正的痛**：內部分析查詢的重讀取，會跟線上交易搶資源，把兩者分離是關鍵方向。

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s37 OLAP 重讀、可稍舊,關鍵是與 OLTP 分離,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：BI 查詢直接打 DB 做聚合
├─ 大方塊：聚合查詢 · 索引
└─ 撐不住的訊號 → 重聚合掃大量列、與線上交易搶主庫

【P2 · 走讀副本 + 快取 + OLTP/OLAP 分離】約 10,000 QPS
├─ 定義：分析查詢移出主庫走讀副本,熱報表快取
├─ 大方塊（綁一起）
│   ├─ 分析查詢走讀副本(禁打 OLTP 主庫)
│   ├─ 熱門報表結果快取
│   └─ 容忍稍舊
├─ 為何綁：不傷線上 = 讀副本 + 快取 + 分離,一起做
└─ 撐不住的訊號 → 重聚合在副本上仍慢、臨時查詢多

【P3 · OLAP 數倉 + 預聚合/物化視圖】約 50,000+ QPS
├─ 定義：分析走專用 OLAP 數倉,熱查詢預聚合/物化視圖
├─ 大方塊：OLAP 數倉 · 預聚合/物化視圖 · 查詢併發隔離
└─ 為何綁：重聚合規模化 = 數倉 + 預聚合,一起做
```
