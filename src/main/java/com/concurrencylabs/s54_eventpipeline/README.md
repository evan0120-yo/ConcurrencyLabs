# 事件資料管線 Pipeline · s54_eventpipeline

> 一句話定位：把各方湧入的事件驗證、分流、扇出給多個下游消費者，異常資料要隔離不擋主流。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：AdTech / 資料流
- 壓力模型：寫爆型 / ingestion（核心）
- 數據角色：ingestion + schema 驗證 + 分流

---

## Block 2：為什麼被列上這些

- **寫爆型 / ingestion** → 管線入口承受海量事件寫入。
- **數據角色** → 驗 schema、分 topic、扇出 consumer 是資料管線的中樞職責。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
各方事件湧入（App / Web / Server）
   │
   ▼
[事件管線]
   ├─ 驗 schema：格式不對的 ──> 進死信隔離，不擋好資料
   ├─ 分流：不同事件 ──> 不同 topic
   └─ 扇出：同一份事件 ──> 多個下游（計費 / 報表 / 推薦）各取所需

────────── 某上游改了格式，吐一堆壞資料 ──────────

平台想要：壞資料被隔離，好資料照常流，下游各自不互相影響
```

### 分段解說

- 管線是「收 → 驗 → 分 → 扇出」的中樞。
- 壞資料要隔離（死信），不能污染或堵住好資料。
- 一份事件多個下游各自消費，彼此速度不同不能互相拖累。

### 痛點分析

- **讀爆還寫爆？** 寫爆為主。
- **壞資料怎辦？** 隔離到死信，不擋主流。
- **扇出** → 多下游解耦，快慢互不影響。
- **真正的痛**：作為多方事件的中樞，要在海量寫入下同時做到「壞資料隔離」與「多下游互不拖累」。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s54 ingestion 中樞、壞資料死信隔離+多下游解耦,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：收事件驗 schema 分流扇出,壞資料死信
├─ 大方塊：schema 驗證 · 分流 topic · 扇出下游
└─ 撐不住的訊號 → 上游改壞格式吐壞資料、多下游快慢互拖

【P2 · schema 驗證 + 死信隔離 + 多下游扇出解耦】約 10,000 QPS
├─ 定義：驗 schema 壞資料進死信隔離,一份事件扇出多下游各自消費
├─ 大方塊（綁一起）
│   ├─ schema 驗證 + 死信隔離(不擋好資料)
│   ├─ 分 topic 路由
│   └─ 多下游 consumer group 解耦(快慢互不拖累)
├─ 為何綁：壞資料隔離 + 多下游互不拖累 = 驗證 + 死信 + 扇出,一起做
└─ 撐不住的訊號 → 寫入量體再放大、單分區吞吐不足

【P3 · topic 分區 + ingestion 分片 + consumer 隔離】約 50,000+ QPS
├─ 定義：入口 ingestion 分片,topic 分區並行,各下游 consumer 資源隔離
├─ 大方塊：ingestion 分片 · topic 分區 · consumer 隔離
└─ 為何綁：規模化中樞 = 分片 + 分區 + consumer 隔離,一起做
```
