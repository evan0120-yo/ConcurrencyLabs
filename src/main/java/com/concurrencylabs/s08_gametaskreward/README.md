# 遊戲任務獎勵系統 · s08_gametaskreward

> 一句話定位：玩家完成任務後發獎，主流程不等發獎，且同一任務只能領一次。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：任務補償型（非同步發獎 + 防重）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 發獎可從主遊戲流程解耦、非同步進行，並需要「防重複領」的冪等保護。
- **博弈 domain** → 任務系統是活動留存的常見手段。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
玩家完成「今日登入 / 累積下注 1000」任務
   │
   ▼
[任務系統] 判定達成 ──> 發獎（金幣 / 道具）
   └─ 同一任務只能領一次（防重複領）

────────── 活動期間大量任務同時達成 ──────────

玩家想要：達成馬上有獎、不要漏發
營運想要：一個任務只發一次，不能被刷重領
```

### 分段解說

- 達成與發獎可以拆開：主遊戲不該卡在發獎上。
- 真正的難點是「不重不漏」——既不能漏發，也不能被重複領。
- 任務狀態像一個小狀態機：未達成 → 達成 → 已領取。

### 痛點分析

- **能不能非同步？** 可以，發獎跟主流程解耦。
- **能不能重複 / 漏發？** 都不行，冪等是核心。
- **平均還是尖峰？** 活動整點常有集中達成。
- **真正的痛**：把「發獎」跟「主遊戲」解耦後，如何保證每個任務不多發也不漏發。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s08 發獎**可解耦、可延遲**，只要冪等不重不漏，放寬讓連鎖變少 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：任務達成同步發獎，grant 單防重領
├─ 大方塊：[DB] 判定達成 · grant 冪等單防重 · 發道具/幣
└─ 撐不住的訊號 → 主遊戲流程被發獎拖慢、活動整點集中達成

【P2 · 發獎進 MQ 非同步 + 冪等】約 10,000 QPS
├─ 定義：達成事件進 MQ，發獎與主遊戲解耦、非同步
├─ 大方塊（綁一起）
│   ├─ [MQ] 達成事件進佇列、worker 消費發獎
│   └─ grant 單冪等（同 user 同 task 只發一次，重投安全）
├─ 為何綁：一解耦，發放靠 at-least-once 投遞 + grant 冪等 = 不多不漏
└─ 撐不住的訊號 → 大量任務並發達成、防重壓力上升

【P3 · 任務狀態機 + 防重事件表】約 50,000+ QPS
├─ 定義：任務走狀態機、去重靠事件表，規模化保證 exactly-once 發放
├─ 大方塊：狀態機(未達成→達成→已領) · 防重事件表(去重高併發達成) · 獨立獎勵服務
└─ 為何綁：規模一大，防重從單筆冪等升級成「事件表 + 狀態機」整套
```
