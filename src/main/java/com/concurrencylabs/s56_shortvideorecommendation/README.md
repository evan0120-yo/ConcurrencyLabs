# 短影音首頁推薦流 · s56_shortvideorecommendation

> 一句話定位：使用者一開 App 就要有一串「可能想看」的影片，推薦要即時給、但重運算得預算好。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：社群 / Feed / 搜尋 / Marketplace
- 壓力模型：讀爆型（每次刷都要一批推薦）
- 數據角色：分析 / 特徵讀（召回 → 排序）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 每次下滑都要生成 / 取一批推薦，讀壓力極高。
- **特徵讀角色** → 推薦要讀使用者與內容特徵，屬線上特徵存取。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者一開 App / 一直下滑
   │
   ▼
[推薦流] 要即時給下一批「可能想看」的影片
   ├─ 召回：從海量影片粗選幾百支
   ├─ 粗排：快速篩到幾十支
   └─ 精排：細算排出最終順序
   *重運算不能臨場全做，否則刷不動*

────────── 每個使用者、每次下滑都要一批 ──────────
```

### 分段解說

- 推薦是「漏斗」：召回 → 粗排 → 精排，一層層縮小。
- 越後面越精細越貴，不能全部臨場算。
- 常把離線 / 近線算好的特徵備著，線上只做輕量排序。

### 痛點分析

- **讀爆還寫爆？** 讀爆（每次刷都要）。
- **能不能預算？** 重運算要離線 / 近線先算好。
- **延遲要求** → 要即時給，慢了使用者就滑走。
- **真正的痛**：推薦品質要靠重運算，但刷的頻率極高，逼你把重活挪到離線，線上只留輕量排序。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s56 讀爆漏斗、重運算離線化,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：線上召回+排序給推薦
├─ 大方塊：召回 · 排序 · 過濾
└─ 撐不住的訊號 → 每次下滑一批、重運算臨場算不動

【P2 · 漏斗(召回/粗排/精排) + 離線特徵預算】約 10,000 QPS
├─ 定義：漏斗分層,embedding/特徵離線預算,線上只精排輕量
├─ 大方塊（綁一起）
│   ├─ 召回(用預算 embedding/標籤)
│   ├─ 粗排輕量 + 精排(線上唯一較重、需來得及)
│   └─ 離線/近線算好特徵與候選池
├─ 為何綁：即時又要品質 = 漏斗 + 離線預算,一起做
└─ 撐不住的訊號 → 特徵讀取量大、召回候選池熱

【P3 · 特徵存儲分片 + 召回並行 + 結果快取】約 50,000+ QPS
├─ 定義：線上特徵存儲分片,召回多路並行,推薦結果短快取
├─ 大方塊：特徵存儲分片 · 召回並行 · 結果快取
└─ 為何綁：規模化推薦 = 特徵分片 + 召回並行 + 快取,一起做
```
