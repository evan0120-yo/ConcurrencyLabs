# SA.md — s56_shortvideorecommendation（短影音首頁推薦流）

> 事實文件（SASD）· 結構化分析。描述「推薦流要做什麼」的**業務行為**，平台無關（P1..Pn 共用；各平台怎麼扛量在各自 SD）。
> 推薦是**漏斗**：召回 → 粗排 → 精排，一層層縮小。刷的頻率極高，逼你把重運算挪到離線 / 近線，線上只留輕量排序。

## 1. Context Diagram

```text
[使用者] --(刷/下滑: userId)--> (推薦流) --(下一批可能想看的影片)--> [使用者]

(推薦流) --(讀使用者/內容特徵)--> [特徵存儲 / util.redis]
(推薦流) --(離線/近線算好的候選、embedding)--> [離線特徵]
```

## 2. DFD（精簡 box flow）

```text
[刷/下滑] --(userId, context)--> [漏斗]
   ├─ 召回：從海量影片粗選幾百支（用預算好的 embedding / 標籤）
   ├─ 粗排：輕量模型快速篩到幾十支
   └─ 精排：細算排出最終順序（線上只做這層輕量計算）
   ▼
[多樣性 / 去重 / 已看過濾] ── 回一批推薦
```

## 3. Process Spec（行為基準，decision table）

```text
[漏斗分層]
├─ 召回：候選來源（興趣標籤 / 協同 / embedding 近鄰）—— 依賴離線預算
├─ 粗排：輕量打分縮小
└─ 精排：最終排序（線上唯一的較重計算，仍需來得及）

[重運算離線化]（核心）
└─ embedding / 特徵 / 候選池 一律離線 / 近線先算好；線上不做重訓練 / 重召回

[即時性]
└─ 要即時給下一批（慢了使用者滑走）；用預算好的候選 + 輕排序達成

[去重 / 過濾]
└─ 已看過 / 不感興趣過濾；同一刷內不重複
```

## 4. State Transition

```text
（服務型，無業務狀態機；一次刷 = 召回 → 粗排 → 精排 → 回批 的請求生命週期）
```

## 5. Data Dictionary（邏輯欄位；實體 DDL 在各平台 SD）

```text
user_feature（使用者特徵，離線/近線更新）
├─ userId(PK) / interests / embedding / recentSeen[]

item_feature（內容特徵）
├─ itemId(PK) / tags / embedding / stats（熱度）

（召回候選池、精排模型為線上服務所讀，見各平台 SD）
```
