# 商品 / 內容搜尋系統 · s58_search

> 一句話定位：使用者輸入關鍵字找商品 / 內容，熱門查詢可快照，尖峰時要能降級保住可用。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：社群 / Feed / 搜尋 / Marketplace
- 壓力模型：讀爆型（核心）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 搜尋是重讀取（掃索引、算相關性、過濾排序），且熱門查詢高度集中。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者搜「藍芽耳機」
   │
   ▼
[搜尋] 掃索引 ──> 算相關性 ──> 套篩選（價格 / 分類）──> 排序
   ├─ 熱門查詢（大促爆款詞）→ 反覆搜同一個 → 可快照
   └─ 尖峰扛不住 → 降級（少算點、給快取結果）

────────── 大促時大家都搜同幾個熱詞 ──────────

使用者想要：搜得到、夠相關、夠快
```

### 分段解說

- 搜尋是重讀取：不只查，還要算相關性與排序。
- 熱門查詢高度集中（大家搜同幾個詞），可快照結果。
- 尖峰扛不住時，寧可降級給「還可以的結果」也不要掛。

### 痛點分析

- **讀爆還寫爆？** 讀爆，且是重計算讀。
- **能不能快照 / 降級？** 熱門可快照，尖峰可降級。
- **有沒有熱點？** 熱門查詢詞是熱點。
- **真正的痛**：搜尋的重計算遇上熱詞集中與尖峰，要靠「熱門快照 + 可降級」在扛不住時仍給出可用結果。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s58 重計算讀爆、熱門可快照/尖峰可降級,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：掃索引算相關性篩選排序
├─ 大方塊：索引查詢 · 相關性 · 排序
└─ 撐不住的訊號 → 熱門查詢集中、重計算讀、尖峰扛不住

【P2 · 熱門查詢快照 + 可降級】約 10,000 QPS
├─ 定義：熱門查詢結果快照直接回,尖峰可降級保可用
├─ 大方塊（綁一起）
│   ├─ 熱門查詢結果快照(命中直接回)
│   ├─ 尖峰降級(少算相關性/給快取/縮結果)
│   └─ 可用性優先(寧降級不掛)
├─ 為何綁：重計算讀爆 = 快照 + 降級,一起做
└─ 撐不住的訊號 → 索引規模大、熱詞單點集中

【P3 · 索引分片 + 多層快取 + 熱詞獨立】約 50,000+ QPS
├─ 定義：搜尋索引分片,結果多層快取,熱詞獨立資源
├─ 大方塊：索引分片 · 多層快取 · 熱詞獨立
└─ 為何綁：規模化搜尋 = 分片 + 快取 + 熱詞隔離,一起做
```
