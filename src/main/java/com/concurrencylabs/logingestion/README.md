# Log Ingestion 系統 · logingestion

> 一句話定位：收下海量服務日誌，緩衝削峰、按租戶分區、冷熱分層，hot tenant 要限流。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Observability
- 壓力模型：寫爆型 / ingestion（核心）、多租戶型（tenant 分區）
- 數據角色：ingestion + 冷熱分層

---

## Block 2：為什麼被列上這些

- **寫爆型 / ingestion** → 日誌是持續海量的寫入流。
- **多租戶型** → 多服務 / 多租戶共用，某個服務暴量刷 log 不能拖垮全部。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
成百上千個服務 / agent 持續吐 log
   service A ─┐
   service B ─┼──> [Log 收集] 緩衝削峰 ──> 存起來
   暴量的 C  ─┘        │
                       ├─ 近期 log → 熱、常查
                       └─ 舊 log → 冷、少查 → 便宜存

────────── 某服務出 bug，log 量暴增 100 倍 ──────────

平台想要：那個爆量服務被限流，別把大家的 log 系統拖垮
```

### 分段解說

- Log 是寫爆的經典，跟 IoT 心跳同家族。
- 某服務故障常伴隨「log 暴增」，反而更需要限流保護。
- 近期 log 熱、歷史 log 冷，要分層存放。

### 痛點分析

- **讀爆還寫爆？** 寫爆為主。
- **能不能丟 / 抽樣？** 大量 log 可抽樣 / 限流。
- **冷熱分層** → 近期熱、歷史冷。
- **真正的痛**：服務出事時 log 暴增，最需要被收的時刻也是最容易把系統寫爆的時刻，hot tenant 限流是保命關鍵。
