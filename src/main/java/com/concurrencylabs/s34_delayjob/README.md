# 延遲任務系統 · s34_delayjob

> 一句話定位：讓任務「到某個時間點才執行」，例如下單 30 分鐘未付款自動取消，且大量到期不能集中爆發。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：任務 / 排程 / 第三方整合
- 壓力模型：任務補償型（延遲觸發）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 屬於「排程 + 到期執行 + 重試」的一種，是背景任務的時間維度延伸。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
下單成功
   └──> 排一個延遲任務：「30 分鐘後若未付款，自動取消」
             │
   ...(等待 30 分鐘)...
             ▼
   到期 ──> [延遲任務] 檢查狀態
             ├─ (已付款) ──> 取消這個延遲任務
             └─ (未付款) ──> 執行取消訂單

────────── 大促時，同一分鐘有 10 萬單同時到期 ──────────
```

### 分段解說

- 延遲任務的難點在「時間」：要在對的時刻、不早不晚觸發。
- 大量任務可能集中在同一時刻到期，形成尖峰。
- 到期時狀態可能已變（已付款），要先檢查再執行。

### 痛點分析

- **時間精度** → 不能太早也不能太晚觸發。
- **集中到期** → 同一時刻大批到期會爆，要打散。
- **狀態校驗** → 到期時要重新確認條件是否還成立。
- **真正的痛**：大量任務集中在同一時間點到期時，如何避免「到期洪峰」壓垮系統。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s34 到期洪峰要打散、到期校驗,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：延遲佇列依 runAt 排,到期掃描 + 狀態校驗執行
├─ 大方塊：延遲結構(有序集) · 到期掃描 · 狀態校驗
└─ 撐不住的訊號 → 同一分鐘 10 萬單到期、到期洪峰壓垮

【P2 · 時間輪/分桶 + 到期打散 + 冪等】約 10,000 QPS
├─ 定義：時間輪/分桶承接海量延遲任務,到期加抖動打散
├─ 大方塊（綁一起）
│   ├─ 時間輪 / 分桶延遲結構
│   ├─ 到期分桶 + 隨機抖動(平滑洪峰)
│   └─ 到期冪等執行 + 狀態校驗
├─ 為何綁：避到期洪峰 = 分桶結構 + 抖動打散 + 冪等,一起做
└─ 撐不住的訊號 → 延遲任務量體再放大、單節點時間輪吃不下

【P3 · 分片時間輪 + 掃描器分片並行】約 50,000+ QPS
├─ 定義：時間輪分片,多掃描器並行觸發,任務按 key 分散
├─ 大方塊：時間輪分片 · 掃描器分片並行 · 到期任務分散
└─ 為何綁：規模化延遲 = 分片時間輪 + 並行掃描,一起做
```
