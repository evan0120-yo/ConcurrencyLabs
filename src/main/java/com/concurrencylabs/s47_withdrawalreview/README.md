# 提幣審核 / 出金系統 · s47_withdrawalreview

> 一句話定位：使用者申請提幣，經風控與冷熱錢包流程審核後上鏈廣播，資產安全優先於即時性。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：交易所 / Web3
- 壓力模型：錢包帳本型（出金真資產）、任務補償型（審核 / 排程 / 上鏈重試）、平台防爆型（限流保安全）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 出金是把真資產送出去，錯了幾乎不可逆。
- **任務補償型** → 審核 → 簽名 → 上鏈廣播是多步流程，上鏈會失敗要重試。
- **平台防爆型** → 提幣要限流、風控把關，安全優先於快。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者申請提幣 1 BTC
   │
   ▼
[提幣審核] 風控 ──> 額度 / 名單 / 行為檢查
   ├─ (可疑 / 大額) ──> 人工 / 多簽審核
   └─ (通過) ──> 從熱錢包簽名 ──> 上鏈廣播
                     ├─ (廣播失敗) ──> 重試
                     └─ (成功) ──> 標記完成

────────── 提幣不可逆，寧可慢、不可錯 ──────────
```

### 分段解說

- 出金跟充值相反：一旦送出鏈上幾乎收不回。
- 所以流程刻意「重審核、可延遲」，安全優先於即時。
- 上鏈廣播會失敗，要能安全重試而不重複出金。

### 痛點分析

- **能不能延遲？** 可以，安全優先於速度。
- **能不能重複 / 錯？** 絕不能，出金不可逆。
- **要不要限流？** 要，異常提幣要能被攔。
- **真正的痛**：面對「不可逆」的出金，要用多重審核與可靠上鏈重試，把「錯誤 / 重複出金」的機率壓到零。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+（量體為提幣申請/審核面）。每跳一個平台 = 一次架構改造。
> s47 出金不可逆但可延遲、重審核,throughput 不高 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：風控審核→熱錢包簽名→上鏈廣播,失敗重試
├─ 大方塊：風控審核 · 簽名上鏈 · withdrawId 去重
└─ 撐不住的訊號 → 上鏈失敗重試風險重複出金、大額可疑需人工

【P2 · 多簽/冷熱錢包 + 上鏈重試冪等 + 限流】約 10,000 QPS
├─ 定義：大額走多簽/冷錢包,上鏈可靠重試不重複出金,提幣限流
├─ 大方塊（綁一起）
│   ├─ 風控分級(小額自動/大額多簽冷錢包)
│   ├─ 上鏈重試冪等(同 withdrawId 綁唯一鏈上交易)
│   └─ 提幣限流(攔異常)
├─ 為何綁：不可逆出金壓到零錯 = 多簽 + 重試冪等 + 限流,一起做
└─ 撐不住的訊號 → 提幣量體上升、多鏈簽名/廣播並行需求

【P3 · 多鏈簽名並行 + 出金資源隔離】約 50,000+ QPS
├─ 定義：多鏈簽名/廣播分片並行,出金流程與線上交易資源隔離
├─ 大方塊：多鏈簽名並行 · 廣播分片 · 出金資源隔離
└─ 為何綁：規模化安全出金 = 多鏈並行 + 資源隔離,一起做
```
