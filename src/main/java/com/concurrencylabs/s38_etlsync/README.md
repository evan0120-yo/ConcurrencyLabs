# ETL / 大量資料同步 · s38_etlsync

> 一句話定位：把資料從來源大量抽取、清洗、寫入目標，用緩衝當蓄水池，分批去重補償。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：企業資料
- 壓力模型：寫爆型 / ingestion（大量寫入目標）
- 數據角色：ingestion + 資料清洗 / 冷熱分層

---

## Block 2：為什麼被列上這些

- **寫爆型 / ingestion** → 目標端要承受大批量資料寫入。
- **數據角色** → 抽取、清洗、去重、入庫是資料管線的核心處理鏈。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
每天 / 每小時把來源資料搬到目標
   [來源系統] ──抽取──> [清洗 / 去重 / 轉換] ──寫入──> [目標庫 / 倉]
                              │
                     用緩衝當蓄水池，削掉尖峰

────────── 來源突然吐一大批（補跑 / 初始化）──────────

工程想要：資料要對、要能重跑不重複、別把目標端寫爆
```

### 分段解說

- ETL 是資料的搬運與加工，量體大、批次性強。
- 尖峰（補跑 / 初始化）需要緩衝當蓄水池削峰。
- 重跑不能造成重複資料，去重與冪等是關鍵。

### 痛點分析

- **讀爆還寫爆？** 目標端寫爆為主。
- **能不能重跑？** 要，且重跑不能產生重複。
- **能不能延遲？** 多數 ETL 可接受批次延遲。
- **真正的痛**：大批量資料搬運要在「正確（不重不漏）」與「不把目標端寫爆」之間取得平衡。
