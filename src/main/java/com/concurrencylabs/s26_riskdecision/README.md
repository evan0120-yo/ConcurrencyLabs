# 風控決策系統 · s26_riskdecision

> 一句話定位：交易當下要快速判 allow / deny，硬規則同步跑，複雜模型非同步補判。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：支付 / 金融
- 壓力模型：平台防爆型（同步硬規則守門）、寫爆型（事件記錄，延伸）

---

## Block 2：為什麼被列上這些

- **平台防爆型** → 風控是交易路徑上的守門員，同步決策必須快、必須穩。
- **寫爆型（延伸）** → 每筆交易都產生風控事件供事後分析，是寫入洪流的一部分。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一筆交易進來
   │
   ▼
[風控] 分兩條路
   ├─ 同步：硬規則（黑名單 / 額度 / 頻率）──> allow / deny（毫秒級）
   └─ 非同步：複雜模型（行為分析 / 關聯圖）──> 事後補判、標記

────────── 同步這關卡住太久，整個付款都變慢 ──────────

營運想要：擋住明顯的壞人，又不能拖慢所有正常人
```

### 分段解說

- 風控有兩種：能秒判的硬規則、要慢慢算的複雜模型。
- 同步路徑只能放「快而確定」的硬規則，否則拖垮交易。
- 複雜模型放到非同步，事後補判、標記可疑。

### 痛點分析

- **延遲要求** → 同步決策要毫秒級，不能拖慢交易。
- **能不能非同步？** 複雜判斷可以，事後補。
- **取捨** → 攔截力 vs 交易體驗，要平衡。
- **真正的痛**：同步這關要「又快又準」，把重運算挪到非同步，避免風控本身變成交易瓶頸。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s26 同步守門需毫秒,複雜模型可非同步事後補,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：同步硬規則(黑名單/額度/頻率)毫秒 allow/deny
├─ 大方塊：硬規則同步判定 · 規則/名單查詢
└─ 撐不住的訊號 → 規則臨場慢查拖慢交易、複雜判斷塞在同步路徑

【P2 · 硬規則快取 + 複雜模型非同步】約 10,000 QPS
├─ 定義：規則與名單快取化,複雜模型挪到非同步事後補判
├─ 大方塊（綁一起）
│   ├─ [Redis] 規則/名單快取(同步毫秒)
│   ├─ [MQ] 風控事件非同步 → 複雜模型補判
│   └─ 補判回饋名單/標記(不阻當下交易)
├─ 為何綁：同步只放快而確定 + 重運算外移,一起保「又快又準」
└─ 撐不住的訊號 → 風控事件寫入洪流、模型算力瓶頸

【P3 · 規則引擎分片 + 事件寫削峰 + 熱名單分散】約 50,000+ QPS
├─ 定義：規則引擎分片並行、事件寫入 MQ 削峰、熱名單分片
├─ 大方塊：規則引擎分片 · 事件寫削峰 · 熱名單分散
└─ 為何綁：同步守門規模化 = 引擎分片 + 寫削峰 + 名單分散,一起做
```
