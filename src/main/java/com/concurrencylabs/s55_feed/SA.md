# SA.md — s55_feed（追蹤動態牆 Feed）

> 事實文件（SASD）· 結構化分析。描述「Feed 要做什麼」的**業務行為**，平台無關（P1..Pn 共用；各平台怎麼扛量在各自 SD）。
> 核心矛盾：發文擴散「寫時推 vs 讀時拉」。同一套 Feed 要同時服務小號（寫擴散）與大 V（讀擴散），逼出**混合策略**。動態牆可容忍秒級延遲。

## 1. Context Diagram

```text
[使用者] --(刷牆: userId)--> (Feed) --(追蹤對象最新貼文列表)--> [使用者]
[發文者] --(發文: authorId, post)--> (Feed) --(擴散: 推 / 待拉)--> [粉絲的牆]

(Feed) --(收件匣 / 貼文)--> [util.redis / PostgreSQL]
```

## 2. DFD（精簡 box flow）

```text
【發文擴散】
[發文] --(authorId 粉絲數判定)-->
   ├─(小號/粉絲少)──> 寫擴散：發文時推進每個粉絲的收件匣
   └─(大 V/粉絲千萬)──> 讀擴散：不全推，只記在作者時間線（粉絲來刷時才拉）

【刷牆】
[刷牆] --> 組牆
   ├─ 讀收件匣（寫擴散來的：小號貼文）
   └─ 拉所追蹤大 V 的最新貼文（讀擴散）
   ▼
合併 / 排序（時間或權重）── 回動態牆（可秒級延遲）
```

## 3. Process Spec（行為基準，decision table）

```text
[擴散策略]（核心）
├─ 作者粉絲數 < 門檻（小號）→ 寫擴散（發文即推粉絲收件匣）
└─ 作者粉絲數 >= 門檻（大 V）→ 讀擴散（發文不全推，讀時拉）
   → 避免大 V 一發文就寫爆千萬收件匣

[刷牆]
└─ 合併「收件匣（寫擴散）」+「所追蹤大 V 最新（讀擴散）」→ 排序

[一致性]
└─ 動態牆容忍秒級延遲（新貼文晚幾秒出現可接受）
```

## 4. State Transition

```text
（服務型，無業務狀態機；貼文的 發布 → 擴散 → 進牆 流程）
```

## 5. Data Dictionary（邏輯欄位；實體 DDL 在各平台 SD）

```text
post（貼文）
├─ postId(PK) / authorId / content / createdAt

inbox（收件匣，寫擴散目標，活在 Redis）
├─ userId / postId / score（排序）

（大 V 走作者時間線，讀時拉，不進收件匣）
```
