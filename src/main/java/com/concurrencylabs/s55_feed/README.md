# 追蹤動態牆 Feed · s55_feed

> 一句話定位：使用者刷動態牆看追蹤對象的貼文，大 V 與小號的分發策略完全不同。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：社群 / Feed / 搜尋 / Marketplace
- 壓力模型：讀爆型（刷牆）、熱點型（大 V）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 刷 Feed 是最高頻的讀操作。
- **熱點型** → 大 V 一則貼文要送給數百萬粉絲，是極端扇出熱點。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者打開 App 刷動態牆
   │
   ▼
[Feed] 要組出「我追蹤的人最新貼了什麼」
   ├─ 小號發文 → 粉絲少 → 發文時就推到粉絲的牆（寫擴散）
   └─ 大 V 發文 → 粉絲千萬 → 發文時全推會爆 → 讀取時才拉（讀擴散）

────────── 大 V 發一則，千萬人的牆都要更新 ──────────

使用者想要：刷開就有最新、順順的
```

### 分段解說

- Feed 的核心矛盾在「發文擴散」：寫時推 vs 讀時拉。
- 小號適合寫擴散（發文時推給少數粉絲）。
- 大 V 適合讀擴散（發文不全推，粉絲來刷時才拉），否則一發文就爆。

### 痛點分析

- **讀爆還寫爆？** 讀爆為主，但大 V 發文是寫爆風險。
- **有沒有熱點？** 大 V 是極端熱點。
- **能不能延遲？** 動態牆可容忍秒級延遲。
- **真正的痛**：同一套 Feed 要同時服務「小號」與「大 V」兩種極端，逼出寫擴散 / 讀擴散的混合策略。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s55 讀爆 + 大 V 極端扇出、寫/讀擴散混合,放寬(秒級延遲) → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：刷牆拉取追蹤對象貼文
├─ 大方塊：貼文儲存 · 拉取組牆
└─ 撐不住的訊號 → 大 V 發文千萬粉絲要更新、刷牆讀爆

【P2 · 寫擴散/讀擴散混合 + 收件匣】約 10,000 QPS
├─ 定義：小號寫擴散推收件匣,大 V 讀擴散讀時拉,合併組牆
├─ 大方塊（綁一起）
│   ├─ 小號寫擴散(發文推粉絲收件匣)
│   ├─ 大 V 讀擴散(不全推,讀時拉)
│   └─ 收件匣 + 大 V 最新合併排序
├─ 為何綁：同時服務小號/大 V = 寫擴散 + 讀擴散 + 合併,一起做
└─ 撐不住的訊號 → 超級大 V、活躍用戶收件匣熱點

【P3 · 收件匣分片 + 大 V 獨立讀擴散 + 多層快取】約 50,000+ QPS
├─ 定義：收件匣按 userId 分片,超級大 V 獨立讀擴散,熱牆多層快取
├─ 大方塊：收件匣分片 · 大 V 獨立鏈路 · 多層讀快取
└─ 為何綁：規模化 Feed = 分片 + 大 V 隔離 + 快取,一起做
```
