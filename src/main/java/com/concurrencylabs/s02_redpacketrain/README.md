# 紅包雨系統 · s02_redpacketrain

> 一句話定位：畫面撒下紅包，使用者瘋狂點螢幕搶，紅包池總金額固定、發完為止。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：搶資源型（核心）、熱點計數型（紅包池餘額）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 池內總金額固定，大量人同時搶同一份有限資源。
- **熱點計數型** → 「紅包池剩餘金額」被高頻扣減，是單一熱 key。
- **博弈 domain** → 典型活動玩法，跟 `s01_lottery`、`s14_couponclaim` 是搶資源三兄弟。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
【畫面】紅包從天上掉，使用者瘋狂點螢幕
   │  每點一下 = 一次搶包請求（一人會點很多下）
   ▼
[紅包池] 總預算固定（例：100 萬）
   ├─ (搶到) ──> 金額落袋
   └─ (池空) ──> 後續點擊都是空手

────────── 千人同時狂點、單人高頻連點 ──────────

使用者想要：手感即時、搶到有錢
營運想要：發出去的總額 = 預算，一毛都不能超
```

### 分段解說

- 手感是體驗核心：點下去要「立刻」有回饋。
- 但金額是硬預算：所有人搶到的總和不能超過池子。
- 單人狂點帶來「重複請求 / 疑似作弊」問題，跟一般搶資源不同。

### 痛點分析

- **能不能超發？** 總額是預算紅線，絕不能超。
- **能不能延遲？** 顯示可延遲，但落袋金額要準。
- **平均還是尖峰？** 極端尖峰，且單人請求密度極高。
- **真正的痛**：既要即時手感、又要總額精準、還要防單人洗版，三個要求同時壓在一個熱點池上。

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s02 與 s01 都 4 個平台，但驅動力不同：s02 是「瞬時洪峰 + 金額拆分」，主軸在切片與排隊。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：Redis 紅包池扛併發，搶到落 DB
├─ 大方塊：[Redis] 紅包池原子扣減 · [Redis] set 防重領 · [DB] 落領取流水
└─ 撐不住的訊號 → 紅包雨是「幾秒內數十萬人同搶」的瞬時洪峰，池餘額單點被打爆

【P2 · 紅包預切片 token 化】約 10,000 QPS
├─ 定義：把「即時算金額」變「取現成的包」
├─ 大方塊（綁一起）
│   ├─ 活動前把總金額切成 N 個固定包（切片 job）
│   ├─ 搶 = [Redis] 原子取一個 token
│   └─ 落庫 schema 從「記金額」改「記 tokenId → 金額」
├─ 為何綁：切片 job + 搶邏輯 + 落庫結構是同一件事的三面，拆不開
└─ 撐不住的訊號 → 單一池 / 單一 token 序列仍是熱點

【P3 · 池分 shard + 跨片防重】約 50,000 QPS
├─ 定義：把單池熱點打散成多 shard
├─ 大方塊
│   ├─ 紅包池分多 shard、使用者路由到某 shard
│   ├─ 防重領改「跨 shard 唯一」（不能各 shard 各領一次）
│   └─ token 在 shard 間均勻分佈
├─ 為何綁：一分片，「一人只能領一次」就不能靠單一 set，得跨片保證
└─ 撐不住的訊號 → 瞬時洪峰打到最前端入口

【P4 · 前端限流 + 排隊發放】約 100,000+ QPS
├─ 定義：把洪峰擋在最前面，發放可延遲
├─ 大方塊：前端限流 · 令牌排隊放行 · 結果延遲顯示
└─ 為何綁：對付 burst 的一整套組合拳（顯示可延遲是前提）
```
