# 紅包雨系統 · s02_redpacketrain

> 一句話定位：畫面撒下紅包，使用者瘋狂點螢幕搶，紅包池總金額固定、發完為止。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：搶資源型（核心）、熱點計數型（紅包池餘額）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 池內總金額固定，大量人同時搶同一份有限資源。
- **熱點計數型** → 「紅包池剩餘金額」被高頻扣減，是單一熱 key。
- **博弈 domain** → 典型活動玩法，跟 `s01_lottery`、`s14_couponclaim` 是搶資源三兄弟。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
【畫面】紅包從天上掉，使用者瘋狂點螢幕
   │  每點一下 = 一次搶包請求（一人會點很多下）
   ▼
[紅包池] 總預算固定（例：100 萬）
   ├─ (搶到) ──> 金額落袋
   └─ (池空) ──> 後續點擊都是空手

────────── 千人同時狂點、單人高頻連點 ──────────

使用者想要：手感即時、搶到有錢
營運想要：發出去的總額 = 預算，一毛都不能超
```

### 分段解說

- 手感是體驗核心：點下去要「立刻」有回饋。
- 但金額是硬預算：所有人搶到的總和不能超過池子。
- 單人狂點帶來「重複請求 / 疑似作弊」問題，跟一般搶資源不同。

### 痛點分析

- **能不能超發？** 總額是預算紅線，絕不能超。
- **能不能延遲？** 顯示可延遲，但落袋金額要準。
- **平均還是尖峰？** 極端尖峰，且單人請求密度極高。
- **真正的痛**：既要即時手感、又要總額精準、還要防單人洗版，三個要求同時壓在一個熱點池上。
