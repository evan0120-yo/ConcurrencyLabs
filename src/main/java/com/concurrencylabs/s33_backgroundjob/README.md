# 背景任務平台 · s33_backgroundjob

> 一句話定位：把耗時工作丟到背景非同步跑，worker 可水平擴展，支援限流、暫停、重跑。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：任務 / 排程 / 第三方整合
- 壓力模型：任務補償型（核心）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 這是所有「非同步、可重試、可調度」工作的通用底座，是任務補償型的原型。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
某操作會觸發耗時工作（寄信 / 產檔 / 同步）
   請求 ──> 丟一個 job ──> [背景任務平台] ──> worker 撿起來跑
                                              ├─ (成功) ──> 標記完成
                                              └─ (失敗) ──> 重試 / 進死信

────────── 任務突然湧入一大批 ──────────

營運想要：能水平加 worker、能限流、能暫停某類任務、能重跑
```

### 分段解說

- 核心價值是「解耦」：請求不等耗時工作做完。
- worker 要能隨負載水平擴展。
- 平台要可控：限流、暫停、重跑是營運的基本需求。

### 痛點分析

- **能不能非同步？** 這題本身就是為非同步而生。
- **失敗怎麼辦？** 要重試、要有死信兜底。
- **可控性** → 限流 / 暫停 / 重跑是必要能力。
- **真正的痛**：作為通用底座，要在「大量任務湧入」時仍可控（不失控雪崩）、可觀測、可人工介入。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s33 非同步任務底座、可重試可控,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：job 進佇列,worker 撿取執行,失敗重試/死信
├─ 大方塊：[MQ] 佇列 · worker 執行 · 重試/DLQ
└─ 撐不住的訊號 → 任務突然湧入、無限流會雪崩、缺營運控制

【P2 · 可控底座:限流 + 暫停 + 死信 + 冪等】約 10,000 QPS
├─ 定義：worker 水平擴展 + 營運控制面(限流/暫停/重跑)
├─ 大方塊（綁一起）
│   ├─ worker 併發限流閥門(保護下游)
│   ├─ 暫停某類 type + DLQ 重跑
│   └─ 執行冪等(重試/重跑不重複副作用)
├─ 為何綁：不失控 = 限流 + 可暫停 + 死信 + 冪等,整套才成通用底座
└─ 撐不住的訊號 → 不同 type 互搶 worker、優先級混雜

【P3 · 任務分片 + type 優先級隔離】約 50,000+ QPS
├─ 定義：任務按 type/分片並行,高優先與低優先隔離資源
├─ 大方塊：任務分片 · type 優先級隔離 · 獨立 worker 池
└─ 為何綁：規模化可控 = 分片 + 優先級隔離,一起做
```
