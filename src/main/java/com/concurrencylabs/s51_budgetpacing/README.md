# 廣告預算 pacing 系統 · s51_budgetpacing

> 一句話定位：即時控制每個廣告活動的花費節奏，別讓預算一下燒光，允許近似扣減、最終對帳修正。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：AdTech / 資料流
- 壓力模型：熱點計數型（活動花費累加）、任務補償型（最終對帳修正）

---

## Block 2：為什麼被列上這些

- **熱點計數型** → 一個活動的「已花金額」被高頻累加，是熱點計數。
- **任務補償型** → 即時扣減用近似值，事後對帳修正，屬最終一致。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
廣告活動有日預算（例：10 萬）
   每次曝光 / 點擊 ──> 花費累加 ──> 逼近預算就要放慢投放
   ▼
   ├─ 若即時精算每一筆 ──> 熱點被打爆、拖慢競價
   └─ 折衷：近似扣減（先估），事後對帳修正

────────── 高流量下，錢燒得飛快 ──────────

廣告主想要：預算平順花一天，別上午就燒光
平台想要：別超花（超花誰買單？）
```

### 分段解說

- pacing 要即時感知「花了多少」來決定投放快慢。
- 每筆都精算會把花費計數器打爆，所以用近似。
- 近似會有誤差，靠事後對帳把帳修平。

### 痛點分析

- **有沒有熱點？** 活動花費計數是熱點。
- **能不能近似？** 即時扣減可近似（求快）。
- **能不能最終一致？** 可以，事後對帳修正誤差。
- **真正的痛**：即時性（近似求快）與準確性（不超花）的拉扯，靠「近似扣減 + 事後對帳」兩段式解決。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s51 花費計數熱點、近似求快+事後對帳,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：花費精算累加,逼近預算放慢
├─ 大方塊：花費累加 · pacing 判定 · 暫停
└─ 撐不住的訊號 → 逐筆精算把花費計數器打爆、拖慢競價

【P2 · 近似扣減 + 事後對帳修正】約 10,000 QPS
├─ 定義：花費近似 counter 求快,pacing 即時決策,事後對帳修正
├─ 大方塊（綁一起）
│   ├─ [Redis] 近似花費 counter(求快)
│   ├─ pacing 即時決策(正常/放慢/暫停)
│   └─ 事後對帳修正誤差(不超花)
├─ 為何綁：即時求快又不超花 = 近似扣減 + 對帳,一起做
└─ 撐不住的訊號 → 活動數多、花費 counter 熱點

【P3 · counter 分片 + 讀聚合 + 分散式對帳】約 50,000+ QPS
├─ 定義：花費 counter 分片,讀時聚合,對帳分散式規模化
├─ 大方塊：花費 counter 分片 · 讀聚合 · 分散式對帳
└─ 為何綁：規模化 pacing = counter 分片 + 聚合 + 對帳,一起做
```
