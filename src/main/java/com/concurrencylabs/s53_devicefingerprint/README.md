# 反詐 / Device Fingerprint · s53_devicefingerprint

> 一句話定位：即時辨識裝置 / 使用者是否可疑，同步只跑硬規則，複雜模型非同步補判。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：AdTech / 資料流
- 壓力模型：平台防爆型（同步硬規則守門）、寫爆型（特徵事件記錄）

---

## Block 2：為什麼被列上這些

- **平台防爆型** → 反詐是守門，同步判斷要快，只放硬規則。
- **寫爆型** → 每次請求都累積裝置 / 行為特徵，是持續寫入。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一個請求 / 一次點擊進來
   │
   ▼
[反詐] 分兩條路
   ├─ 同步：硬規則（黑名單裝置 / 明顯機器人特徵）──> 立刻擋 / 放
   └─ 非同步：複雜模型（關聯分析 / 行為序列）──> 事後標記可疑

────────── 詐騙流量偽裝成正常人混進來 ──────────

平台想要：擋掉明顯的假流量，又不能拖慢正常請求
```

### 分段解說

- 跟風控決策同構：能秒判的走同步，要慢算的走非同步。
- 同步只放「快而確定」的硬規則。
- 裝置與行為特徵持續累積，餵養非同步模型。

### 痛點分析

- **延遲要求** → 同步判斷要快，不拖慢正常流量。
- **能不能非同步？** 複雜模型可以，事後補判。
- **對抗性** → 詐騙會偽裝進化，規則要能更新。
- **真正的痛**：跟 `s26_riskdecision` 同一套哲學——同步只放硬規則求快，重運算挪到非同步，避免守門變瓶頸。
