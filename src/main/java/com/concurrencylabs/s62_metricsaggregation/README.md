# Metrics Aggregation 系統 · s62_metricsaggregation

> 一句話定位：收集海量時序指標，控制標籤基數、預聚合降採樣，長期資料冷存。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Observability
- 壓力模型：寫爆型 / ingestion（指標寫入）、讀爆型（Dashboard 查詢）
- 數據角色：即時聚合 / 時序資料

---

## Block 2：為什麼被列上這些

- **寫爆 + 讀爆** → 指標持續寫入是寫爆，Dashboard 查詢是讀爆。
- **時序資料角色** → 需要預聚合、降採樣、冷存等時序特有處理。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
每個服務每秒吐一堆指標（QPS / 延遲 / 錯誤率 …）
   │  每個指標還帶一堆標籤（service / region / endpoint / ...）
   ▼
[Metrics] 寫入時序庫
   ├─ 標籤組合爆炸（cardinality）→ 要控制，否則庫撐爆
   ├─ 秒級原始資料 → 短期保留
   └─ 長期 → 降採樣（1 分 / 1 小時）、冷存

────────── 有人加了個高基數標籤（如 userId）──────────

平台想要：別讓標籤組合無限膨脹把時序庫撐爆
```

### 分段解說

- 指標的殺手不是筆數，是「標籤基數（cardinality）」爆炸。
- 一個高基數標籤（如 userId）能讓時間序列數量暴增。
- 長期資料不需要秒級精度，降採樣 + 冷存省成本。

### 痛點分析

- **讀爆還寫爆？** 兩頭都有。
- **隱藏地雷** → 標籤基數爆炸，比流量更致命。
- **能不能降精度？** 長期資料可降採樣。
- **真正的痛**：真正撐爆時序庫的常常不是流量，而是「標籤基數失控」，控制 cardinality 是這題的獨門課題。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s62 寫爆+讀爆,殺手是標籤基數,控 cardinality/降採樣,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：指標寫時序庫,dashboard 查詢
├─ 大方塊：指標寫入 · 時序庫 · 查詢
└─ 撐不住的訊號 → 標籤基數爆炸撐爆庫、寫爆+讀爆

【P2 · cardinality 控制 + 降採樣 + 讀寫分離】約 10,000 QPS
├─ 定義：控制標籤基數,長期降採樣冷存,查詢走預聚合
├─ 大方塊（綁一起）
│   ├─ cardinality 守門(擋高基數標籤)
│   ├─ 秒級短存 + 降採樣長存
│   └─ dashboard 查走預聚合層
├─ 為何綁：撐爆兇手是基數 = 控 cardinality + 降採樣 + 讀寫分離,一起做
└─ 撐不住的訊號 → 指標量再放大、時序庫寫入瓶頸

【P3 · 時序分片 + 預聚合 + 冷存分層】約 50,000+ QPS
├─ 定義：時序庫按 metric 分片,多層預聚合,長期冷存
├─ 大方塊：時序分片 · 多層預聚合 · 冷存分層
└─ 為何綁：規模化 metrics = 分片 + 預聚合 + 冷存,一起做
```
