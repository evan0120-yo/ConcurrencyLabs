# SA.md — s50_impressionclicktracking（曝光 / 點擊事件追蹤）

> 事實文件（SASD）· 結構化分析。描述「曝光點擊追蹤要做什麼」的**業務行為**，平台無關（P1..Pn 共用；各平台怎麼扛量在各自 SD）。
> 與 IoT 心跳同構：海量低價值事件裡混著少數「計費關鍵」事件，**分流對待**——一般曝光可抽樣 / 延遲，計費點擊不可丟。

## 1. Context Diagram

```text
[廣告端] --(曝光/點擊事件: adId, userId, type, ts)--> (事件追蹤) --(ack 收下)--> [廣告端]

(事件追蹤) --(緩衝削峰)--> [MQ]
(事件追蹤) --(一般事件抽樣入湖)--> [資料湖]（可延遲）
(事件追蹤) --(計費事件準確)------> [計費]（不可丟、去重）
```

## 2. DFD（精簡 box flow）

```text
[事件湧入] --> [快速收下]（先 ack，不同步加工）
   ▼
[緩衝削峰] --> [MQ]
   ▼
[分流]
   ├─【一般曝光】量大、價值低 → 可抽樣、可延遲入湖
   └─【計費點擊】關鍵 → 不可丟、去重、準確計費
```

## 3. Process Spec（行為基準，decision table）

```text
[接收]
└─ 快速 ack；洪峰進緩衝，不在請求路徑同步處理

[一般曝光]（放寬）
├─ 可抽樣（保留統計代表性即可）
└─ 可延遲入湖（分析用，秒級/分鐘級延遲可接受）

[計費事件]（不可放寬，核心）
├─ 不可丟（每筆影響金流）
└─ 去重（同一次點擊只計費一次）

[分流]（核心不變量）
└─ 依 billable 旗標分兩條路：一般走抽樣冷入湖、計費走可靠去重
```

## 4. State Transition

```text
（ingestion 型，無業務狀態機；事件的 收下 → 緩衝 → 分流 → 落地 流程）
```

## 5. Data Dictionary（邏輯欄位；實體 DDL 在各平台 SD）

```text
ad_event（廣告事件）
├─ eventId(PK, 計費事件去重用) / adId / userId
├─ type（IMPRESSION|CLICK）/ billable（是否計費）
└─ ts
```
