# Token / Session / Refresh 系統 · s29_tokensessionrefresh

> 一句話定位：登入後每個請求都要驗身分，token 要能撤銷，refresh 要安全，驗證不能變成瓶頸。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：會員 / Gateway / 多租戶
- 壓力模型：讀爆型（每個請求都驗）、平台防爆型（撤銷 / 黑名單）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 幾乎每一個 API 請求都要驗一次身分，驗證是最高頻的讀。
- **平台防爆型** → token 要能被撤銷（登出 / 盜用），黑名單機制要即時又不能拖慢驗證。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
登入後，之後每一個請求都帶 token
   請求 ──> [驗 token] ──> 有效 / 過期 / 已撤銷
                              │
                              └─ 過期 ──> 用 refresh 換新 token

────────── 使用者登出 / token 被盜 ──────────

   └──> 要能「立刻」讓那個 token 失效（撤銷）
```

### 分段解說

- 驗 token 是全站最高頻的動作，快慢直接決定整體延遲。
- 難點在「撤銷」：token 本來設計成無狀態，但登出 / 盜用要能即時作廢。
- refresh 是敏感操作，被盜會延長攻擊者壽命。

### 痛點分析

- **讀爆還寫爆？** 極端讀爆（每請求一次）。
- **能不能撤銷？** 要能，且要快，這跟無狀態設計天生矛盾。
- **一致性** → 撤銷可容忍極短延遲同步。
- **真正的痛**：驗證要極快（不打後端），但撤銷又要即時生效，兩者之間的取捨是核心。
