# Token / Session / Refresh 系統 · s29_tokensessionrefresh

> 一句話定位：登入後每個請求都要驗身分，token 要能撤銷，refresh 要安全，驗證不能變成瓶頸。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：會員 / Gateway / 多租戶
- 壓力模型：讀爆型（每個請求都驗）、平台防爆型（撤銷 / 黑名單）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 幾乎每一個 API 請求都要驗一次身分，驗證是最高頻的讀。
- **平台防爆型** → token 要能被撤銷（登出 / 盜用），黑名單機制要即時又不能拖慢驗證。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
登入後，之後每一個請求都帶 token
   請求 ──> [驗 token] ──> 有效 / 過期 / 已撤銷
                              │
                              └─ 過期 ──> 用 refresh 換新 token

────────── 使用者登出 / token 被盜 ──────────

   └──> 要能「立刻」讓那個 token 失效（撤銷）
```

### 分段解說

- 驗 token 是全站最高頻的動作，快慢直接決定整體延遲。
- 難點在「撤銷」：token 本來設計成無狀態，但登出 / 盜用要能即時作廢。
- refresh 是敏感操作，被盜會延長攻擊者壽命。

### 痛點分析

- **讀爆還寫爆？** 極端讀爆（每請求一次）。
- **能不能撤銷？** 要能，且要快，這跟無狀態設計天生矛盾。
- **一致性** → 撤銷可容忍極短延遲同步。
- **真正的痛**：驗證要極快（不打後端），但撤銷又要即時生效，兩者之間的取捨是核心。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s29 極端讀爆(每請求驗) + 撤銷即時,靠短命token+黑名單,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：session store 驗身分,登出即刪 session
├─ 大方塊：[store] session 查驗 · refresh 換發 · 登出刪除
└─ 撐不住的訊號 → 每請求都驗一次、驗證打 store 成瓶頸

【P2 · 無狀態驗證(本地) + 撤銷黑名單】約 10,000 QPS
├─ 定義：access token 本地驗簽(快),撤銷走短命黑名單
├─ 大方塊（綁一起）
│   ├─ 短命 access token 本地驗簽(不打後端)
│   ├─ [Redis] 撤銷黑名單(登出/盜用即時失效)
│   └─ refresh rotation + 重放偵測
├─ 為何綁：驗證極快 + 撤銷即時是天生矛盾,靠短命+黑名單+rotation一起解
└─ 撐不住的訊號 → 黑名單查詢成新熱點

【P3 · 黑名單分散 + 多層快取驗證】約 50,000+ QPS
├─ 定義：撤銷黑名單分散/邊緣同步,驗證走多層(local+redis)快取
├─ 大方塊：黑名單分散傳播 · 多層快取驗證 · 邊緣驗簽
└─ 為何綁：極端讀爆下,驗證下沉邊緣 + 黑名單傳播,一起做
```
