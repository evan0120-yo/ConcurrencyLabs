# 遊戲房間配對系統 · s09_matchmaking

> 一句話定位：玩家按「開始」，系統依區域 / 段位 / 玩法湊人開局，湊不滿就排隊等待。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：長連線型（等待中要維持連線）、供需匹配型（湊人）

---

## Block 2：為什麼被列上這些

- **長連線型** → 排隊配對期間玩家連線要保持，能即時被拉進房。
- **供需匹配型** → 本質是把「想玩的人」依條件配成一局，與 `s72_dispatchmatching`（派單）同構。
- **博弈 domain** → 對戰 / 房間類遊戲的入口。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
玩家按「開始遊戲」
   │
   ▼
[配對池] 依 區域 / 段位 / 玩法 湊人
   ├─ (湊滿) ──> 開房、進場
   └─ (湊不滿) ──> 排隊等待（連線保持）

────────── 熱門玩法擠爆、冷門玩法湊不到人 ──────────

玩家想要：快點湊到、對手實力接近
營運想要：熱門不崩、冷門也別讓人等到放棄
```

### 分段解說

- 配對品質（實力接近）與等待時間是一組 tradeoff。
- 熱門玩法供給過剩、冷門玩法供給不足，壓力天生不均。
- 等待中維持連線，是這題跟「一次性請求」情境最大的差別。

### 痛點分析

- **供需均衡** → 熱門擠、冷門空，要分池處理。
- **等待可接受** → 玩家願意等一下換更好的對手。
- **連線型** → 等待期間要能主動把人拉進房。
- **真正的痛**：在「配對品質」與「等待時間」之間拿捏，且熱門與冷門要用不同策略。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s09 不涉及錢、**等待可接受**，放寬讓連鎖變少 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：Redis 查可用房間 / 池內湊人，湊不滿排隊
├─ 大方塊：[Redis] 配對池 · 湊人條件比對 · 開房建 room
└─ 撐不住的訊號 → 熱門玩法擠爆、冷門湊不到人，單池撮合變慢

【P2 · Redis queue 分玩法配對】約 10,000 QPS
├─ 定義：按玩法拆成獨立 queue，各自撮合、降低互相干擾
├─ 大方塊（綁一起）
│   ├─ [Redis] 每玩法一條配對 queue
│   ├─ 撮合 worker per queue
│   └─ 等待逾時放寬條件策略
├─ 為何綁：一分 queue，撮合器、逾時策略都得 per-queue 切
└─ 撐不住的訊號 → 區域/段位維度再放大、熱門玩法單 queue 又爆

【P3 · 按區域/段位/玩法拆 pool + 熱門獨立配對服務】約 50,000+ QPS
├─ 定義：多維度拆 pool，熱門玩法獨立配對服務、允許等待
├─ 大方塊：多維度 pool 分片 · 熱門玩法獨立部署 · 冷門放寬/擴桶配
└─ 為何綁：熱門與冷門用不同策略 → 分池 + 獨立服務一起做
```
