# 玩家錢包與帳本 · s06_gamewalletledger

> 一句話定位：玩家的錢在這裡進出，每一筆都要對得起帳、不能算錯、不能重複扣。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：錢包帳本型（核心）、熱點型（大戶 / 活動帳號時）
- 數據角色：帳本流水為 append-only 歷史（非 ingestion 洪流）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 涉及餘額增減，要求冪等、可追溯、對得起帳；錯一分錢都是資損事故。
- **熱點型（條件觸發）** → 大戶 / 主播 / 活動帳號會被高頻扣加，單一 accountId 變成熱點。
- **博弈 domain** → 下注、派彩、儲值、轉入轉出全都打這個錢包，是整個平台的地基。
- **append-only 歷史** → 流水只增不改，是事後對帳與爭議仲裁的唯一真相來源。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
【一筆正常扣款】
玩家下注 100
   │
   ▼
[錢包] 餘額 1000 ──(扣 100)──> 餘額 900
   └─ 同時記一筆流水：-100，原因=下注，單號=xxx

────────── 但真實世界會出這些狀況 ──────────

【狀況 A：重複請求】網路重送，同一筆下注來兩次
   └─ 期望：只扣一次（靠單號辨識，不能扣成 800）

【狀況 B：餘額不足】玩家只有 50 卻要下注 100
   └─ 期望：直接拒絕，餘額不動

【狀況 C：事後對帳】營運問「這 900 是怎麼來的」
   └─ 期望：流水攤開，每一筆加減都有據可查
```

### 分段解說

- 正常路徑很單純：餘額減、記一筆流水。
- 難點不在併發量，而在「正確性」：重送、失敗、對帳三種情況都要不出錯。
- 餘額是「當下結果」，流水是「歷史真相」——兩者必須永遠一致。

### 痛點分析

- **能不能丟？** 一筆都不能丟，也不能重複算。
- **能不能延遲 / 最終一致？** 餘額顯示可以稍微延遲，但「帳目正確」不能妥協。
- **有沒有熱點？** 大戶 / 活動帳號會變成熱 accountId。
- **真正的痛**：這是全平台信任的根。下注、派彩、儲值、提現全靠它算對；一旦錯帳或被非核心高流量拖垮，整個平台就失信。

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s06 是全平台的錢地基，**錯一分錢都是事故** → 正確性壓力最高，共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：餘額表 + 流水表，樂觀鎖(條件更新)扣款，同交易記流水
├─ 大方塊：[DB] balance 條件更新(不透支) · ledger append · bizNo 去重
└─ 撐不住的訊號 → 熱帳號(大戶/活動)高頻打同一 accountId、樂觀鎖衝突升高

【P2 · ledger append-only + 餘額快照】約 10,000 QPS
├─ 定義：以流水為真相(append-only)，餘額改快照，讀寫分離
├─ 大方塊（綁一起）
│   ├─ ledger 純 append（免行鎖競爭）
│   ├─ 餘額快照由流水推導 / 定期物化
│   └─ 讀餘額走快照(可稍延遲)、寫走流水
├─ 為何綁：餘額一改快照，讀語意變「稍舊」、對帳靠流水重算，三件一起
└─ 撐不住的訊號 → 單一超熱 accountId 仍是寫熱點

【P3 · 分 accountId 分表 + 熱帳號限頻】約 50,000 QPS
├─ 定義：帳本按 accountId 分表/分片，熱帳號額外限頻
├─ 大方塊：分片路由 · 每片獨立流水/快照 · 熱帳號 per-account 限頻
└─ 為何綁：一分片，冪等/對帳收斂片內；熱帳號要單獨壓

【P4 · 錢包獨立 cell：讀快照、寫流水】約 100,000+ QPS
├─ 定義：錢包整包獨立 cell 部署，與其他業務資源隔離
├─ 大方塊：獨立 cell(庫/服務) · 讀快照寫流水徹底分層 · 跨 cell 對帳
└─ 為何綁：cell 化 = 部署隔離 + 資料分層 + 跨 cell 一致性，一起變
```
