# 派彩結算系統 · s05_payoutsettlement

> 一句話定位：開獎後批次算出誰中獎、中多少，把錢加回各自錢包，且絕不能重複派或錯帳。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：錢包帳本型（派錢）、任務補償型（批次、冪等、可延遲）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 派彩是往玩家錢包加真錢，要對得起下注、不可錯帳。
- **任務補償型** → 開獎後一次要處理一大批派彩，天生是可非同步、需冪等、要能重跑補償的批次工作。
- **博弈 domain** → 與 `s04_bettingentry` 成對：一個收注、一個派彩。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
開獎！這一局有 3 萬人押注
   │
   ▼
[派彩結算] 批次算出誰中、中多少 ──> 加回各自錢包
   └─ 每筆派彩冪等：同一局重跑也不能派兩次

────────── 一次開獎 = 一大批錢要進帳 ──────────

玩家想要：中獎的錢趕快進來
營運想要：算對、不重複派、跟下注對得起來
```

### 分段解說

- 派彩是「批次」不是「即時」：一次開獎觸發一整批。
- 可以晚幾秒到帳，但絕不能算錯或重複派。
- 結算若卡住，不能反過來拖垮還在收注的入口。

### 痛點分析

- **能不能延遲？** 可以，晚幾秒到帳玩家能接受。
- **能不能錯 / 重複？** 不行，冪等是底線（重跑不能多派）。
- **要不要隔離？** 要，結算慢不能拖垮 `s04_bettingentry`。
- **真正的痛**：允許延遲但不允許錯帳，批次重跑時的「冪等」是整個系統的正確性關鍵。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s05 允許延遲但**絕不能錯帳 / 重複派**，冪等是命脈 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：開獎觸發，同步批次算中獎 → 逐筆冪等派彩加錢
├─ 大方塊：[DB] 撈該局注 · payout 冪等單防重派 · [account] 以 payoutId 加錢
└─ 撐不住的訊號 → 大局(數萬注)一次算完拖很久、與收注入口互搶 DB

【P2 · 派彩進 MQ + consumer 冪等】約 10,000 QPS
├─ 定義：每筆派彩丟 MQ，consumer 冪等消費，削峰又可重跑
├─ 大方塊（綁一起）
│   ├─ [MQ] 每筆(或每批)派彩事件進佇列
│   ├─ consumer 冪等消費（payout 單 exactly-once 效果）
│   └─ 失敗重投 + 對帳（派彩總額核對）
├─ 為何綁：一走非同步，冪等 + 重投 + 對帳三者才能保「不重不漏不錯帳」
└─ 撐不住的訊號 → 單佇列 / 單庫成為瓶頸

【P3 · 按期號 / 桌號分片結算】約 50,000 QPS
├─ 定義：結算按 roundId / tableId 分片，各片獨立並行
├─ 大方塊：分片路由 · 每片獨立 consumer/庫 · 分片內對帳
└─ 為何綁：一分片，冪等鍵 / 對帳範圍都要收斂到片內

【P4 · 結算與下注入口隔離】約 100,000+ QPS
├─ 定義：結算資源與 s04 收注徹底隔離，允許延遲、不可錯帳
├─ 大方塊：獨立結算叢集 · 背壓不回噴收注 · 延遲到帳可接受
└─ 為何綁：隔離 = 資源 + 背壓策略 + SLA 語意，一起變
```
