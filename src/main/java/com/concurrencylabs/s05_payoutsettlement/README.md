# 派彩結算系統 · s05_payoutsettlement

> 一句話定位：開獎後批次算出誰中獎、中多少，把錢加回各自錢包，且絕不能重複派或錯帳。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：錢包帳本型（派錢）、任務補償型（批次、冪等、可延遲）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 派彩是往玩家錢包加真錢，要對得起下注、不可錯帳。
- **任務補償型** → 開獎後一次要處理一大批派彩，天生是可非同步、需冪等、要能重跑補償的批次工作。
- **博弈 domain** → 與 `s04_bettingentry` 成對：一個收注、一個派彩。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
開獎！這一局有 3 萬人押注
   │
   ▼
[派彩結算] 批次算出誰中、中多少 ──> 加回各自錢包
   └─ 每筆派彩冪等：同一局重跑也不能派兩次

────────── 一次開獎 = 一大批錢要進帳 ──────────

玩家想要：中獎的錢趕快進來
營運想要：算對、不重複派、跟下注對得起來
```

### 分段解說

- 派彩是「批次」不是「即時」：一次開獎觸發一整批。
- 可以晚幾秒到帳，但絕不能算錯或重複派。
- 結算若卡住，不能反過來拖垮還在收注的入口。

### 痛點分析

- **能不能延遲？** 可以，晚幾秒到帳玩家能接受。
- **能不能錯 / 重複？** 不行，冪等是底線（重跑不能多派）。
- **要不要隔離？** 要，結算慢不能拖垮 `s04_bettingentry`。
- **真正的痛**：允許延遲但不允許錯帳，批次重跑時的「冪等」是整個系統的正確性關鍵。
