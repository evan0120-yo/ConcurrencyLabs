# 多租戶 Observability 隔離 · s65_multitenantobservability

> 一句話定位：一套可觀測性平台服務多客戶，查詢與資料要隔離，大客戶不能拖垮小客戶。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Observability
- 壓力模型：多租戶型（核心）、平台防爆型（hot tenant 查詢隔離）

---

## Block 2：為什麼被列上這些

- **多租戶型** → 多客戶共用一套 observability 平台，資料與查詢要隔離。
- **平台防爆型** → 大客戶的重查詢 / 巨量資料不能拖垮小客戶。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一套 observability 平台服務很多客戶
   客戶 A ─┐
   客戶 B ─┼──> [平台] 存 log / metrics / trace + 查詢
   大客戶 C┘        │
                    ├─ 資料隔離：A 查不到 B 的資料
                    └─ 資源隔離：C 跑一個掃全量的重查詢
                         └─ 不能因此拖垮 A / B 的查詢

────────── 大客戶下一個掃 30 天全量的查詢 ──────────

小客戶想要：我查我的最近 1 小時，別被隔壁的重查詢卡住
```

### 分段解說

- 跟 B2B 多租戶同構，但壓力在「查詢」與「資料量」。
- 大客戶資料多、查詢重，容易吃光共用查詢資源。
- 大到一定程度要獨立 shard / cell。

### 痛點分析

- **資料隔離** → 跨租戶絕不可見。
- **資源隔離** → hot tenant 查詢要獨立資源 / quota。
- **分級** → 大客戶獨立 shard / cell。
- **真正的痛**：又是「吵鬧鄰居」——大客戶的重查詢不能拖垮小客戶，查詢資源與資料量都要按租戶隔離。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s65 多租戶查詢/資料隔離(大客戶不拖垮小),放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：多客戶共用,tenantId 資料隔離
├─ 大方塊：tenantId 資料隔離 · 共用查詢池
└─ 撐不住的訊號 → 大客戶掃 30 天全量重查詢拖垮小客戶

【P2 · per-tenant 查詢 quota + hot tenant 隔離】約 10,000 QPS
├─ 定義：每租戶查詢 quota,大客戶重查詢獨立資源/限流
├─ 大方塊（綁一起）
│   ├─ per-tenant 查詢 quota/併發限制
│   ├─ 大客戶重查詢獨立資源/限流
│   └─ 資料隔離強制(跨租戶不可見)
├─ 為何綁：吵鬧鄰居 = 查詢 quota + hot 隔離 + 資料隔離,一起做
└─ 撐不住的訊號 → 大客戶資料量/查詢量共用池裝不下

【P3 · 大客戶獨立 shard/cell + 查詢資源分級】約 50,000+ QPS
├─ 定義：大客戶分級到獨立 shard/cell,查詢資源按租戶分級
├─ 大方塊：租戶分級路由 · 大客戶獨立 shard/cell · 查詢資源分級
└─ 為何綁：分級隔離 = 分級路由 + 獨立部署,一起做
```
