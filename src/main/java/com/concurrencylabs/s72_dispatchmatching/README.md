# 外送 / 派單 / 供需匹配 · s72_dispatchmatching

> 一句話定位：把訂單派給最適合的司機 / 騎手，區域供需不均，熱區要獨立調度、可近似計算。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：地理 / 派單
- 壓力模型：供需匹配型（核心）、熱點型（熱區）、長連線型（司機位置上報）

---

## Block 2：為什麼被列上這些

- **供需匹配型** → 把「訂單」與「司機」即時配對，與 `s09_matchmaking` 同構。
- **熱點型** → 熱門區域（市中心 / 用餐時段）供需壓力集中。
- **長連線型** → 司機端持續上報位置，是移動的即時資料流。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一張外送單進來
   │
   ▼
[派單] 在附近找最適合的司機
   ├─ 考量：距離 / 順路 / 司機負載 / ETA
   ├─ 熱區（用餐尖峰）→ 訂單爆多、司機不夠 → 獨立調度
   └─ 冷區 → 司機少、訂單稀 → 別讓人等太久

────────── 中午 12 點，全城同時叫外送 ──────────

顧客想要：快點派到、ETA 準
司機想要：派給我的單是順路的
平台想要：整體效率最高（別讓司機空跑）
```

### 分段解說

- 派單是「雙邊匹配」：訂單找司機、司機找訂單。
- 供需嚴重不均：熱區訂單爆炸、司機不足。
- ETA / 價格這類計算在尖峰可用近似值求快。

### 痛點分析

- **有沒有熱點？** 熱區在尖峰極度集中。
- **供需匹配** → 雙邊即時配對，要兼顧多方目標。
- **能不能近似？** ETA / 定價尖峰可近似。
- **真正的痛**：跟 `s09_matchmaking` 同構的雙邊匹配，但多了「地理供需不均」，熱區尖峰要獨立調度才不會崩。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s72 雙邊匹配 + 地理供需不均,熱區獨立調度/近似,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：訂單附近找司機匹配指派
├─ 大方塊：附近候選 · 匹配打分 · 指派
└─ 撐不住的訊號 → 中午 12 點全城叫外送、熱區訂單爆司機不足

【P2 · 雙邊匹配 + 熱區獨立調度 + 近似 ETA】約 10,000 QPS
├─ 定義：訂單↔司機雙邊匹配,熱區獨立調度/排隊,ETA/定價近似
├─ 大方塊（綁一起）
│   ├─ 雙邊匹配打分(距離/順路/負載/ETA)
│   ├─ 熱區獨立調度池 + 排隊
│   └─ ETA/定價近似 + 司機位置流
├─ 為何綁：地理供需不均 = 雙邊匹配 + 熱區獨立 + 近似,一起做
└─ 撐不住的訊號 → 熱區尖峰再放大、單調度器瓶頸

【P3 · 按區域分片調度 + 熱區獨立服務 + 司機池分散】約 50,000+ QPS
├─ 定義：調度按區域分片並行,熱區獨立調度服務,司機池分散
├─ 大方塊：區域分片調度 · 熱區獨立服務 · 司機池分散
└─ 為何綁：規模化派單 = 分片 + 熱區獨立 + 池分散,一起做
```
