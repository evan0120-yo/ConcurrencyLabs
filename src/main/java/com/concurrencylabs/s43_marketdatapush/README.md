# Market Data 行情推送 · s43_marketdatapush

> 一句話定位：把最新成交、盤口、K 線即時推給海量訂閱者，熱門 symbol 訂閱數巨大。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：交易所 / Web3
- 壓力模型：長連線型（海量訂閱連線）、低延遲型（行情要即時）、讀爆型（同一行情廣播給眾人）

---

## Block 2：為什麼被列上這些

- **長連線型** → 每個看盤的人一條長連線持續收行情。
- **低延遲型** → 行情慢一拍就失去價值，尤其對交易者。
- **讀爆型** → 同一份行情要廣播給大量訂閱者，是一寫多讀的極致。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一筆成交發生
   │
   ▼
[行情推送] 要把它推給所有訂閱這個 symbol 的人
   ├─ 熱門 symbol → 幾十萬人同時訂閱
   └─ 分層推送：tick 級 / 秒級 / 聚合級（不同人要不同精細度）

────────── 行情劇烈波動，更新頻率飆高 ──────────

交易者想要：越即時越好
一般看客：秒級 / 聚合就夠
```

### 分段解說

- 一份行情要扇出給海量訂閱者，是典型一寫多讀。
- 不同訂閱者要不同精細度：交易者要 tick、看客要聚合。
- 波動時更新頻率暴增，推送壓力跟著飆。

### 痛點分析

- **長連線** → 海量訂閱連線同時維持。
- **延遲要求** → 行情即時性是價值核心。
- **能不能分層 / 降頻？** 可以，非交易者可收聚合 / 降頻資料。
- **真正的痛**：同一份高頻行情要扇出給海量連線，靠「分層推送」把不需要 tick 的人降頻，才扛得住。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+（量體為訂閱連線數）。每跳一個平台 = 一次架構改造。
> s43 一寫多讀扇出、非交易者可降頻,分層放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：成交/盤口即時推給訂閱者
├─ 大方塊：[gateway] 長連線 · 訂閱推送
└─ 撐不住的訊號 → 熱門 symbol 幾十萬人訂閱、波動更新頻率飆

【P2 · 分層推送 + 扇出優化】約 10,000 QPS
├─ 定義：行情分 tick/秒級/聚合層,非交易者降頻,一寫多讀扇出
├─ 大方塊（綁一起）
│   ├─ 分層生成(tick/秒級/聚合)
│   ├─ 依訂閱精細度扇出(交易者tick/看客聚合)
│   └─ 波動時降頻層加大節流
├─ 為何綁：扛海量扇出 = 分層 + 精細度路由 + 節流,一起做
└─ 撐不住的訊號 → 連線數再翻倍、單 gateway 連線吃滿

【P3 · gateway 分區 + 熱門 symbol 獨立推送鏈路】約 50,000+ QPS
├─ 定義：連線 gateway 分區水平擴,熱門 symbol 獨立推送鏈路
├─ 大方塊：gateway 分區 · 熱門 symbol 獨立鏈路 · 訂閱分散
└─ 為何綁：海量長連線 = gateway 分區 + 熱門獨立,一起做
```
