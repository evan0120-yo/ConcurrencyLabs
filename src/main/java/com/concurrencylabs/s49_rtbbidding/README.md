# RTB 即時競價系統 · s49_rtbbidding

> 一句話定位：廣告曝光機會來時，在數十毫秒內算出出價回應，超時就等於棄權。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：AdTech / 資料流
- 壓力模型：低延遲型（核心）、平台防爆型（嚴格 timeout / fallback）

---

## Block 2：為什麼被列上這些

- **低延遲型** → RTB 有硬性 timeout（常是幾十毫秒），慢一點就出局。
- **平台防爆型** → 超時要能乾脆放棄 / fallback，不能拖累整體。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者打開一個 App / 網頁，出現一個廣告位
   │
   ▼
[競價] 交易所在 ~50ms 內問所有買方：「這個曝光你出多少？」
   ├─ (我算得完 + 有興趣) ──> 回出價
   └─ (算不完 / 超時) ──> 等於棄權，這次機會沒了

────────── 每秒有海量曝光機會，每個都在倒數 ──────────

買方想要：算得夠準又夠快，別因為慢而錯過好曝光
```

### 分段解說

- RTB 的殘酷在於「硬 timeout」：超時的回應直接作廢。
- 所以寧可用「夠好且來得及」的答案，也不要「最佳但太慢」。
- 候選廣告、素材要預先備好，臨場只做輕量計算。

### 痛點分析

- **延遲要求** → 硬性 timeout，是這題的天條。
- **超時策略** → 來不及就棄權 / fallback，不硬撐。
- **平均還是尖峰？** 持續高吞吐 + 每筆都趕時間。
- **真正的痛**：在幾十毫秒的硬邊界內完成決策，逼你把重活預算好，臨場只留最輕量的計算。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s49 硬 timeout 幾十毫秒、超時即棄權,重活預算好臨場輕算 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：競價請求即時撈候選算出價,硬 timeout
├─ 大方塊：候選匹配 · 出價計算 · timeout 棄權
└─ 撐不住的訊號 → 幾十毫秒硬邊界、臨場重算來不及

【P2 · 候選預算 + 輕量計算 + fallback】約 10,000 QPS
├─ 定義：定向/候選/素材預備,臨場只輕算,超時乾脆 no-bid
├─ 大方塊（綁一起）
│   ├─ 定向候選預備(免臨場重查)
│   ├─ 臨場輕量出價計算
│   └─ 硬 timeout + no-bid fallback
├─ 為何綁：卡死 deadline = 重活預算 + 輕算 + fallback,一起做
└─ 撐不住的訊號 → 持續高吞吐、單節點延遲抖動

【P3 · 分片並行 + 邊緣就近部署降延遲】約 50,000+ QPS
├─ 定義：競價服務分片並行,就近邊緣部署壓低網路延遲
├─ 大方塊：競價分片 · 邊緣就近部署 · 候選本地快取
└─ 為何綁：規模化低延遲 = 分片 + 邊緣 + 本地快取,一起做
```
