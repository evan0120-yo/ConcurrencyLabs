# 商品詳情頁爆流量 · s17_productdetailtraffic

> 一句話定位：爆款商品被百萬人同時看同一頁，幾乎全是讀，靜態內容與動態庫存要分開對待。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：讀爆型（核心）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 同一個爆款頁被海量使用者高頻讀取，是最典型的讀爆場景。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
爆款商品，百萬人同時看同一頁
   │ (幾乎全是讀)
   ▼
[商品頁]
   ├─ 圖文 / 價格 → 很少變 ──> 可重度快取
   └─ 庫存數字   → 一直變 ──> 顯示可近似

────────── 直播帶貨一喊，流量瞬間灌爆同一頁 ──────────
```

### 分段解說

- 頁面內容其實分兩種：幾乎不變的（圖文）與一直變的（庫存）。
- 大部分流量看的是「不變的那部分」，可以大力快取。
- 庫存顯示不用精準到個位數，近似即可。

### 痛點分析

- **讀爆還寫爆？** 幾乎純讀爆。
- **靜態 vs 動態** → 拆開處理，別讓庫存變動拖累整頁。
- **能不能近似？** 庫存顯示可近似（「剩不多」）。
- **真正的痛**：同一頁裡「可快取的靜態」與「一直變的庫存」混在一起，要拆開才扛得住讀爆。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s17 幾乎純讀爆、庫存可近似,大力快取即可 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：商品資料放 Redis cache,庫存近似顯示
├─ 大方塊：[Redis] 靜態內容快取 · 庫存近似值 · 回源填 cache
└─ 撐不住的訊號 → 爆款頁被百萬人讀、熱 key 快取擊穿風險

【P2 · CDN + 多層 cache】約 10,000 QPS
├─ 定義：靜態內容上 CDN + 多層快取(邊緣→應用),大降回源
├─ 大方塊（綁一起）
│   ├─ CDN 扛靜態圖文
│   ├─ 多層 cache(邊緣/應用)
│   └─ 防擊穿(單飛回源、TTL 抖動)
├─ 為何綁：多層快取 + 防擊穿是「扛讀爆」的一整套
└─ 撐不住的訊號 → 超級爆款單 key 仍可能打穿到源

【P3 · 熱門商品 local cache + 靜態頁/動態小接口分離】約 50,000+ QPS
├─ 定義：熱門商品進程內 local cache;整頁靜態化,只留庫存等動態小接口
├─ 大方塊：熱 key local cache · 靜態頁生成 · 動態小接口(庫存近似)分離
└─ 為何綁：把「不變的整頁」與「一直變的庫存」徹底拆開,一起做
```
