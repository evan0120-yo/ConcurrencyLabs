# 活動排行榜系統 · s07_leaderboard

> 一句話定位：玩家分數不斷變動即時排名，所有人一直刷「Top N 是誰、我第幾名」。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：熱點計數型（分數更新）、讀爆型（大家看榜）

---

## Block 2：為什麼被列上這些

- **熱點計數型** → 大量玩家分數同時變動，且 Top N 是被集中關注的熱點。
- **讀爆型** → 看榜的人遠多於更新分數的人，讀壓力是主戲。
- **博弈 domain** → 活動 / 賽季常見玩法。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
【寫】玩家得分變動 ──> 更新排名分數（很多人同時變動）
【讀】所有人一直刷「Top 100 是誰、我第幾名」
   ▼
[排行榜]
   ├─ Top N   ──> 大家都想看（讀爆熱點）
   └─ 我的名次 ──> 個人查詢

────────── 榜還分：日榜 / 週榜 / 區服榜 ──────────
```

### 分段解說

- 寫來自分數變動，讀來自無數次的圍觀。
- Top N 被反覆讀，是最熱的一小段資料。
- 名次可以「準到大概」——即時性換取扛得住流量。

### 痛點分析

- **讀爆還寫爆？** 讀遠多於寫。
- **能不能延遲 / 近似？** 排名可稍微延遲、近似，多數場景可接受。
- **有沒有熱點？** Top N 是熱點；榜的維度（日/週/區服）會放大資料量。
- **真正的痛**：更新頻繁但真正被看的只有頂端一小段，要把「熱的那段」預算好，別讓全榜即時排序拖垮。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s07 是**讀爆 + 排名可近似**（不涉及錢），放寬讓連鎖變少 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：Redis ZSet 直接更新分數、直接讀 Top N / 名次
├─ 大方塊：[Redis] ZINCRBY 更新 · ZREVRANGE 讀榜
└─ 撐不住的訊號 → 分數高頻變動、Top N 被圍觀讀爆

【P2 · 寫進 MQ 批量更新 + 排名近似】約 10,000 QPS
├─ 定義：得分事件先進 MQ，批量合併更新 ZSet；排名接受近似
├─ 大方塊（綁一起）
│   ├─ [MQ] 得分事件削峰、批量 flush 進 ZSet
│   └─ 排名 / 名次語意改「近似、稍延遲」【關鍵放寬】
├─ 為何綁：一批量，讀到的排名就有延遲 → 名次語意必須接受近似
└─ 撐不住的訊號 → 多維度榜(日/週/區服)放大資料量、全榜排序變重

【P3 · 分維度榜 + Top N 預計算（冷名次延遲）】約 50,000+ QPS
├─ 定義：榜按日/週/區服分區；Top N 預算好，非 Top N 名次延遲/近似
├─ 大方塊：各維度獨立 board 分片 · Top N 預計算快取 · 冷名次按需/延遲
└─ 為何綁：真正被看的只有頂端一小段 → 預算 Top N + 冷段降級，一起做
```
