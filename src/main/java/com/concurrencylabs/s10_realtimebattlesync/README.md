# 即時對戰狀態同步 · s10_realtimebattlesync

> 一句話定位：同房間玩家各自操作，動作要幾乎同時同步給所有人，結果最後落地存檔。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：長連線型（每人一條連線）、低延遲型（動作同步）

---

## Block 2：為什麼被列上這些

- **長連線型** → 每個玩家一條持續連線，狀態要即時互推。
- **低延遲型** → 一個動作要讓房內所有人幾乎同時看到，慢了就「卡」。
- **博弈 domain** → 即時對戰 / 多人房間類遊戲。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
同一房間 4 名玩家，各自操作
   玩家動作 ──> [房間] ──即時廣播──> 其他 3 人畫面同步
   │
   └─ 對局結束 ──> 結果落地存檔

────────── 一個動作要讓房內所有人幾乎同時看到 ──────────

玩家想要：不卡、不延遲、大家畫面一致
系統面對：大量房間 × 每房高頻狀態更新
```

### 分段解說

- 房間內的事件要有順序，不能你先我後亂掉。
- 觀戰者（旁觀）可以用比玩家更低的頻率接收。
- 對局過程可在記憶體高速跑，只有結果需要可靠落地。

### 痛點分析

- **延遲要求** → 極低，體感是即時。
- **順序性** → 單一房間內事件順序不能亂。
- **長連線** → 大量連線同時維持。
- **真正的痛**：房間數 × 每房高頻更新造成的長連線壓力，同時要保單房間內動作順序與低延遲。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s10 是**海量長連線 + 極低延遲 + 單房順序**，複雜度高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：WebSocket gateway 推狀態，房內廣播
├─ 大方塊：[WS gateway] 長連線管理 · 房內事件廣播
└─ 撐不住的訊號 → 連線數上升、房內高頻更新、狀態同步壓力

【P2 · 房間狀態記憶體 + 結果落 DB】約 10,000 QPS
├─ 定義：對局狀態在記憶體高速跑，只有結果可靠落地
├─ 大方塊（綁一起）
│   ├─ 房間狀態放記憶體(免每幀持久化)
│   └─ 對局結束結果落 [DB]（冪等一次）
├─ 為何綁：過程走記憶體 → 落地只在結束、且要冪等，兩件一起
└─ 撐不住的訊號 → 單房高頻事件、跨房順序/隔離需求

【P3 · 房間 actor / shard 單房順序處理】約 50,000 QPS
├─ 定義：每房一個 actor/shard，房內事件單線 FIFO，跨房並行
├─ 大方塊：房間 actor 化 · 單房順序保證 · 房按 shard 分散
└─ 為何綁：順序保證 = actor 化 + 分片路由，一起變

【P4 · gateway 分區 + 觀戰事件降頻】約 100,000+ QPS
├─ 定義：連線 gateway 分區水平擴，觀戰走降頻通道
├─ 大方塊：gateway 分區/分層 · 觀戰降頻推送 · 玩家與觀戰分流
└─ 為何綁：海量連線 = gateway 分區 + 觀戰降頻，一起做
```
