# 充值入帳系統 · s46_depositcrediting

> 一句話定位：掃鏈上到帳事件，確認後把對應資產入帳到使用者，入帳必須冪等、不可重複加。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：交易所 / Web3
- 壓力模型：錢包帳本型（入帳真資產）、任務補償型（掃鏈 / 確認 / 冪等）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 入帳是往使用者帳戶加真資產，錯 / 重複都是資損。
- **任務補償型** → 靠掃鏈事件觸發，事件會重放，需冪等與確認機制。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者往充值地址轉入 1 BTC
   │
   ▼
[掃鏈] 偵測到這筆轉入 ──> 等足夠確認數 ──> 入帳
   ├─ (確認足夠) ──> 帳戶 +1 BTC（同一筆只能加一次）
   └─ (確認不足) ──> 繼續等，先不入帳

────────── 掃鏈事件可能重放、鏈可能分叉回滾 ──────────

使用者想要：到帳快
平台想要：不能因為分叉 / 重放而重複入帳
```

### 分段解說

- 入帳前要等「足夠確認數」，避免鏈回滾造成假到帳。
- 掃鏈事件會重放，同一筆轉入不能被入帳兩次。
- 這是真資產，錯帳直接等於虧錢。

### 痛點分析

- **能不能重複？** 絕不能，冪等是命脈。
- **確認機制** → 要等足夠確認避免回滾。
- **能不能延遲？** 到帳可稍延遲（等確認），但不可錯。
- **真正的痛**：在「事件會重放、鏈會回滾」的不確定環境下，保證每筆充值只入帳一次、且金額正確。

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s46 真資產但入帳委派錢包,核心是確認+冪等+回滾,可延遲不可錯 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：掃鏈偵測轉入,等確認後冪等入帳
├─ 大方塊：掃鏈 · 確認數判定 · txHash 冪等入帳
└─ 撐不住的訊號 → 掃鏈事件重放、鏈回滾、多幣種多鏈掃描壓力

【P2 · 確認機制 + 冪等 + 回滾處理】約 10,000 QPS
├─ 定義：足夠確認才入帳,txHash 冪等,回滾/分叉作廢
├─ 大方塊（綁一起）
│   ├─ 確認數門檻(避假到帳)
│   ├─ txHash 冪等(重放不重複加)
│   └─ 回滾/孤塊處理(未確認前分叉作廢)
├─ 為何綁：不確定環境保只入一次 = 確認 + 冪等 + 回滾,一起做
└─ 撐不住的訊號 → 多鏈多幣種掃描與確認量體大

【P3 · 多鏈分片並行掃描 + 熱門幣種分流】約 50,000+ QPS
├─ 定義：掃鏈按鏈/幣種分片並行,熱門幣種獨立入帳鏈路
├─ 大方塊：多鏈分片掃描 · 熱幣種分流 · 入帳並行
└─ 為何綁：規模化入帳 = 分片掃描 + 熱幣種分流,一起做
```
