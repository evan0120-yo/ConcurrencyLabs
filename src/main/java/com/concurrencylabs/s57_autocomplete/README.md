# 搜尋 Autocomplete · s57_autocomplete

> 一句話定位：使用者每敲一個字就要即時跳出候選詞，頻率極高、延遲要極低，熱詞可預生成。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：社群 / Feed / 搜尋 / Marketplace
- 壓力模型：讀爆型（每敲一鍵一次查）、低延遲型（要跟得上打字）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 每個按鍵都可能觸發一次查詢，頻率是普通搜尋的好幾倍。
- **低延遲型** → 慢了就跟不上打字，體驗立刻崩。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者在搜尋框打字
   打「i」→ 跳候選；打「ip」→ 更新候選；打「iph」→ 再更新
   │ (每一鍵都是一次請求)
   ▼
[Autocomplete] 依前綴給最可能的候選
   ├─ 熱門前綴（如「iph」→ iphone）→ 可預生成
   └─ 前端 debounce → 別每個鍵都送

────────── 大量使用者同時打字 ──────────

使用者想要：候選跟得上手速、秒跳
```

### 分段解說

- 這是「按鍵級」的查詢頻率，比一般搜尋高一個量級。
- 熱門前綴的候選其實是固定的，可預先算好。
- 前端 debounce 能先削掉一部分無謂請求。

### 痛點分析

- **讀爆還寫爆？** 極端讀爆（按鍵級）。
- **延遲要求** → 要跟得上打字，極低延遲。
- **能不能預生成？** 熱詞前綴可預生成。
- **真正的痛**：按鍵級的查詢頻率放大了讀爆，靠「熱門前綴預生成 + 前端 debounce」把真正打到後端的量壓下來。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s57 按鍵級極端讀爆、熱詞可預生成,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：前綴索引查候選
├─ 大方塊：前綴索引 · top-K 候選
└─ 撐不住的訊號 → 按鍵級查詢頻率、每敲一鍵一次、延遲要極低

【P2 · 熱詞前綴預生成 + debounce + 快取】約 10,000 QPS
├─ 定義：熱門前綴預生成放快取,前端 debounce 削量
├─ 大方塊（綁一起）
│   ├─ 熱門前綴預生成(候選固定)
│   ├─ 前端 debounce(削無謂請求)
│   └─ [Redis] 前綴候選快取
├─ 為何綁：按鍵級讀爆 = 預生成 + debounce + 快取,一起壓量
└─ 撐不住的訊號 → 長尾前綴多、單索引節點吃緊

【P3 · 前綴索引分片 + 邊緣就近 + 多層快取】約 50,000+ QPS
├─ 定義：前綴索引分片,候選就近邊緣,多層(local+redis)快取
├─ 大方塊：前綴索引分片 · 邊緣就近 · 多層快取
└─ 為何綁：極低延遲讀爆 = 分片 + 邊緣 + 多層快取,一起做
```
