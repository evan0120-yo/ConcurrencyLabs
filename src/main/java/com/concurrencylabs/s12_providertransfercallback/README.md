# 遊戲商轉入轉出 / 回調結算 · s12_providertransfercallback

> 一句話定位：玩家在平台錢包與遊戲商之間搬錢，靠對方 callback 對帳，兩邊金額要最終一致。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：錢包帳本型（真錢進出）、任務補償型（callback 冪等 / 補償）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 錢在平台與遊戲商之間流動，涉及真錢、不可錯帳。
- **任務補償型** → 靠 callback 回報結果，callback 會遲到 / 重送 / 遺失，需要冪等與補償對帳。
- **博弈 domain** → 與 `s11_gameproviderapi` 成對：一個是玩、一個是搬錢。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
玩家把錢從平台錢包轉進遊戲商（或轉回）
   │
   ▼
[平台錢包] 扣款 ──> [遊戲商] 入帳
   │                     │
   └──── 等 callback ────┘（遊戲商回報結果）
        ├─ (成功 callback) ──> 兩邊對平
        └─ (遲到 / 重送 / 遺失) ──> 補償對帳補上
```

### 分段解說

- 轉帳跨了兩個系統，中間狀態靠 callback 收斂。
- callback 不可靠：可能重送（要冪等）、可能遺失（要主動補償）。
- 過程中可以短暫不一致，但最終必須對得起來。

### 痛點分析

- **能不能最終一致？** 可以，過程允許短暫不一致，結果必須一致。
- **callback 可靠嗎？** 不可靠，重送要冪等、遺失要補償。
- **涉及真錢** → 不可錯帳、不可重複加。
- **真正的痛**：跨系統搬真錢，用一個「不可靠的 callback」當收斂機制，冪等與補償是命脈。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s12 是**真錢跨系統 + 不可靠 callback**，冪等/補償/對帳壓力高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：轉入轉出同步扣加款
├─ 大方塊：[account] 扣加平台錢包 · 呼叫 provider 入帳 · transferNo 去重
└─ 撐不住的訊號 → callback 遲到/重送/遺失、跨系統中間狀態難收斂

【P2 · callback 冪等 + 失敗補償】約 10,000 QPS
├─ 定義：靠 callback 收斂，callback 冪等，逾時主動補償對帳
├─ 大方塊（綁一起）
│   ├─ callback 冪等（同 transferNo 只處理一次）
│   ├─ 逾時主動查詢對方最終狀態
│   └─ 補入帳 / 回滾退款（真錢不重複加）
├─ 為何綁：不可靠 callback → 冪等 + 主動補償 + 回滾，三件保最終一致
└─ 撐不住的訊號 → 大量 callback 湧入、單一消費瓶頸

【P3 · callback 分區消費】約 50,000 QPS
├─ 定義：callback 按 provider / 帳號分區並行消費
├─ 大方塊：分區路由 · 每區獨立冪等/補償 · 順序保證 per 帳號
└─ 為何綁：一分區，冪等鍵/對帳/順序收斂到區內

【P4 · 平台錢包與 provider 狀態最終一致】約 100,000+ QPS
├─ 定義：對帳與補償規模化，平台與各 provider 帳最終對平
├─ 大方塊：定期對帳批次 · 差異自動補償 · 跨系統帳務隔離
└─ 為何綁：規模化最終一致 = 對帳 + 自動補償 + 隔離，一起做
```
