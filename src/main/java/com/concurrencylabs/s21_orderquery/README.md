# 訂單查詢中心 · s21_orderquery

> 一句話定位：使用者與客服不斷查訂單狀態與歷史，讀多寫少，近期熱、歷史冷。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：讀爆型（核心）
- 數據角色：查詢 / 分析讀（多維度查詢）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 查詢遠多於寫入（寫入來自下單與狀態變更）。
- **查詢讀角色** → 需要依狀態 / 時間 / 商品等多維度查，跟「寫訂單」是不同的存取型態。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者 / 客服一直查「我的訂單、狀態、歷史」
   │ (讀多寫少；寫來自下單 / 狀態變更)
   ▼
[訂單查詢]
   ├─ 近期訂單 → 熱、常查
   └─ 歷史訂單 → 冷、偶爾查

────────── 大促後客服湧入查單、對帳 ──────────
```

### 分段解說

- 訂單「寫」的路徑（下單）跟「查」的路徑，型態完全不同。
- 查詢維度多，且近期單被查得最兇。
- 歷史單量大但很少被碰，是冷資料。

### 痛點分析

- **讀爆還寫爆？** 讀爆。
- **冷熱分離** → 近期熱、歷史冷，該分開放。
- **不該打主庫** → 查詢流量不能拖垮下單的寫入。
- **真正的痛**：讀寫型態差異大，如何讓「查詢」走自己的路，不去干擾「下單」的寫入正確性與效能。
