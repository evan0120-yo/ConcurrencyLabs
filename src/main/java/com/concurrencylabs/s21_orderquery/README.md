# 訂單查詢中心 · s21_orderquery

> 一句話定位：使用者與客服不斷查訂單狀態與歷史，讀多寫少，近期熱、歷史冷。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：讀爆型（核心）
- 數據角色：查詢 / 分析讀（多維度查詢）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 查詢遠多於寫入（寫入來自下單與狀態變更）。
- **查詢讀角色** → 需要依狀態 / 時間 / 商品等多維度查，跟「寫訂單」是不同的存取型態。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者 / 客服一直查「我的訂單、狀態、歷史」
   │ (讀多寫少；寫來自下單 / 狀態變更)
   ▼
[訂單查詢]
   ├─ 近期訂單 → 熱、常查
   └─ 歷史訂單 → 冷、偶爾查

────────── 大促後客服湧入查單、對帳 ──────────
```

### 分段解說

- 訂單「寫」的路徑（下單）跟「查」的路徑，型態完全不同。
- 查詢維度多，且近期單被查得最兇。
- 歷史單量大但很少被碰，是冷資料。

### 痛點分析

- **讀爆還寫爆？** 讀爆。
- **冷熱分離** → 近期熱、歷史冷，該分開放。
- **不該打主庫** → 查詢流量不能拖垮下單的寫入。
- **真正的痛**：讀寫型態差異大，如何讓「查詢」走自己的路，不去干擾「下單」的寫入正確性與效能。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s21 讀多寫少、查詢可稍舊,走 CQRS 讀側放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：查 DB index,依單號/使用者查
├─ 大方塊：[DB] 查詢索引 · 近期單常查
└─ 撐不住的訊號 → 查詢流量打到下單寫主庫、多維度查慢

【P2 · CQRS read model + cache】約 10,000 QPS
├─ 定義：下單事件投影出 read model,查詢走副本 + cache
├─ 大方塊（綁一起）
│   ├─ 寫側事件投影 read model(與寫主庫解耦)
│   ├─ 查詢走 read model + cache
│   └─ 容忍稍舊(投影秒級延遲)
├─ 為何綁：CQRS 投影 + 讀副本 + 容忍延遲,一起把查詢移出寫路徑
└─ 撐不住的訊號 → 訂單量大、多維度查與歷史單量體上升

【P3 · 訂單分表 + 走 read DB/ES + 冷熱分離】約 50,000+ QPS
├─ 定義：訂單分表,多維度查走 ES,歷史單冷熱分離歸檔
├─ 大方塊：訂單分表 · 多維度走 ES · 近期熱/歷史冷分離
└─ 為何綁：讀規模化 = 分表 + ES + 冷熱分離,一起做
```
