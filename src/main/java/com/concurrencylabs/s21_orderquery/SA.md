# SA.md — s21_orderquery（訂單查詢中心）

> 事實文件（SASD）· 結構化分析。描述「訂單查詢要做什麼」的**業務行為**，平台無關（P1..Pn 共用；各平台怎麼扛量在各自 SD）。
> 特性：**讀多寫少**（寫來自下單 / 狀態變更）；多維度查詢，近期熱、歷史冷。查詢流量不得干擾下單的寫。

## 1. Context Diagram

```text
[使用者 / 客服] --(查訂單: 依狀態/時間/商品/單號)--> (查詢中心) --(訂單 / 列表)--> [查詢方]

[下單 / 狀態變更事件] --(投影)--> (查詢中心 read model)
(查詢中心) --(讀查詢模型，不打寫主庫)--> [read DB / cache / ES]
```

## 2. DFD（精簡 box flow）

```text
【寫側（來源）】[下單/狀態變更] --(事件)--> [投影更新 read model]（與寫主庫解耦）

【讀側（本情境主戲）】
[查詢請求]
   ├─(單號 / 近期)──> 走熱 read model / cache（常查、快）
   ├─(多維度)──> 走查詢索引（狀態/時間/商品；如 ES）
   └─(歷史)──> 走冷儲存（歸檔區，偶爾查、可較慢）
   ▼
回應（允許稍舊：read model 同步有秒級延遲）
```

## 3. Process Spec（行為基準，decision table）

```text
[讀寫分離]（核心不變量）
└─ 查詢一律走 read model / 副本，絕不打下單的寫主庫

[查詢維度]
├─ 單號 / userId → 直查（熱）
└─ 狀態 / 時間 / 商品 → 走多維度索引

[冷熱分離]
├─ 近期訂單 → 熱區（常查、低延遲）
└─ 歷史訂單 → 冷區（歸檔、量大少碰、可較慢）

[一致性]
└─ 查詢允許稍舊（read model 投影有秒級延遲）；不要求與寫主庫強一致
```

## 4. State Transition

```text
（讀取型，本身無業務狀態機；訂單狀態由下單 / 支付 / 結算等來源情境決定，這裡只投影與查）
```

## 5. Data Dictionary（邏輯欄位；實體 DDL 在各平台 SD）

```text
order_read（訂單查詢投影，read model）
├─ orderId(PK) / userId / status / amount
├─ items[] / createdAt / updatedAt
└─ （多維度索引欄位：status, createdAt, itemId…）

（來源真相為下單 / 訂單狀態機情境；此處只讀投影）
```
