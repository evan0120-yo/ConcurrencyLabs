# 檔案上傳 / 圖片影片處理 · s40_fileuploadmediaprocessing

> 一句話定位：使用者上傳大檔，後端接收後要掃毒、縮圖、轉碼，重活分級非同步處理。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：企業資料
- 壓力模型：任務補償型（轉碼 pipeline）、寫爆型（大檔傳輸）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 掃毒 / 縮圖 / 轉碼是耗時的後處理，應非同步、可重試、可分級。
- **寫爆型** → 大檔上傳本身是重 IO / 頻寬壓力。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者上傳一支 2GB 影片
   │
   ├─ 走後端中轉？ ──> 佔頻寬、佔連線、易失敗
   └─ 直傳儲存 + 後端拿到通知
             │
             ▼
   [後處理 pipeline] 掃毒 ──> 縮圖 ──> 多解析度轉碼
       ├─ (小圖) ──> 快、優先
       └─ (4K 轉碼) ──> 慢、可排隊分級

────────── 一堆人同時上傳大檔 ──────────
```

### 分段解說

- 上傳與後處理是兩件事：先安全收下，再慢慢加工。
- 後處理有輕有重（縮圖 vs 4K 轉碼），要分級排隊。
- 大檔上傳容易斷，要能續傳、要保護頻寬。

### 痛點分析

- **能不能非同步？** 後處理可以，使用者不必等轉碼完。
- **輕重分級** → 縮圖優先、重轉碼排隊。
- **大檔風險** → 上傳易失敗，要能分段 / 續傳。
- **真正的痛**：把「即時要的（縮圖）」跟「慢工（轉碼）」分級，別讓重活塞住輕活，也別讓大檔傳輸拖垮系統。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s40 先收下再加工、後處理輕重分級,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：後端接檔後同步掃毒/縮圖/轉碼
├─ 大方塊：接檔 · 掃毒 · 縮圖/轉碼
└─ 撐不住的訊號 → 大檔中轉佔頻寬連線、重轉碼塞住輕活

【P2 · 直傳 storage + 後處理 pipeline 非同步分級】約 10,000 QPS
├─ 定義：pre-signed 直傳+續傳,後處理進 pipeline 非同步、輕重分級
├─ 大方塊（綁一起）
│   ├─ pre-signed 直傳 storage(不中轉) + 分段續傳
│   ├─ [MQ] 後處理 pipeline 非同步
│   └─ 縮圖優先佇列 / 重轉碼低優先排隊
├─ 為何綁：先收下再加工 = 直傳 + pipeline + 輕重分級,一起做
└─ 撐不住的訊號 → 重轉碼任務堆積、worker 不足

【P3 · 轉碼 worker 分級池 + 分片並行 + CDN 分發】約 50,000+ QPS
├─ 定義：轉碼 worker 依輕重分獨立池,任務分片並行,產物走 CDN
├─ 大方塊：worker 分級池 · 轉碼分片並行 · CDN 分發
└─ 為何綁：媒體規模化 = 分級池 + 分片 + CDN,一起做
```
