# 鏈下撮合 Engine · matchingengine

> 一句話定位：交易所的心臟，把買賣單依價格時間優先撮合成交，同一 symbol 必須嚴格順序、極低延遲。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：交易所 / Web3
- 壓力模型：低延遲型（核心）、熱點型（熱門 symbol）

---

## Block 2：為什麼被列上這些

- **低延遲型** → 撮合是交易所延遲最敏感的核心，微秒級都要計較。
- **熱點型** → 熱門 symbol 的訂單量遠高於其他，會集中在單一撮合單元。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
買單、賣單不斷湧入同一個 symbol（如 BTC/USDT）
   │
   ▼
[撮合 Engine] 依「價格優先、時間優先」撮合
   ├─ 買一 >= 賣一 ──> 成交
   └─ 否則掛進訂單簿等待
   *同一 symbol 內，必須嚴格照順序處理，不能亂*

────────── 行情劇烈波動，訂單像機關槍一樣射進來 ──────────
```

### 分段解說

- 撮合的正確性建立在「順序」上：同一 symbol 的單必須依序處理。
- 為了順序與速度，核心常是單執行緒順序化，周邊全部解耦出去。
- 不同 symbol 之間互相獨立，可以分開跑。

### 痛點分析

- **延遲要求** → 極端低延遲，是交易所的招牌。
- **順序性** → 同一 symbol 內絕對順序，不可並發亂序。
- **有沒有熱點？** 熱門 symbol 是壓力集中點。
- **真正的痛**：要同時做到「極低延遲」與「嚴格順序」，這逼核心走順序化，把所有非撮合工作都趕出去。
