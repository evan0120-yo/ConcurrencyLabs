# Marketplace 商家庫存同步 · s59_merchantinventorysync

> 一句話定位：平台要同步成千上萬商家的商品 / 庫存變更，大商家不能拖垮小商家的同步。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：社群 / Feed / 搜尋 / Marketplace
- 壓力模型：任務補償型（非同步同步 + 失敗補償）、多租戶型（商家隔離）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 商家庫存 / 商品變更用非同步同步，失敗要能補償重試。
- **多租戶型** → 每個商家是一個租戶，大商家不能拖垮小商家。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
商家在自己系統改了價格 / 庫存
   商家 A ─┐
   商家 B ─┼──> [同步] 非同步把變更拉進平台
   大商家 C┘        │
                    ├─ (成功) ──> 平台商品更新
                    └─ (失敗) ──> 補償重試

────────── 大商家一次改 10 萬個 SKU ──────────

小商家想要：我改一個庫存，別因為大商家在洗全站就卡住我
```

### 分段解說

- 同步是非同步的：商家改完，平台在背後拉。
- 大商家的批量變更會佔滿同步通道，拖累小商家。
- 失敗要能補償，不能因為某次同步失敗就資料錯亂。

### 痛點分析

- **能不能非同步？** 可以，同步天生是背景工作。
- **要不要隔離？** 要，大商家獨立通道，別拖垮小商家。
- **失敗補償** → 同步失敗要能重試補平。
- **真正的痛**：跟多租戶「吵鬧鄰居」同構——大商家的批量同步不能佔滿通道，害小商家的變更遲遲同步不上。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s59 非同步同步 + 多租戶隔離(大商家不拖垮小),放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：商家變更非同步拉取套用,失敗重試
├─ 大方塊：非同步同步 · 套用 · 補償重試
└─ 撐不住的訊號 → 大商家一次改 10 萬 SKU 佔滿通道、拖垮小商家

【P2 · per商家隔離通道 + 大商家限流 + 補償冪等】約 10,000 QPS
├─ 定義：per-merchant 通道/quota,大商家獨立限流分批,補償冪等
├─ 大方塊（綁一起）
│   ├─ per-merchant 隔離通道/quota
│   ├─ 大商家(hot)獨立通道/限流分批
│   └─ 失敗補償 + 冪等(依版本不重複套用)
├─ 為何綁：吵鬧鄰居 = 隔離通道 + 大商家限流 + 補償,一起做
└─ 撐不住的訊號 → 商家數再多、同步任務量體大

【P3 · 同步分片並行 + 大商家獨立通道】約 50,000+ QPS
├─ 定義：同步任務按 merchant 分片並行,大商家獨立通道規模化隔離
├─ 大方塊：同步分片並行 · 大商家獨立通道 · 小商家保即時
└─ 為何綁：規模化同步 = 分片 + 大商家隔離,一起做
```
