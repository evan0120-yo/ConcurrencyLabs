# 批次匯出 / 大量報表任務 · s36_reportexport

> 一句話定位：使用者要匯出大量資料，改成任務化非同步產生、產完通知下載，不能拖垮線上主庫。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：任務 / 排程 / 第三方整合
- 壓力模型：任務補償型（非同步產檔）
- 數據角色：分析 / 查詢讀（大量掃資料）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 大匯出應非同步化、可重試，產完再通知，不阻塞使用者。
- **分析讀角色** → 匯出要掃大量資料，屬於重讀取，會壓到資料源。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者點「匯出全部訂單（50 萬筆）」
   │
   ├─ 同步產？ ──> 錯！瀏覽器轉圈幾分鐘、還可能拖垮主庫
   └─ 任務化：
        丟 job ──> [背景產檔] 分批查、慢慢拼 ──> 產完通知「可下載了」

────────── 月底大家同時匯報表 ──────────

營運想要：報表能產出，但別讓匯出把線上服務拖慢
```

### 分段解說

- 大匯出天生慢，同步做只會卡使用者、壓垮資料源。
- 改成任務化：先受理、背景慢慢產、好了再通知。
- 掃資料要分批，避免一口氣把主庫吸乾。

### 痛點分析

- **能不能非同步？** 應該，產檔不該卡在請求裡。
- **會不會壓垮資料源？** 會，要分批 / 走副本。
- **要不要隔離？** 要，匯出讀不該碰線上 OLTP 主庫。
- **真正的痛**：大量資料匯出的重讀取，如何不拖垮支撐線上交易的主資料庫。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+（量體指匯出任務/併發面）。每跳一個平台 = 一次架構改造。
> s36 非同步任務化、走副本不壓主庫,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：匯出任務化非同步,背景分批產檔,產完通知
├─ 大方塊：任務化非同步 · 分批查 · 產檔通知
└─ 撐不住的訊號 → 大匯出掃資料壓主庫、月底集中匯出

【P2 · 分批走副本 + 資料源隔離 + 併發控制】約 10,000 QPS
├─ 定義：掃資料走讀副本/數倉,游標分批,限制同時匯出數
├─ 大方塊（綁一起）
│   ├─ 走讀副本/數倉(禁打線上主庫)
│   ├─ 游標分批(控每批量)
│   └─ 匯出併發上限(不吃光資源)
├─ 為何綁：不壓垮主庫 = 走副本 + 分批 + 併發控制,一起做
└─ 撐不住的訊號 → 超大匯出單任務仍慢、副本也吃緊

【P3 · 匯出分片並行 + 專用資源池隔離】約 50,000+ QPS
├─ 定義：大匯出分片並行產出,匯出走專用資源池與線上隔離
├─ 大方塊：匯出分片並行 · 專用資源池 · 與 OLTP 徹底隔離
└─ 為何綁：規模化匯出 = 分片並行 + 資源隔離,一起做
```
