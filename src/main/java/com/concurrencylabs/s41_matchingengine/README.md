# 鏈下撮合 Engine · s41_matchingengine

> 一句話定位：交易所的心臟，把買賣單依價格時間優先撮合成交，同一 symbol 必須嚴格順序、極低延遲。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：交易所 / Web3
- 壓力模型：低延遲型（核心）、熱點型（熱門 symbol）

---

## Block 2：為什麼被列上這些

- **低延遲型** → 撮合是交易所延遲最敏感的核心，微秒級都要計較。
- **熱點型** → 熱門 symbol 的訂單量遠高於其他，會集中在單一撮合單元。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
買單、賣單不斷湧入同一個 symbol（如 BTC/USDT）
   │
   ▼
[撮合 Engine] 依「價格優先、時間優先」撮合
   ├─ 買一 >= 賣一 ──> 成交
   └─ 否則掛進訂單簿等待
   *同一 symbol 內，必須嚴格照順序處理，不能亂*

────────── 行情劇烈波動，訂單像機關槍一樣射進來 ──────────
```

### 分段解說

- 撮合的正確性建立在「順序」上：同一 symbol 的單必須依序處理。
- 為了順序與速度，核心常是單執行緒順序化，周邊全部解耦出去。
- 不同 symbol 之間互相獨立，可以分開跑。

### 痛點分析

- **延遲要求** → 極端低延遲，是交易所的招牌。
- **順序性** → 同一 symbol 內絕對順序，不可並發亂序。
- **有沒有熱點？** 熱門 symbol 是壓力集中點。
- **真正的痛**：要同時做到「極低延遲」與「嚴格順序」，這逼核心走順序化，把所有非撮合工作都趕出去。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+（撮合以延遲/吞吐為主,量體為訂單流入）。每跳一個平台 = 一次架構改造。
> s41 極低延遲 + 嚴格順序 + 熱門 symbol,交易所心臟複雜度最高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：單 symbol 訂單簿撮合,同步處理
├─ 大方塊：訂單簿 · 價格時間優先撮合 · 成交事件外拋
└─ 撐不住的訊號 → 同 symbol 順序要求、波動訂單像機關槍、記帳拖慢撮合

【P2 · 核心單執行緒順序化 + 周邊全解耦】約 10,000 QPS
├─ 定義：撮合核心單執行緒序列化,記帳/通知/行情全事件外拋非同步
├─ 大方塊（綁一起）
│   ├─ 核心單執行緒(同 symbol 嚴格順序、記憶體訂單簿)
│   ├─ 非撮合工作全趕出關鍵路徑(事件外拋)
│   └─ 事件序可回放(恢復)
├─ 為何綁：極低延遲 + 嚴格順序 → 核心順序化 + 周邊解耦,一起做
└─ 撐不住的訊號 → 熱門 symbol 單元吃滿、跨 symbol 未並行

【P3 · 按 symbol 分片撮合單元】約 50,000 QPS
├─ 定義：不同 symbol 分到不同撮合單元並行,單 symbol 仍單線順序
├─ 大方塊：symbol 分片路由 · 每 symbol 獨立單元 · 單元內順序保證
└─ 為何綁：一分片,訂單/成交/訂單簿都收斂到該 symbol 單元

【P4 · 熱門 symbol 獨立部署 + 內存訂單簿快照恢復】約 100,000+ QPS
├─ 定義：超熱 symbol 獨立高規部署,內存訂單簿 + 快照/事件回放快速恢復
├─ 大方塊：熱 symbol 獨立單元 · 內存訂單簿 · 快照+事件回放恢復
└─ 為何綁：極致低延遲+高可用 = 獨立部署 + 內存簿 + 快照恢復,一起做
```
