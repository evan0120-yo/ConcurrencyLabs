# 即時報表 / Streaming Aggregation · s52_streamingaggregation

> 一句話定位：事件流即時聚合出報表數字（近 N 分鐘曝光 / 點擊 / 收益），近似即時、延遲修正。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：AdTech / 資料流
- 壓力模型：寫爆型 / ingestion（事件流入）、讀爆型（看報表）
- 數據角色：即時聚合 / stream processing

---

## Block 2：為什麼被列上這些

- **即時聚合** → 對持續事件流做時間窗聚合，是 stream processing 的典型。
- **寫爆 + 讀爆** → 進來的事件是寫爆，產出的即時報表被大量讀。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
事件源源不絕
   曝光 / 點擊 / 轉換 ──> [即時聚合] 依時間窗算
                           ├─ 近 1 分鐘曝光數
                           ├─ 近 5 分鐘收益
                           └─ 各維度即時數字
   ▼
   Dashboard：廣告主 / 營運一直刷「現在跑得怎樣」

────────── 遲到的事件晚幾秒才到 ──────────

要嘛先給近似值、稍後修正，要嘛等齊了才給（但就不即時了）
```

### 分段解說

- 即時聚合是在「事件還在流」的時候就出數字。
- 事件會遲到，所以即時值往往是近似，之後修正。
- 即時性與準確性又是一組經典拉扯。

### 痛點分析

- **能不能近似？** 即時報表可先近似，稍後修正。
- **讀爆還寫爆？** 兩頭都有（事件寫爆、報表讀爆）。
- **遲到事件** → 要決定「先給近似」或「等齊再給」。
- **真正的痛**：事件會遲到，逼你在「即時但近似」與「準確但延遲」之間選邊，多半選前者 + 事後修正。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s52 事件流即時聚合、近似即時+遲到修正,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：事件流即時時間窗聚合,dashboard 讀
├─ 大方塊：時間窗聚合 · 即時輸出 · 讀
└─ 撐不住的訊號 → 事件寫爆+報表讀爆、遲到事件、單聚合節點吃緊

【P2 · stream 窗口聚合 + 近似即時 + 遲到修正】約 10,000 QPS
├─ 定義：stream 依 eventTime 歸窗,先給近似即時值,遲到 watermark 修正
├─ 大方塊（綁一起）
│   ├─ stream 時間窗聚合(依 eventTime)
│   ├─ 近似即時輸出(不等齊)
│   └─ 遲到事件 watermark 修正 + 定案落地
├─ 為何綁：即時 vs 準確 = 近似輸出 + 遲到修正,一起做
└─ 撐不住的訊號 → 維度/窗數放大、聚合狀態量大

【P3 · 分片聚合 + 讀快取分層 + 定案落 OLAP】約 50,000+ QPS
├─ 定義：聚合按 key 分片並行,即時值多層讀快取,定案落 OLAP
├─ 大方塊：分片聚合 · 讀快取分層 · 定案落 OLAP
└─ 為何綁：寫爆聚合+讀爆報表 = 分片 + 讀快取 + OLAP,一起做
```
