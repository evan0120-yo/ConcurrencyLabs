# SA.md — s67_globalratelimit（全域 Rate Limit Service）

> 事實文件（SASD）· 結構化分析。描述「全域限流要做什麼」的**業務行為**，平台無關（P1..Pn 共用；各平台怎麼扛量在各自 SD）。
> 難點在分散式下算總量：各台各算會漏擋、全問中央又慢又是瓶頸。在「準確算總量」與「不引入延遲 / 不變瓶頸」間取捨——**本地預扣 + 中央對齊**是典型解。

## 1. Context Diagram

```text
[多機 / 多服務請求] --(限流判斷: key)--> (全域限流) --(ALLOW / DENY)--> [呼叫方]

(全域限流) --(本地預扣額度)--> [各機本地配額]
(全域限流) --(中央總量對齊)--> [util.redis 中央計數]
```

## 2. DFD（精簡 box flow）

```text
[限流判斷] --(key)--> [本地配額]
   ├─(本地還有預扣額度)──> 直接放行 / 扣本地（不打中央，極快）
   │(本地額度用完)
   ▼
[跟中央對齊]（原子取下一批額度 / 檢查總量）--> [中央計數]
   ├─(總量未超)──> 領一批本地額度，繼續
   └─(總量超額)──> DENY（回 429）
```

## 3. Process Spec（行為基準，decision table）

```text
[本地預扣]（降延遲、不變瓶頸）
└─ 每台先向中央領一批額度，本地扣；用完再領（多數判斷不打中央）

[中央對齊]（算準總量）
└─ 中央維護「該 key 這窗口的總量」；領額度 / 超額判斷走中央原子操作

[取捨]
├─ 全問中央 → 準但慢、成瓶頸
├─ 各台各算 → 快但漏擋（總量超放）
└─ 折衷：本地預扣批量 + 中央對齊（略近似，總量可控）

[熱 key]
└─ 熱門 key 的中央計數是熱點 → 分片 / 近似（見各平台 SD）
```

## 4. State Transition

```text
（無業務狀態機；限流計數的 窗口滾動 / 額度領取 週期）
```

## 5. Data Dictionary（邏輯欄位；實體 DDL 在各平台 SD）

```text
rate_rule（限流規則）
├─ key(PK) / limit / windowSec / dimension

central_counter（中央計數，Redis）
├─ key + window / consumed / 剩餘額度

local_lease（本地預扣額度，機內）
├─ key / granted / used
```
