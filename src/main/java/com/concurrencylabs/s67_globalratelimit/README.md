# 全域 Rate Limit Service · s67_globalratelimit

> 一句話定位：跨多台 / 多服務的統一限流，要在分散式環境下把「總量」算準，且自己要極快。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Cloud / 平台治理
- 壓力模型：平台防爆型（核心）、熱點計數型（限流計數器）

---

## Block 2：為什麼被列上這些

- **平台防爆型** → 這是保護全站的限流基礎設施。
- **熱點計數型** → 限流本質是「對某 key 高頻計數並判斷是否超額」，計數器是熱點。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
限流不只在一台機器，而是跨整個叢集
   請求打到 機器1 / 機器2 / 機器3 …
   ▼
[全域限流] 要知道「這個 user 在整個叢集這一秒共打了幾次」
   ├─ 每台各自算 → 加起來會超過總額（漏擋）
   └─ 都問中央 → 中央變瓶頸、每次多一跳延遲

────────── 每秒百萬次限流判斷，還要跨機器一致 ──────────

平台想要：總量算得準，但限流服務自己不能變成瓶頸
```

### 分段解說

- 單機限流簡單，難的是「分散式下算總量」。
- 各台各算會漏擋，全問中央又慢又是瓶頸。
- 常見折衷是本地預扣一批額度，再跟中央對齊。

### 痛點分析

- **延遲要求** → 限流在關鍵路徑，自己要極快。
- **分散式一致** → 跨機器的總量要算準。
- **有沒有熱點？** 熱門 key 的計數器是熱點。
- **真正的痛**：在「準確算總量」與「不引入延遲 / 不變瓶頸」之間取捨，本地預扣 + 中央對齊是典型解法。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s67 分散式算總量、本地預扣+中央對齊,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：中央計數限流,每次問中央
├─ 大方塊：中央計數 · 超額判定
└─ 撐不住的訊號 → 每次問中央變瓶頸多一跳、各台各算漏擋

【P2 · 本地預扣 + 中央對齊】約 10,000 QPS
├─ 定義：每台向中央領一批額度本地扣,用完再領,中央算總量
├─ 大方塊（綁一起）
│   ├─ 本地預扣批量額度(多數判斷不打中央)
│   ├─ 中央原子對齊總量(算準)
│   └─ 窗口滾動 + 額度領取
├─ 為何綁：準又不變瓶頸 = 本地預扣 + 中央對齊,一起做
└─ 撐不住的訊號 → 熱門 key 中央計數熱點

【P3 · 中央計數分片 + 近似總量 + 熱 key 分散】約 50,000+ QPS
├─ 定義：中央計數按 key 分片,熱 key 近似總量,分散熱點
├─ 大方塊：中央計數分片 · 近似總量 · 熱 key 分散
└─ 為何綁：規模化限流 = 計數分片 + 近似 + 熱 key 分散,一起做
```
