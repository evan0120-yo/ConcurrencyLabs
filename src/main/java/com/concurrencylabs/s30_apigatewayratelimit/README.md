# API Gateway / Rate Limit · s30_apigatewayratelimit

> 一句話定位：所有流量的第一道門，做路由、認證與限流，要在最前面擋掉超量與濫用。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：會員 / Gateway / 多租戶
- 壓力模型：平台防爆型（核心）、多租戶型（按 user / tenant 限流）

---

## Block 2：為什麼被列上這些

- **平台防爆型** → Gateway 是全站入口，限流 / 防濫用是它的核心職責。
- **多租戶型** → 要能按 user / IP / tenant 分別限額，避免單一來源吃光資源。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
所有請求都先經過 Gateway
   請求 ──> [Gateway]
             ├─ 路由：轉給正確的後端服務
             ├─ 認證：這人是誰、能不能進
             └─ 限流：這人 / 這 IP 這秒是不是超量
                  ├─ (正常) ──> 放行
                  └─ (超量) ──> 擋下、回 429

────────── 某個 client 抽風狂打，想吃光資源 ──────────
```

### 分段解說

- Gateway 在最前面，它慢全站都慢，所以本身要極輕快。
- 限流要「準」：多算了誤傷正常人，少算了擋不住濫用。
- 分散式環境下，限流計數要跨多台 Gateway 一致。

### 痛點分析

- **延遲要求** → 它在關鍵路徑最前端，必須極快。
- **限流精度** → 誤傷 vs 漏擋要平衡。
- **分散式一致** → 多台 Gateway 的計數要收斂。
- **真正的痛**：在全站最前端做又快又準的限流，且限流狀態要在多台之間保持一致。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s30 全站入口要極快、限流計數多台一致,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：單機限流 + 路由 + 認證
├─ 大方塊：路由 · 認證 · 單機計數限流
└─ 撐不住的訊號 → 多台 gateway 各算各的、整體超放;某 client 抽風

【P2 · 分散式限流計數一致】約 10,000 QPS
├─ 定義：限流計數走共享儲存(Redis 原子),多台收斂一致
├─ 大方塊（綁一起）
│   ├─ [Redis] 原子計數/Lua(令牌桶/滑窗)
│   ├─ 多台 gateway 共享計數
│   └─ 本地快取 + 共享計數混合(降延遲)
├─ 為何綁：多台一致 = 共享原子計數 + 本地快取,一起做
└─ 撐不住的訊號 → 全站入口限流計數本身成熱點

【P3 · 多維度限流 + 計數分片/近似】約 50,000+ QPS
├─ 定義：per user/IP/tenant/API 多維度限流,計數分片或近似降熱點
├─ 大方塊：多維度限流 · 計數分片/近似 · 濫用來源隔離
└─ 為何綁：入口極快又準 = 多維度 + 計數分片,一起做
```
