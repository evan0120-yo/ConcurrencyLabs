# API Gateway / Rate Limit · s30_apigatewayratelimit

> 一句話定位：所有流量的第一道門，做路由、認證與限流，要在最前面擋掉超量與濫用。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：會員 / Gateway / 多租戶
- 壓力模型：平台防爆型（核心）、多租戶型（按 user / tenant 限流）

---

## Block 2：為什麼被列上這些

- **平台防爆型** → Gateway 是全站入口，限流 / 防濫用是它的核心職責。
- **多租戶型** → 要能按 user / IP / tenant 分別限額，避免單一來源吃光資源。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
所有請求都先經過 Gateway
   請求 ──> [Gateway]
             ├─ 路由：轉給正確的後端服務
             ├─ 認證：這人是誰、能不能進
             └─ 限流：這人 / 這 IP 這秒是不是超量
                  ├─ (正常) ──> 放行
                  └─ (超量) ──> 擋下、回 429

────────── 某個 client 抽風狂打，想吃光資源 ──────────
```

### 分段解說

- Gateway 在最前面，它慢全站都慢，所以本身要極輕快。
- 限流要「準」：多算了誤傷正常人，少算了擋不住濫用。
- 分散式環境下，限流計數要跨多台 Gateway 一致。

### 痛點分析

- **延遲要求** → 它在關鍵路徑最前端，必須極快。
- **限流精度** → 誤傷 vs 漏擋要平衡。
- **分散式一致** → 多台 Gateway 的計數要收斂。
- **真正的痛**：在全站最前端做又快又準的限流，且限流狀態要在多台之間保持一致。
