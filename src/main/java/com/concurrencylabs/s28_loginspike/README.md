# 登入尖峰系統 · s28_loginspike

> 一句話定位：活動 / 開服瞬間大量人同時登入，還混著攻擊流量，要擋壞人又不拖慢正常人。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：會員 / Gateway / 多租戶
- 壓力模型：平台防爆型（限流 / 風控守門）、讀爆型（驗帳密 / 查 session）

---

## Block 2：為什麼被列上這些

- **平台防爆型** → 登入是攻擊重災區（暴力破解 / 撞庫），要限流、擋異常。
- **讀爆型** → 尖峰時大量驗證帳密、查 / 建 session，讀壓力集中。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
【正常】使用者輸入帳密 ──> 驗證 ──> 發 session ──> 登入成功

────────────── 開服 / 活動瞬間 ──────────────

大量正常人 + 混雜攻擊流量 一起湧入
   │
   ├─ 正常人想要：登入順、別排隊太久
   ├─ 攻擊者：暴力破解 / 撞庫 / 用機器人洗
   └─ 系統面對：分不清誰是誰，全部同時打進來
        ├─ (可疑) ──> captcha / 限頻 / 擋
        └─ (正常) ──> 放行
```

### 分段解說

- 登入尖峰不只是量大，還混著惡意流量。
- 難點是「在洪流中分辨正常與攻擊」，且不能誤傷正常人。
- 熱點是熱門帳號與攻擊來源 IP。

### 痛點分析

- **平均還是尖峰？** 尖峰（開服 / 活動）。
- **有沒有惡意？** 有，暴力破解 / 撞庫是常態。
- **有沒有熱點？** 熱門帳號、攻擊 IP。
- **真正的痛**：尖峰洪流裡混著攻擊，要擋住壞流量又不拖慢、不誤傷正常登入。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s28 驗證讀爆 + 防攻擊,可快取可分級,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：驗帳密 + 發 session,基本失敗計數
├─ 大方塊：[DB] 驗帳密 · session store · 失敗計數
└─ 撐不住的訊號 → 開服/活動尖峰、混雜暴力破解/撞庫、熱帳號與攻擊IP

【P2 · 限流 + 風控守門 + CAPTCHA 分級】約 10,000 QPS
├─ 定義：登入前置限流/異常偵測,可疑升級 CAPTCHA,不誤傷正常
├─ 大方塊（綁一起）
│   ├─ [Redis] per account/IP 限頻
│   ├─ 撞庫/暴力特徵偵測 → CAPTCHA/限頻分級
│   └─ 驗證讀走快取(免每次打主庫)
├─ 為何綁：洪流分辨正常/攻擊 = 限流 + 偵測 + 分級防護,一起做
└─ 撐不住的訊號 → 熱門帳號 + 攻擊IP 單點熱

【P3 · 熱帳號/攻擊源分散 + 驗證讀多層快取】約 50,000+ QPS
├─ 定義：熱帳號與攻擊來源分散承接,帳密驗證走多層快取
├─ 大方塊：熱點分散 · 攻擊源隔離 · 驗證多層快取
└─ 為何綁：把熱點與攻擊關在局部 + 驗證不打後端,一起做
```
