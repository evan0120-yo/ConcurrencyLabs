# 對帳與補償平台 · s27_reconciliationcompensation

> 一句話定位：把自家帳與第三方帳逐筆比對，找出差異單，走補償流程補平，主交易不等它。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：支付 / 金融
- 壓力模型：任務補償型（核心）
- 數據角色：批次比對 / 分析讀

---

## Block 2：為什麼被列上這些

- **任務補償型** → 對帳是批次拉檔、找差異、走補償狀態機的典型後台工作。
- **批次分析讀** → 需要大量讀取雙方帳目做比對。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
每天 / 每小時
   [我方帳] ──┐
              ├──> [對帳] 逐筆比對
   [第三方帳]─┘        │
                       ├─ (一致) ──> 打勾
                       └─ (差異) ──> 差異單 ──> 補償流程補平

────────── 主交易絕不等對帳，對帳在背後慢慢跑 ──────────

營運想要：帳最終要平，但別讓對帳拖慢即時交易
```

### 分段解說

- 對帳是「事後」的：交易當下不等它。
- 找出差異只是第一步，補償把差異收斂才是目的。
- 補償本身是個狀態機（待處理 → 處理中 → 已補平 / 需人工）。

### 痛點分析

- **能不能延遲？** 可以，對帳天生是事後批次。
- **要不要隔離？** 要，主交易不能被對帳拖累。
- **一致性** → 最終要對得平，差異要有交代。
- **真正的痛**：在雙方帳都可能出錯 / 遲到的情況下，穩定地找出差異並把它補平，且完全不影響即時交易。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+（對帳為事後批次,量體指補償/差異處理面）。每跳一個平台 = 一次架構改造。
> s27 事後批次、可延遲、主交易不等它,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：批次拉雙方帳逐筆比對,差異走補償
├─ 大方塊：拉檔比對 · 差異單 · 補償狀態機
└─ 撐不住的訊號 → 帳量大單機比對慢、差異補償串行

【P2 · 拉檔分批 + 差異單 + 補償狀態機】約 10,000 QPS
├─ 定義：帳檔分批流式比對,差異進補償狀態機非同步收斂
├─ 大方塊（綁一起）
│   ├─ 帳檔分批流式比對
│   ├─ 差異單 + 補償狀態機(PENDING→補平/人工)
│   └─ 補償冪等(同差異只補一次)
├─ 為何綁：分批比對 + 差異狀態機 + 冪等補償,一起收斂差異
└─ 撐不住的訊號 → 帳量再放大、對帳窗口趕不完

【P3 · 分片並行對帳 + 與主交易隔離】約 50,000+ QPS
├─ 定義：對帳按帳戶/期號分片並行,對帳資源與主交易徹底隔離
├─ 大方塊：分片並行對帳 · 對帳獨立資源 · 主交易不受影響
└─ 為何綁：規模化事後對帳 = 分片並行 + 資源隔離,一起做
```
