# 遊戲任務獎勵系統 · gametaskreward

> 一句話定位：玩家完成任務後發獎，主流程不等發獎，且同一任務只能領一次。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：任務補償型（非同步發獎 + 防重）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 發獎可從主遊戲流程解耦、非同步進行，並需要「防重複領」的冪等保護。
- **博弈 domain** → 任務系統是活動留存的常見手段。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
玩家完成「今日登入 / 累積下注 1000」任務
   │
   ▼
[任務系統] 判定達成 ──> 發獎（金幣 / 道具）
   └─ 同一任務只能領一次（防重複領）

────────── 活動期間大量任務同時達成 ──────────

玩家想要：達成馬上有獎、不要漏發
營運想要：一個任務只發一次，不能被刷重領
```

### 分段解說

- 達成與發獎可以拆開：主遊戲不該卡在發獎上。
- 真正的難點是「不重不漏」——既不能漏發，也不能被重複領。
- 任務狀態像一個小狀態機：未達成 → 達成 → 已領取。

### 痛點分析

- **能不能非同步？** 可以，發獎跟主流程解耦。
- **能不能重複 / 漏發？** 都不行，冪等是核心。
- **平均還是尖峰？** 活動整點常有集中達成。
- **真正的痛**：把「發獎」跟「主遊戲」解耦後，如何保證每個任務不多發也不漏發。
