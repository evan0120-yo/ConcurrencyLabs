# 秒殺限量商品 · s13_flashsale

> 一句話定位：限量商品開賣瞬間，遠超庫存的人同時搶，要保證不超賣、且體驗不崩。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：搶資源型（核心）、讀爆型（開賣前商品頁預熱）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 庫存有限、需求遠大於供給，本質是「一大群人搶同一份有限資源」。
- **讀爆型** → 開賣前商品頁被瘋狂刷新看庫存 / 倒數，讀壓力先於寫壓力爆掉。
- **電商 domain** → 綁在行銷活動、拉流量場景，跟 `s14_couponclaim`、`s15_inventorydeduction` 同源。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
【開賣前 3 分鐘】
消費者一直刷頁面看倒數
   │ (只讀不寫)
   ▼
[商品頁] 顯示「庫存 100、01:59 後開搶」

────────────── 00:00 開賣瞬間 ──────────────

10 萬人同一秒按下「搶購」
   │
   ├─ 消費者想要：按下去馬上知道搶到沒
   ├─ 營運想要：100 個就是 100 個，一個都不能超賣
   └─ 系統面對：10 萬筆寫入搶 100 個名額
        ├─ (搶到的 100 人) ──> 進結帳
        └─ (其餘 99,900 人) ──> 秒回「已售完」，不要卡住
```

### 分段解說

- 開賣前：壓力是「讀」——所有人盯著同一頁看倒數 / 庫存。
- 開賣瞬間：壓力切換成「寫」——極端集中的寫入，搶極少的名額。
- 是同一情境內「先讀爆、再寫爆」的接力，兩段壓力性質完全不同。

### 痛點分析

- **能不能丟？** 「搶購請求」本身可以被拒（售完就拒）；但「已扣庫存卻沒建單」絕不能發生。
- **能不能超賣？** 絕對不行，這是成本與信譽的紅線。
- **平均還是尖峰？** 極端尖峰，99% 的壓力集中在開賣後幾秒。
- **真正的痛**：99,900 個註定失敗的請求，不能拖垮那 100 個成功的人，更不能讓後端資料庫陪葬。

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s13 絕不超賣 + 讀爆接寫爆,正確性壓力高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：Redis 預扣庫存 + DB 建訂單,售完快速失敗
├─ 大方塊：[Redis] 原子預扣庫存 · idem 去重 · [DB] 建秒殺單 · 失敗回補
└─ 撐不住的訊號 → 開賣前商品頁讀爆、開賣瞬間寫爆,DB 建單被沖

【P2 · 排隊 + MQ 削峰】約 10,000 QPS
├─ 定義：搶購請求排隊,MQ 削峰,建單非同步
├─ 大方塊（綁一起）
│   ├─ 前端排隊 / 令牌放行(擋掉大部分註定失敗的請求)
│   ├─ [MQ] 搶購事件削峰、worker 建單
│   └─ 預扣(Redis)↔建單(異步) 對帳補償
├─ 為何綁：建單一異步,「扣了必建單」要靠 MQ + 對帳補償
└─ 撐不住的訊號 → 單一商品庫存 key 熱點

【P3 · 商品庫存 token 分片】約 50,000 QPS
├─ 定義：單一庫存拆成多 token 桶,分散預扣熱點
├─ 大方塊：庫存 token 分桶 · 跨桶預扣路由 · 桶餘量聚合
└─ 為何綁：一分桶,預扣邏輯 + 餘量統計都要跟著改

【P4 · 秒殺獨立系統 + 令牌桶分批放行】約 100,000+ QPS
├─ 定義：秒殺整包獨立部署,令牌桶分批放量,與一般交易隔離
├─ 大方塊：秒殺獨立叢集 · 令牌桶分批放行 · 與主站資源隔離
└─ 為何綁：隔離 = 獨立部署 + 放量策略 + 資源牆,一起變
```
