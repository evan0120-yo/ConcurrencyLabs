# 下注入口系統 · s04_bettingentry

> 一句話定位：玩家在某桌 / 某局下注，系統先驗餘額、凍結金額再受理，封盤前尖峰湧入。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：錢包帳本型（凍結金額）、熱點型（熱門桌 / 局）、低延遲型（下注要快）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 下注要先凍結金額，不能讓玩家押超過餘額（透支）。
- **熱點型** → 熱門桌 / 熱門局的注會集中打同一個 gameId / tableId。
- **低延遲型** → 玩家在乎「有沒有押到」，且封盤前分秒必爭。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
玩家在某桌下注 100
   │
   ▼
[下注入口] 驗餘額 ──> 凍結 100 ──> 受理這注
   ├─ (餘額夠) ──> 受理，等開獎
   └─ (餘額不足 / 已封盤) ──> 拒絕

────────── 封盤前最後幾秒、熱門桌 ──────────

玩家想要：下注要快、馬上確認押到沒
營運想要：不能透支、封盤後一注都不收
```

### 分段解說

- 正常流程：驗餘額 → 凍結 → 受理，很直白。
- 難點在時間邊界（封盤瞬間）與熱點集中（同一桌）。
- 凍結是錢包帳本的一部分，錯了就是資損。

### 痛點分析

- **有沒有熱點？** 有，熱門桌 / 局集中。
- **時間邊界** → 封盤是硬邊界，封盤後的注必須乾淨拒絕。
- **延遲要求** → 下注體感要快，慢了玩家會抱怨押不到。
- **真正的痛**：熱門桌尖峰 + 封盤瞬間 + 凍結正確性，三個壓力擠在同一個入口。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s04 因「錢不能透支 + 封盤硬邊界」，正確性壓力高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：同步驗餘額 → 凍結 → 受理，DB 落注
├─ 大方塊：[account] 原子凍結金額 · [Redis] idem 去重 · [DB] 同步建注
└─ 撐不住的訊號 → 熱門桌集中打同一 tableId、封盤前尖峰、DB 建注 P99 飆

【P2 · 凍結移 Redis + 落注非同步】約 10,000 QPS
├─ 定義：把「凍結」搬到 Redis 原子帳、落注轉非同步，削 DB 壓力
├─ 大方塊（必須一起改）
│   ├─ [Redis] ledger 預凍結金額（原子，取代同步打 account/DB）
│   ├─ 建注進 [MQ]、worker 消費落 DB
│   └─ 凍結↔落注對帳 + 失敗解凍補償（跨儲存最終一致）
├─ 為何綁：凍結一離開同步 DB 交易，就得靠對帳 + 補償保「凍了就一定有注、失敗一定解凍」
└─ 撐不住的訊號 → 單一熱門桌 / 局仍是熱點

【P3 · 按 gameId / tableId 分區】約 50,000 QPS
├─ 定義：把注按桌 / 局分區，熱點打散、各區獨立處理
├─ 大方塊
│   ├─ 請求按 tableId / roundId 路由到分區
│   ├─ 每區獨立的凍結帳 + 受理序列
│   └─ 封盤時間邊界 per 區判定
├─ 為何綁：一分區，凍結帳、受理序、封盤判定都得跟著切到區內，否則跨區不一致
└─ 撐不住的訊號 → 少數超熱門桌吃掉整區資源

【P4 · 熱門桌獨立資源池 + 冷玩法限流】約 100,000+ QPS
├─ 定義：把超熱門桌隔離成獨立池，非核心玩法限流讓路
├─ 大方塊：熱桌獨立部署池 · 冷玩法令牌桶限流 · 封盤前流量整形
└─ 為何綁：隔離熱桌 = 資源池 + 路由 + 限流策略，一起變
```
