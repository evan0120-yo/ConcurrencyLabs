# 下注入口系統 · s04_bettingentry

> 一句話定位：玩家在某桌 / 某局下注，系統先驗餘額、凍結金額再受理，封盤前尖峰湧入。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：錢包帳本型（凍結金額）、熱點型（熱門桌 / 局）、低延遲型（下注要快）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 下注要先凍結金額，不能讓玩家押超過餘額（透支）。
- **熱點型** → 熱門桌 / 熱門局的注會集中打同一個 gameId / tableId。
- **低延遲型** → 玩家在乎「有沒有押到」，且封盤前分秒必爭。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
玩家在某桌下注 100
   │
   ▼
[下注入口] 驗餘額 ──> 凍結 100 ──> 受理這注
   ├─ (餘額夠) ──> 受理，等開獎
   └─ (餘額不足 / 已封盤) ──> 拒絕

────────── 封盤前最後幾秒、熱門桌 ──────────

玩家想要：下注要快、馬上確認押到沒
營運想要：不能透支、封盤後一注都不收
```

### 分段解說

- 正常流程：驗餘額 → 凍結 → 受理，很直白。
- 難點在時間邊界（封盤瞬間）與熱點集中（同一桌）。
- 凍結是錢包帳本的一部分，錯了就是資損。

### 痛點分析

- **有沒有熱點？** 有，熱門桌 / 局集中。
- **時間邊界** → 封盤是硬邊界，封盤後的注必須乾淨拒絕。
- **延遲要求** → 下注體感要快，慢了玩家會抱怨押不到。
- **真正的痛**：熱門桌尖峰 + 封盤瞬間 + 凍結正確性，三個壓力擠在同一個入口。
