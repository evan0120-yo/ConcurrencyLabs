# 權限中心 RBAC / ABAC · s31_rbacabac

> 一句話定位：每個操作都要問「這個人能不能做」，權限查詢極高頻，變更要能即時生效。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：會員 / Gateway / 多租戶
- 壓力模型：讀爆型（每操作一問）、多租戶型（tenant / role 分區）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 幾乎每個受保護操作都要查一次權限，是高頻讀。
- **多租戶型** → 權限依 tenant / role 劃分，不同租戶規則獨立。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者要做某操作（看報表 / 改設定 / 刪資料）
   │
   ▼
[權限中心] 問：這個 role / 屬性 能做這件事嗎？
   ├─ (可以) ──> 放行
   └─ (不行) ──> 拒絕

────────── 管理員剛剛收回某人的權限 ──────────

   └──> 要「馬上」生效，不能讓他還能用舊權限操作
```

### 分段解說

- 權限查詢頻率極高，慢了整體體驗都受影響，所以要快取。
- 但快取跟「即時生效」矛盾：權限被收回時，舊快取要能失效。
- 多租戶讓規則更複雜：每個租戶有自己的角色體系。

### 痛點分析

- **讀爆還寫爆？** 讀爆（每操作一問）。
- **快取 vs 即時** → 收權要即時生效，快取要能失效。
- **不該打主庫** → 查權限的路徑不能拖垮主資料庫。
- **真正的痛**：極高頻的權限查詢要靠快取扛，但「權限變更即時生效」又逼快取必須能快速失效。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s31 極高頻權限查詢靠快取扛、變更要即時失效,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：每操作查權限庫(RBAC/ABAC 評估)
├─ 大方塊：[DB] role/policy 查詢 · 權限評估
└─ 撐不住的訊號 → 每操作一問打主庫、變更後舊權限殘留

【P2 · 權限快取 + 變更即時失效】約 10,000 QPS
├─ 定義：權限決策快取扛讀爆,變更主動失效使即時生效
├─ 大方塊（綁一起）
│   ├─ [Redis] 決策快取(讀爆靠這扛)
│   ├─ 變更事件主動失效相關快取
│   └─ 容忍極短傳播延遲
├─ 為何綁：快取扛讀 + 變更即時失效是矛盾,靠主動失效一起解
└─ 撐不住的訊號 → 多租戶規則放大、熱門權限查詢集中

【P3 · 多租戶分區 + 多層快取(local+redis)】約 50,000+ QPS
├─ 定義：權限按 tenant 分區,決策走 local + redis 多層快取
├─ 大方塊：tenant 分區 · 多層快取 · 失效傳播
└─ 為何綁：極高頻查詢下,分區 + 多層快取 + 失效傳播,一起做
```
