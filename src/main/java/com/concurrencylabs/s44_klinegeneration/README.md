# K 線生成系統 · s44_klinegeneration

> 一句話定位：把成交 tick 聚合成各種週期的 K 線（1 分 / 5 分 / 日線），熱門即時、冷門延遲。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：交易所 / Web3
- 壓力模型：寫爆型 / ingestion（tick 流入）、讀爆型（看圖）
- 數據角色：即時聚合 / stream processing

---

## Block 2：為什麼被列上這些

- **寫爆 + 即時聚合** → 成交 tick 是高頻資料流，要即時聚合成不同時間窗。
- **讀爆型** → 生成的 K 線被大量看盤者讀取。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
成交 tick 源源不絕
   tick tick tick ──> [K 線生成] 依時間窗聚合
                         ├─ 1 分 K / 5 分 K / 1 小時 K / 日 K
                         └─ 熱門 symbol → 即時算；冷門 → 延遲算
   ▼
   使用者看圖：要能看到「正在成形」的那根 K

────────── 一根 K 線收尾瞬間，要正確定案 ──────────
```

### 分段解說

- K 線是 tick 的「時間窗聚合」，同一份 tick 要餵多個週期。
- 當前這根 K 一直在變（未收），收尾時要正確定案。
- 熱門 symbol 值得即時算，冷門可以延遲產。

### 痛點分析

- **讀爆還寫爆？** 進來寫爆（tick）、出去讀爆（看圖）。
- **即時聚合** → 多週期同時維護，當前 K 持續更新。
- **熱冷分級** → 熱門即時、冷門延遲。
- **真正的痛**：高頻 tick 要同時餵養多個時間週期的 K 線，還要正確處理「當前未收 K」的即時更新與收尾定案。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+（tick 流入量）。每跳一個平台 = 一次架構改造。
> s44 寫爆聚合+讀爆看圖、熱冷可分級,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：tick 即時聚合多週期 K,當前 K 更新+收尾定案
├─ 大方塊：多週期聚合 · 當前 K 更新 · 收尾落地
└─ 撐不住的訊號 → 高頻 tick 餵多週期、看圖讀爆、當前 K 高頻更新

【P2 · stream 多週期聚合 + 當前 K 快取 + 讀寫分離】約 10,000 QPS
├─ 定義：tick 走 stream 聚合各週期,當前 K 放快取即時供讀,已收 K 落地
├─ 大方塊（綁一起）
│   ├─ stream 多週期時間窗聚合
│   ├─ [Redis] 當前 K 快取(即時供看圖)
│   └─ 已收 K 落時序庫 + 讀快取
├─ 為何綁：寫爆聚合 + 讀爆看圖 = stream + 當前K快取 + 讀寫分離,一起做
└─ 撐不住的訊號 → symbol 數多、熱冷混雜、單聚合節點吃不下

【P3 · 按 symbol 分片聚合 + 熱冷分級 + 讀快取分層】約 50,000+ QPS
├─ 定義：聚合按 symbol 分片並行,熱門即時/冷門延遲,看圖多層讀快取
├─ 大方塊：symbol 分片聚合 · 熱冷分級 · 讀快取分層
└─ 為何綁：規模化 K 線 = 分片聚合 + 熱冷分級 + 讀快取,一起做
```
