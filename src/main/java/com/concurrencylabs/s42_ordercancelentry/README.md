# 下單 / 撤單入口 · s42_ordercancelentry

> 一句話定位：撮合前的守門員，驗證、風控、凍結資產後才把單送進撮合，爆量時要優先保住撮合。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：交易所 / Web3
- 壓力模型：低延遲型（入口要快）、錢包帳本型（下單凍結資產）

---

## Block 2：為什麼被列上這些

- **低延遲型** → 下單入口在撮合前，慢了整體體驗都差。
- **錢包帳本型** → 下單要凍結資產，撤單要解凍，涉及資產正確性。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者下單 / 撤單
   │
   ▼
[下單入口] 驗證 ──> 風控 ──> 凍結資產 ──> 送進撮合
   ├─ (通過) ──> 排進撮合
   └─ (餘額不足 / 風控擋) ──> 拒絕

────────── 行情暴動，大家瘋狂下單撤單 ──────────

營運想要：入口爆量時，優先保住撮合本體不被拖垮
```

### 分段解說

- 入口做的是「前置檢查 + 凍結」，然後才交給撮合。
- 下單與查詢要隔離：查詢流量不能擠掉下單。
- 爆量時要有取捨：保撮合優先於保周邊。

### 痛點分析

- **延遲要求** → 入口要快，別成為撮合前的瓶頸。
- **資產正確** → 凍結 / 解凍不能錯。
- **爆量取捨** → 優先保撮合，查詢可讓路。
- **真正的痛**：作為撮合前的閘門，爆量時要守住「下單資產正確」又不能拖累後面的撮合核心。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s42 撮合前守門、凍結委派錢包,重點在低延遲+爆量保撮合 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：驗證→風控→凍結資產→送撮合,同步
├─ 大方塊：驗證風控 · 凍結資產 · 送撮合 · clientOrderId 去重
└─ 撐不住的訊號 → 行情暴動瘋狂下單撤單、查詢擠掉下單

【P2 · 凍結原子化 + 下單/查詢隔離】約 10,000 QPS
├─ 定義：資產凍結走原子帳,下單路徑與查詢路徑資源隔離
├─ 大方塊（綁一起）
│   ├─ 資產凍結/解凍原子(委派 s48 錢包)
│   ├─ 下單與查詢隔離(查詢不擠下單)
│   └─ 風控前置快取化(毫秒)
├─ 為何綁：入口要快又不錯凍結 = 原子凍結 + 讀寫隔離,一起做
└─ 撐不住的訊號 → 熱門交易對入口集中、爆量拖累撮合送單

【P3 · 按 symbol 分區入口 + 爆量保撮合降級】約 50,000+ QPS
├─ 定義：入口按 symbol 分區,爆量時優先保撮合送單、查詢/周邊降級
├─ 大方塊：symbol 分區入口 · 爆量保撮合優先級 · 查詢讓路降級
└─ 為何綁：爆量取捨 = 分區 + 優先保撮合 + 降級,一起做
```
