# 抽獎活動系統 · s01_lottery

> 一句話定位：使用者點一下抽獎，系統從「有限獎品池」公平發獎，尖峰時大量人同時抽。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：博弈 / 遊戲
- 壓力模型：搶資源型（核心）、熱點計數型（剩餘獎品數）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 獎品數量有限，大量人同時搶同一批庫存，怕超發。
- **熱點計數型** → 「剩餘獎品數」是單一熱 key，尖峰被瘋狂讀寫。
- **博弈 domain** → 通常掛在活動 / 拉新場景，跟 `s02_redpacketrain` 同源。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
【平常】玩家點「抽獎」──> 立刻看到「中 5 元」或「銘謝惠顧」

────────────── 整點開搶，10 萬人同一秒抽 ──────────────

10 萬人同時點抽獎
   │
   ├─ 玩家想要：點下去馬上知道中沒中、覺得公平
   ├─ 營運想要：獎品發完為止，絕不能超發（成本會失控）
   └─ 系統面對：同一批有限獎品被 10 萬人同時搶
        ├─ (中獎) ──> 發獎、記錄
        └─ (沒中) ──> 秒回銘謝惠遠
```

### 分段解說

- 平常：抽獎是零星請求，隨手一點。
- 尖峰：整點開放瞬間所有人湧入，同搶一批固定獎品。
- 核心矛盾：獎品是有限成本，發多了就是實質資損。

### 痛點分析

- **能不能超發？** 絕對不行，超發 = 直接資損。
- **公平性** → 中獎機率要可控，且不能被腳本刷爆。
- **平均還是尖峰？** 極端尖峰，集中在開放的頭幾秒。
- **真正的痛**：剩餘獎品數這個熱點，要同時扛住「高頻爭搶」與「絕不超發」兩個要求。

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。往上每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> QPS 是改造完的容量估算（果），不是目標。s01 因「錢＋獎品都不能錯」，連鎖綁得最緊 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS ── 已實作（s01/P1）
├─ 定義：無狀態 app ×N，Redis 扛併發正確性，DB 落結果
├─ 大方塊：[Redis] 原子扣獎品庫存(Lua 判空+扣減) · [Redis] SET NX 去重 · [DB] 同步寫 draw_record
└─ 撐不住的訊號 → draw_record insert P99 飆、account 呼叫拖慢

【P2 · 寫路徑削峰】約 10,000 QPS
├─ 定義：抽獎與落庫解耦，尖峰只保證抽獎結果，落庫轉非同步
├─ 大方塊（必須一起改）
│   ├─ [MQ] 抽獎請求進 MQ、worker 消費落 draw_record
│   ├─ 回應語意改「先回中獎結果、後落庫」
│   ├─ 冪等從『DB 唯一鍵』上移到『[Redis] 去重』保 exactly-once
│   └─ [account] 扣積分加補償（先扣、失敗回退）
├─ 為何綁：draw_record 一改非同步，回應就不能等 DB → 逼你把「只發生一次」
│          整包搬到 Redis，補償才成立。這四塊拆不開
└─ 撐不住的訊號 → 單一 stock key 熱點、抽獎運算變重

【P3 · 熱點打散 / 預生成】約 50,000 QPS
├─ 定義：拆掉「單一庫存熱 key」與「即時運算抽獎」
├─ 大方塊
│   ├─ [Redis] stock key 分桶（單 key 熱點打散成多桶）
│   ├─ 抽獎演算法改「跨桶挑選 + 權重重分配」
│   └─（或）活動前預生成中獎序列，搶 = 取 token（把運算變取號）
├─ 為何綁：庫存一分桶，挑選邏輯必須跟著改；走預生成又得多一條離線 job + token 池
└─ 撐不住的訊號 → 活動級洪峰波及其他業務

【P4 · 活動獨立 cell】約 100,000+ QPS
├─ 定義：整個抽獎活動隔離成獨立資源池，犧牲即時性換容量
├─ 大方塊：活動整包獨立部署(獨立 Redis/DB/app pool) · 結果延遲揭曉 · 活動後資料回寫主庫
└─ 為何綁：隔離 = 部署 + 資料同步 + 前端延遲揭曉語意，一起變
```
