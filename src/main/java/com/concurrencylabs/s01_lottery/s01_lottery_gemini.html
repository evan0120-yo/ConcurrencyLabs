Block 3 // 元件架構演進詳解
1. Redis 庫存原子操作 & 叢集 L1 & L3

Redis 負責活動的去重攔截與庫存即時原子扣減，其架構因 QPS 規模而劇烈演進。

  • 去重鎖定使用 `SET idem:key PENDING NX EX 60` 進行去重攔截。
  • Lua 原子扣庫存依靠單執行緒 Lua 腳本實作「判斷大於 0 並扣減庫存」的原子指令，防止超發。
-- Redis Lua script atomic decr
local r = redis.call('GET', KEYS[1])
if r == false then return -1 end
if tonumber(r) > 0 then
    redis.call('DECR', KEYS[1])
    return 1
end
return 0
  • 去重 Key 分散將去重 Key 依 user_id 計算雜湊，天然分散至 Redis Cluster 各個 Slot 分片節點。
  • 預分配 Token list將單一獎品餘量 Key 改為 Token list 機制（把獎品變成實體序列如 `T_IPHONE_01` 並打散）。App 僅對分片 List 進行 `LPOP`，避開了單一 Hot Key 被 Lua 單點頻繁鎖定的效能上限。
  • QPS 3,000 (L1)單點 Redis 足以應對此量級，去重與扣庫存置於同一 instance，結構最為單純。
  • QPS 50,000 (L3)必須分離去重資料與庫存資料。去重依靠 Redis Cluster 分片；庫存則改用 Token 預發機制以繞過熱點寫入瓶頸。
2. 消息佇列 (MQ) 削峰 L2 引入

為避免資料庫寫入效能崩潰，利用 MQ 作為流量蓄水池，將寫入與同步抽獎請求解耦。

  • 同步阻塞應用伺服器在扣減庫存成功後，同步執行 `drawRecordRepository.save(...)`。當 DB 壓力上升，主線程將全面卡死。
  • 異步削峰 (Write-Behind)抽獎完成後立即向 MQ 發送流水落庫事件並返回 Response。流水資料寫入由後台 Kafka Consumer 分批批次寫入 (Batch Insert)。
  • 回放幂等保證異步落庫期間，使用者重試仍必須由 Redis 中保留的完成狀態 (`idem:done`) 來進行最終幂等攔截。
3. PostgreSQL 同步與分表 L1 & L3

資料庫負責儲存流水與活動 Meta 唯讀資料。隨著 QPS 的增長，必須變更儲存架構以降低鎖競爭。

  • 行級樂觀鎖扣點[AccountRepository.java:L18-L21](file:///D:/WorkSpace/ConcurrencyLabs/src/main/java/com/concurrencylabs/common/account/AccountRepository.java#L18-L21) 透過 DB 原生的樂觀鎖行級控制 `points = points - :amount where id = :id and points >= :amount` 安全扣除點數。
  • 唯一去重索引抽獎紀錄表依 `idempotency_key` 建立唯一索引以進行最後的資料完整性防線。
  • 分表 (Sharding)`lottery_draw_record` 使用 `user_id` 作為分片鍵進行水平分片，將大量的併發 Insert 流量分散到不同的實體硬碟與 CPU 上。
  • 冷熱資料分離高頻寫入的流水在 7 天後定期歸檔移出主庫，維持線上 DB 索引樹的極致讀寫速度。
4. 入口網關限流與 JVM 本地快取 L4 引入

面對數十萬 QPS 的洪水型流量，必須在最上游層層節流（漏斗防禦），避免系統被打穿。

  • 直接命中所有請求直接穿透至 Usecase 處理，未開始的活動若遭遇大量腳本刷單，會瞬間打滿 DB 與 Redis 連接池。
  • API Gateway 令牌桶在網關層 Kong/Nginx 控制並丟棄超過系統穩態極限的流量，回傳售罄或流量限流回應。
  • JVM 本地快取將活動的唯讀參數快取在伺服器本機的 JVM Heap 內（使用 Caffeine，TTL 5秒），使 checkActivity 的前置檢查免除 Redis/DB 調用，保護核心儲存介質。
5. 積分扣除與 Saga 事務補償 跨等級

抽獎包含扣點與原子發獎，兩者分散在 PostgreSQL 與 Redis 中。此處利用 Saga 補償保持最終一致。

  • 正向扣點後原子抽獎主鏈路先行調用 [AccountService.java:L22](file:///D:/WorkSpace/ConcurrencyLabs/src/main/java/com/concurrencylabs/common/account/AccountService.java#L22) 扣積分。扣點成功後才執行 Redis Lua 原子扣庫存。
  • 反向退款補償若 Redis 扣庫存失敗、或記流水出現異常，正向邏輯將在中斷後觸發 [AccountService.java:L29](file:///D:/WorkSpace/ConcurrencyLabs/src/main/java/com/concurrencylabs/common/account/AccountService.java#L29) 退還積分 (refund)。整個 Usecase 為無 DB 交易鎖的 Saga 設計。
