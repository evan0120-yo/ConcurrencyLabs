Block 3 // 元件架構演進詳解
1. Redis 去重與庫存熱點 P1 & P3

Redis 負責活動的去重攔截與庫存即時原子扣減；真正要看的不是總 QPS，而是哪個平台瓶頸正在逼近天花板。

  • 去重鎖定使用 `SET idem:key PENDING NX EX 60` 進行去重攔截。
  • Lua 原子扣庫存依靠單執行緒 Lua 腳本實作「判斷大於 0 並扣減庫存」的原子指令，防止超發。
P1 不變量：
- 去重 claim 必須是共享且原子的
- 庫存扣減必須把判斷與扣減放在同一個原子步驟
- 教學重點是不變量，不是對應到某個 Java 方法
  • 去重 Key 分散將去重 Key 依 user_id 計算雜湊，天然分散至 Redis 各個 Slot；庫存熱點則靠分桶或 token 池拆解。
  • 預分配 Token 清單將單一獎品餘量 Key 改為 Token 清單 機制（把獎品變成實體序列如 `T_IPHONE_01` 並打散）。App 僅對分片 List 進行 `LPOP`，避開了單一 熱 key 被 Lua 單點頻繁鎖定的效能上限。
  • P1 基線（約 3,000 QPS）單點 Redis 足以應對此量級，去重與扣庫存置於同一 instance，結構最為單純。
  • P3 熱點平台（約 50,000 QPS）必須分離去重資料與庫存資料。去重依靠 Redis 分片或分桶；庫存則改用 Token 預發機制以繞過熱點寫入瓶頸。
2. 消息佇列 (MQ) 削峰 P2 引入

為避免資料庫寫入效能崩潰，利用 MQ 作為流量蓄水池，將寫入與同步抽獎請求解耦。

  • 同步阻塞P1 在抽獎完成後同步寫入 draw_record；當 DB insert 與索引更新變慢，整條抽獎請求都會被寫路徑拖住。
  • 寫路徑削峰抽獎結果先成立並回應，流水落庫改由 MQ / outbox 承接，背景工作者批次寫入，避免抽獎請求等待 DB。
  • 回放幂等保證異步落庫期間，使用者重試仍必須由 Redis 中保留的完成狀態 (`idem:done`) 來進行最終幂等攔截。
3. PostgreSQL 流水與對帳 P1 & P3

資料庫負責活動設定與抽獎流水。P1 先保留同步落庫；P2 開始把流水寫入移出同步路徑，但仍要保留可回放、可對帳的結果事件。

  • 行級樂觀鎖扣點積分扣減靠共享資料庫的原子條件更新，確保餘額不足時不透支；這是共用 account 能力，不在抽獎頁綁定程式碼位置。
  • 唯一去重索引抽獎紀錄表依 `idempotency_key` 建立唯一索引以進行最後的資料完整性防線。
  • 分表 (Sharding)`lottery_draw_record` 使用 `user_id` 作為分片鍵進行水平分片，將大量的併發 Insert 流量分散到不同的實體硬碟與 CPU 上。
  • 冷熱資料分離高頻寫入的流水在 7 天後定期歸檔移出主庫，維持線上 DB 索引樹的極致讀寫速度。
4. 入口網關限流與 JVM 本地快取 P4 引入

面對大型活動尖峰，P4 的主角是活動獨立 cell：入口配額先擋住洪峰，後端 app / Redis / DB pool 也要與其他活動隔離。

  • 直接命中P1 所有請求直接進入核心抽獎流程；若活動入口被腳本或重試洪峰打滿，DB、Redis、account 連接池都會一起承壓。
  • 入口配額與 cell 准入在進入活動 cell 前先控配額；超出容量的請求走排隊、售罄或限流回應，避免進入有狀態路徑後才失控。
  • JVM 本地快取將活動的唯讀參數快取在伺服器本機記憶體內（例如 Caffeine，TTL 5 秒），讓活動前置檢查少打 Redis/DB，保護核心儲存介質。
5. 積分扣除與補償邊界 跨平台

抽獎包含扣點與原子發獎，兩者分散在 PostgreSQL 與 Redis 中。此處利用 Saga 補償保持最終一致。

  • 先扣點，再抽獎抽獎流程先扣 200 積分，扣點成功後才碰獎品庫存，避免先送出稀缺獎品卻沒有收到成本。
  • 系統失敗才補償若扣點成功後，抽獎品或記錄結果發生系統失敗，才退還積分；LOSE 是正常結果，不做退款。
