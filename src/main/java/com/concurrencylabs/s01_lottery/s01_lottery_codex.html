ConcurrencyLabs / S01 抽獎

抽獎系統 P1-P4 容量平台演進圖

這份 HTML 是教學用視覺稿：先把 P1 的抽獎流程放到底層，再把 P2、P3、P4 變成一層層疊上去的平台能力。

核心概念 架構大圖 細節

Block 1 / 核心概念

S01 的主線不是「抽中什麼獎」而已，而是高併發下仍要守住：不重複扣點、不超賣獎品、失敗能補償、重試能拿到同一個結果。

故事 / 平台 / 觀察

使用者故事

  • 請求
    玩家用 200 點參加抽獎 送出 activityId、userId、idempotencyKey，進入抽獎 API。
  • 守門
    系統先守活動狀態與重試語意 活動必須 RUNNING；同一個 idempotencyKey 不能產生兩次扣點或兩筆有效抽獎結果。
  • 抽獎
    扣點後執行權重抽獎與庫存扣減 中獎時要用 Redis Lua 原子扣庫存；扣不到庫存就回 LOSE。
  • 記錄
    記錄中獎 / 未中，失敗時補償扣點 抽獎與寫紀錄之間若出現系統失敗，需要退回積分並釋放去重標記。

容量平台下要看什麼

P1 / 約 3k

基線平台：同步抽獎、Redis 去重與原子扣庫存、DB 同步落流水，先證明不超發與不重扣。

P2 / 約 10k

寫路徑削峰：抽獎結果先成立，流水落庫轉 MQ / outbox，去重與補償語意要一起上移。

P3 / 約 50k

看同步路徑長度：DB 寫入、積分服務、結果回放是否拖長 P99。

P4 / 100k+

活動獨立 cell：入口配額、服務 / Redis / DB 池 隔離，保護核心一致性元件。

正確性 超發、重複扣點、重複流水、退款不一致。
延遲 P95/P99、Redis Lua 延遲、積分服務延遲、DB 寫入延遲。
容量 單次請求 Redis 操作數、DB 連線池、hot key 傾斜、重試比例。

Block 2 / 架構大圖

這張圖改成四層立體平台：P1 是正確性基線，P2 疊上寫路徑削峰，P3 疊上熱點打散與預生成，P4 疊上入口准入與活動獨立 cell。每往上一層，就是切到下一個 P 時新增的能力。

P1-P4 立體分層
P1 基線正確性 P2 寫路徑削峰 P3 熱點打散 / 預生成 P4 活動獨立 cell

Block 3 / 細節

細節區把大圖上的節點拆成「症狀、處方、要觀察」三件事。重點是容量平台的能力層，不把核心敘事綁到某個程式檔或行號。

症狀 / 處方 / 觀察

HTTP 入口與 寫入緩衝

P1 -> P2

P1 目前重點是同步抽獎正確完成。到 P2 時，重點不是單純加機器，而是把抽獎結果與落流水寫入拆開，避免 DB insert 拖住整條鏈路。

  • 症狀draw_record 寫入 P99 飆高，積分服務呼叫變慢，同步請求時間被寫路徑拉長。
  • 處方App 保持無狀態，前面放負載平衡；跨請求狀態都放在 Redis 或 DB。
  • 觀察單機 P99、執行緒池佇列、重試比例、重複 idempotencyKey 回放。

P1 / 基線 Redis

P1

P1 的 Redis 同時扛兩件事：去重 claim 與獎品庫存扣減。教學重點是先證明原子性，而不是一開始就追求最終規模。

  • 症狀沒有外部 claim 會造成重試重複扣點；沒有 Lua 原子扣庫存會造成超賣。
  • 處方idempotency 使用 SET NX + TTL；stock 使用 GET/DECR 包在 Lua script 中。
  • 觀察Redis Lua 延遲、庫存 key 遺失、庫存負數、PENDING key 殘留量。

P3 / 庫存分桶與 token

P3

P3 處理的不是「Redis 換成叢集」這麼單純，而是單一獎品 stock key 的熱點。常見解法是分桶扣減，或活動前預生成中獎 token，讓即時請求從運算變成取號。

  • 症狀單 Redis CPU、network、memory 或 failover 風險成為瓶頸。
  • 處方idem key 可自然分散；stock key 需檢查獎品熱點，必要時做 bucket stock。
  • 觀察slot 分布、hot key 操作量、叢集切換時間、Lua cross-slot 限制。

Redis 外框 / 平台選型

跨平台

Redis 不是單看總流量，而是看每次抽獎打幾次 Redis、key 是否集中、失敗時是否會被 retry 放大。

  • 估算每次 draw 約有 claim、get claim、stock eval、mark done；高 retry 會再放大。
  • 拆分idem 是大量分散 key；stock 是少量熱點 key，兩者壓力型態不同。
  • 升級P1 看原子性，P2 看寫路徑削峰，P3 看 hot key 拆解，P4 看活動 cell 與入口配額。
如果某個獎品很熱門，Redis 叢集只能把不同 key 分到不同 slot；同一個獎品庫存 key 仍可能集中在單一 shard。

common.account 扣點與補償

P1

抽獎不是只改 lottery 狀態，還會碰到使用者點數。P1 用同步扣點與失敗 refund 讓流程容易看懂，但這也是後續 P99 的重要來源。

  • 症狀積分服務延遲會抬高整體抽獎延遲；失敗時補償不完整會造成點數不一致。
  • 處方扣點必須有可重試語意；退款要能和抽獎失敗對帳。
  • 觀察扣點成功率、退款筆數、帳務不一致、積分服務逾時。

PostgreSQL 抽獎流水

P1

P1 同步寫入 draw_record，讓重送請求可以用 idempotencyKey 找回結果。這張表既是結果歷史，也是重試語意的落點。

  • 症狀容量上升後，寫入、索引更新、連線池等待會影響同步抽獎路徑。
  • 處方idempotencyKey 保持唯一；結果回放查詢需要穩定索引。
  • 觀察寫入 P99、唯一鍵衝突、連線池等待、activityId/userId 查詢比例。

P2 / MQ、outbox 與結果投影

P2

P2 的精神是縮短同步抽獎路徑，把抽獎結果與流水落庫解耦；通知、分析、排行榜、報表投影都不應拉長 抽獎請求。

  • 症狀同步路徑塞太多寫入與外部呼叫，P99 會被最慢的依賴綁住。
  • 處方核心結果先可靠落地；其他衍生資料走 outbox 或 MQ consumer。
  • 觀察消費延遲、outbox 重試、投影新鮮度、回放一致性。

P4 / 活動獨立 cell

P4

P4 不再只靠後端硬扛，而是把大型活動切成獨立 cell：入口先控配額，後端用獨立 服務 / Redis / DB 池 承接，避免一個活動拖垮共用資源。

  • 症狀尖峰 burst 遠大於穩態容量，後端即使可水平擴展也會被瞬間重試放大。
  • 處方token bucket、排隊、預發 draw token、售罄快速回覆、依活動分流。
  • 觀察准入比例、拒絕原因、排隊等待、核心抽獎路徑的穩態 P99。
