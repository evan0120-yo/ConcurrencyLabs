Block 1 / 核心概念

S01 的主線不是「抽中什麼獎」而已，而是高併發下仍要守住：不重複扣點、不超賣獎品、失敗能補償、重試能拿到同一個結果。

user story + QPS + observation

User Story

  • request
    玩家用 200 點參加抽獎 送出 activityId、userId、idempotencyKey，呼叫 POST /api/lottery/{activityId}/draw。
  • guard
    系統先守活動狀態與重試語意 活動必須 RUNNING；同一個 idempotencyKey 不能產生兩次扣點或兩筆有效抽獎結果。
  • draw
    扣點後執行權重抽獎與庫存扣減 中獎時要用 Redis Lua 原子扣庫存；扣不到庫存就回 LOSE。
  • record
    記錄 WIN / LOSE，失敗時補償扣點 抽獎與寫紀錄之間若出現系統失敗，需要 refund points 並釋放 claim。

這些 QPS 下要看什麼

L1 / 3k

先看正確性：idempotency、Redis Lua、DB unique key、補償是否能兜住基本併發。

L2 / 10k

看橫向擴展：App replicas、Redis cluster、DB pool、熱點獎品 key 是否集中。

L3 / 30k

看同步路徑長度：DB insert、account service、record replay 是否拖長 P99。

L4 / burst

看入口治理：排隊、令牌、預扣名額、降級，以及能不能快速回覆 sold out。

Correctness oversell、duplicate charge、duplicate record、refund mismatch。
Latency P95/P99、Redis eval latency、account latency、DB insert latency。
Capacity Redis ops/request、DB connection pool、hot key skew、retry ratio。

Block 3 / Detail

Detail 區把大圖上的節點拆成「症狀、處方、要觀察」三件事。之後 L2-L4 真正開始 coding，可以把對應 package 的實作連回這些 detail。

symptom / prescription / watch

HTTP 入口與 App replicas

L1 -> L2

L1 目前重點是單一同步 usecase 正確完成抽獎。到 L2 時，API 層可以水平擴展，但 claim、庫存、紀錄都不能依賴單機記憶體。

  • 症狀QPS 上升後，單一 Spring instance CPU、thread pool、DB pool 都會先被打滿。
  • 處方App stateless 化，前面放 LB；所有跨 request 狀態放 Redis 或 DB。
  • 觀察每台 instance 的 P99、thread pool queue、retry ratio、duplicate idemKey replay。

L1 / Single Redis

L1

L1 的 Redis 同時扛兩件事：idempotency claim 與 prize stock 扣減。教學重點是先證明原子性，而不是一開始就追求最終規模。

  • 症狀沒有外部 claim 會造成重試重複扣點；沒有 Lua 原子扣庫存會造成超賣。
  • 處方idempotency 使用 SET NX + TTL；stock 使用 GET/DECR 包在 Lua script 中。
  • 觀察Redis eval latency、missing key、stock negative、PENDING key 殘留量。

L2 / Redis Cluster

L2

L2 把 Redis 從單點變成可擴展或可 failover 的形狀。抽獎庫存通常會遇到 hot key，cluster 解掉容量問題但不會自動解掉單獎品熱點。

  • 症狀單 Redis CPU、network、memory 或 failover 風險成為瓶頸。
  • 處方idem key 可自然分散；stock key 需檢查獎品熱點，必要時做 bucket stock。
  • 觀察slot distribution、hot key ops、cluster failover time、Lua cross-slot 限制。

Redis 外框 / QPS 對選型的影響

Cross level

Redis 不是單看總 QPS，而是看每次抽獎打幾次 Redis、key 是否集中、失敗時是否會被 retry 放大。

  • 估算每次 draw 約有 claim、get claim、stock eval、mark done；高 retry 會再放大。
  • 拆分idem 是大量分散 key；stock 是少量熱點 key，兩者壓力型態不同。
  • 升級L1 看原子性，L2 看 cluster，L3 看事件化，L4 看入口削峰與預扣名額。
如果某個獎品很熱門，Redis cluster 只能把不同 key 分到不同 slot；同一個 prize stock key 仍可能集中在單一 shard。

common.account 扣點與補償

L1

抽獎不是只改 lottery 狀態，還會碰到使用者點數。L1 用同步扣點與失敗 refund 讓流程容易看懂，但這也是後續 P99 的重要來源。

  • 症狀account latency 抬高整體 draw latency；失敗時補償不完整會造成點數不一致。
  • 處方扣點必須有可重試語意；refund 要能和 draw failure 對帳。
  • 觀察deduct success rate、refund count、ledger mismatch、account timeout。

PostgreSQL draw_record

L1

L1 同步寫入 draw_record，讓 replay 可以用 idempotencyKey 找回結果。這張表既是結果歷史，也是重試語意的落點。

  • 症狀高 QPS 下 insert、index update、connection pool 會影響同步 draw path。
  • 處方idempotencyKey 保持 unique；查詢 replay 需要穩定索引。
  • 觀察insert P99、unique violation、pool wait、activityId/userId 查詢比例。

L3 / Outbox、MQ 與結果投影

L3

L3 的精神是縮短同步抽獎路徑，把通知、分析、排行榜、報表投影等非核心工作移出 draw request。

  • 症狀同步路徑塞太多寫入與外部呼叫，P99 會被最慢的依賴綁住。
  • 處方核心結果先可靠落地；其他衍生資料走 outbox 或 MQ consumer。
  • 觀察consumer lag、outbox retry、projection freshness、replay 一致性。

L4 / Traffic gate、令牌與降級

L4

L4 不再只靠後端硬扛，而是在入口先決定誰能進入抽獎核心路徑。這能保護 Redis、DB、account，不讓尖峰把一致性元件打穿。

  • 症狀尖峰 burst 遠大於穩態 QPS，後端即使可水平擴展也會被瞬間重試放大。
  • 處方token bucket、排隊、預發 draw token、售罄快速回覆、依活動分流。
  • 觀察admission rate、reject reason、queue wait、核心 draw path 的穩態 P99。
