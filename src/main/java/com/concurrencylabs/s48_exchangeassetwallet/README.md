# 交易所資產錢包 · s48_exchangeassetwallet

> 一句話定位：管理每個使用者的可用 / 凍結資產，下單凍結、成交扣減，核心不可被非核心流量拖垮。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：交易所 / Web3
- 壓力模型：錢包帳本型（核心）、熱點型（活躍帳戶 / 熱門幣種）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 可用 / 凍結資產的每次變動都要正確、可追溯、冪等。
- **熱點型** → 活躍交易者與熱門幣種的資產帳戶被高頻讀寫。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
資產有兩種狀態：可用 / 凍結
   下單    ──> 可用 -X、凍結 +X
   成交    ──> 凍結 -X、對手方資產 +Y
   撤單    ──> 凍結 -X、可用 +X

   [資產錢包] 被下單、撮合、充提、活動全部依賴

────────── 撮合成交事件如雨點般打進來 ──────────

平台想要：資產核心跟行情 / 活動 / 通知完全隔離
```

### 分段解說

- 資產有「可用 / 凍結」兩態，各種操作在兩態間搬。
- 它是交易所的地基，被撮合、充提、活動全部依賴。
- 必須跟行情、活動、通知這些非核心流量隔離，不能被拖垮。

### 痛點分析

- **能不能錯？** 資產是真錢，一分都不能錯。
- **有沒有熱點？** 活躍帳戶 / 熱門幣種是熱點。
- **要不要隔離？** 要，資產核心不可被非核心流量拖垮。
- **真正的痛**：作為交易所地基，資產帳本要在「成交事件雨」下維持每筆正確，並跟所有非核心流量硬隔離。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s48 交易所地基,成交事件雨下每筆正確 + 與非核心硬隔離,壓力最高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：可用/凍結兩態帳,原子變更+append 流水,bizNo 冪等
├─ 大方塊：兩態原子變更(不透支) · asset_ledger append · bizNo 去重
└─ 撐不住的訊號 → 成交事件雨、活躍帳戶/熱門幣種熱點、非核心流量拖累

【P2 · append 流水 + 餘額快照 + 讀寫分離】約 10,000 QPS
├─ 定義：以流水為真相 append-only,餘額快照,核心寫與查詢讀分離
├─ 大方塊（綁一起）
│   ├─ asset_ledger 純 append(免行鎖競爭)
│   ├─ 可用/凍結快照由流水推導
│   └─ 資產核心與非核心(行情/活動/通知)硬隔離
├─ 為何綁：不錯帳又不被拖垮 = 流水 + 快照 + 硬隔離,一起做
└─ 撐不住的訊號 → 單一活躍帳戶/熱幣種仍是寫熱點

【P3 · 按帳戶/幣種分片 + 熱點分桶】約 50,000 QPS
├─ 定義：資產帳按 userId/asset 分片,熱帳戶/熱幣種分桶承接
├─ 大方塊：帳戶/幣種分片 · 熱點分桶 · 讀聚合
└─ 為何綁：一分片,冪等鍵/餘額聚合/流水收斂到桶

【P4 · 資產錢包獨立 cell + 與非核心徹底隔離】約 100,000+ QPS
├─ 定義：資產核心獨立 cell 部署,與所有非核心流量徹底資源隔離
├─ 大方塊：獨立 cell(庫/服務) · 成交事件專用鏈路 · 跨 cell 對帳
└─ 為何綁：地基級隔離 = 獨立 cell + 專用鏈路 + 跨 cell 對帳,一起做
```
