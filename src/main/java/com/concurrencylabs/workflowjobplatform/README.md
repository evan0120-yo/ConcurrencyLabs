# Workflow / Job Platform · workflowjobplatform

> 一句話定位：把多步驟、有依賴關係的任務編排成 DAG 執行，支援重試、狀態機、暫停與回放。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Cloud / 平台治理
- 壓力模型：任務補償型（核心）

---

## Block 2：為什麼被列上這些

- **任務補償型** → 這是 `backgroundjob` 的進階：不只單一任務，而是「有前後依賴的多步驟流程」的編排與補償。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一個業務流程有很多步，還互相依賴
   步驟1 ──> 步驟2 ──┬──> 步驟3 ──> 步驟5
                     └──> 步驟4 ──┘
   (步驟3、4 要等步驟2 完成才能跑，步驟5 要等 3、4 都完成)
   ▼
[Workflow 平台] 依 DAG 編排
   ├─ 某步失敗 → 重試 / 從該步續跑（不用整個重來）
   └─ 支援暫停、回放、查看每步狀態

────────── 大量長流程同時在跑，某步卡住 ──────────

工程想要：能看到卡在哪、能從失敗那步續跑、能暫停某類流程
```

### 分段解說

- 跟背景任務差在「步驟之間有依賴」（DAG）。
- 失敗要能「從那步續跑」，而不是整個流程重來。
- 長流程要可觀測、可暫停、可回放。

### 痛點分析

- **狀態管理** → 每步狀態要持久、可續跑。
- **失敗補償** → 從失敗步續跑，不整個重來。
- **可控性** → 暫停 / 回放 / 隔離不同流程。
- **真正的痛**：管理「有依賴的多步驟長流程」的狀態，讓任一步失敗都能精準地從斷點續跑而非全部重來。
