# Data Lake Ingestion / 冷熱資料分層 · s74_datalakeingestion

> 一句話定位：把各方資料大量灌進資料湖，處理 schema 演進與小檔合併，熱資料即時查、冷資料延遲入湖。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：ML / 資料平台
- 壓力模型：寫爆型 / ingestion（核心）
- 數據角色：ingestion + schema 演進 + 冷熱分層

---

## Block 2：為什麼被列上這些

- **寫爆型 / ingestion** → 資料湖是各方資料的匯集地，寫入量巨大。
- **數據角色** → schema 演進、小檔合併（compaction）、冷熱分層是資料湖的核心課題。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
各方資料源源不絕灌進資料湖
   交易 / 事件 / log / 第三方 ──> [Data Lake]
   │
   ├─ schema 會變：上游今天多加了一個欄位 → 不能讓舊資料讀不了
   ├─ 小檔問題：串流一直寫小檔 → 太多小檔查詢會很慢 → 要合併
   └─ 冷熱：熱資料要能即時查、冷資料延遲入湖 + 便宜存

────────── 上游改了 schema，或小檔堆成千上萬個 ──────────

資料工程想要：資料進得來、schema 變了也讀得到、查得動
```

### 分段解說

- 資料湖是「先存了再說」，但存法會影響之後查不查得動。
- schema 會隨上游演進，要能向前 / 向後相容。
- 串流寫入產生大量小檔，要定期合併（compaction）。

### 痛點分析

- **讀爆還寫爆？** 寫爆為主。
- **schema 演進** → 上游變欄位不能讓歷史資料失效。
- **小檔地雷** → 小檔堆積會拖垮查詢，要 compaction。
- **真正的痛**：資料湖不只是「灌進來」，還要處理 schema 演進與小檔合併，否則存得下卻查不動。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s74 寫爆 ingestion、schema 演進+小檔 compaction,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：資料灌入寫湖,基本查詢
├─ 大方塊：資料灌入 · 寫入 · 查詢
└─ 撐不住的訊號 → 上游改 schema、串流小檔堆積查不動

【P2 · schema 演進 + 小檔 compaction + 冷熱分層】約 10,000 QPS
├─ 定義：schema 演進相容,小檔定期 compaction,冷熱分層
├─ 大方塊（綁一起）
│   ├─ schema 演進(向前後相容,歷史不失效)
│   ├─ 小檔 compaction(合併大檔,查得動)
│   └─ 冷熱分層(熱即時查/冷延遲便宜存)
├─ 為何綁：存得下又查得動 = schema 演進 + compaction + 冷熱分層,一起做
└─ 撐不住的訊號 → 寫入量再放大、分區/compaction 吞吐不足

【P3 · ingestion 分片 + 並行 compaction + 冷熱歸檔】約 50,000+ QPS
├─ 定義：ingestion 分片,compaction 並行,冷資料歸檔入湖
├─ 大方塊：ingestion 分片 · 並行 compaction · 冷熱歸檔
└─ 為何綁：規模化資料湖 = 分片 + 並行 compaction + 歸檔,一起做
```
