# Distributed Trace Ingestion · s63_traceingestion

> 一句話定位：收集跨服務呼叫鏈的 span，全量太貴要抽樣，但關鍵錯誤鏈路要全保留。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Observability
- 壓力模型：寫爆型 / ingestion（核心）
- 數據角色：ingestion + 抽樣

---

## Block 2：為什麼被列上這些

- **寫爆型 / ingestion** → 每個請求跨多服務會產生多個 span，量體驚人。
- **抽樣角色** → 全存不切實際，抽樣策略是這題的核心。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一個請求穿過 10 個服務
   ──> 產生 10+ 個 span，串成一條 trace
   海量請求 × 每請求多 span = 天量資料
   ▼
[Trace 收集]
   ├─ 全部存？ ──> 太貴，存不起
   ├─ 隨機抽樣 ──> 省，但可能剛好漏掉出錯的那條
   └─ 尾部抽樣（tail-based）──> 先看完再決定：出錯 / 慢的全留，正常的抽樣

────────── 一個罕見錯誤，剛好沒被抽樣到 ──────────

工程想要：正常的可以抽樣，但出錯 / 異常慢的那條一定要留
```

### 分段解說

- Trace 資料量是「請求數 × 每請求 span 數」，極大。
- 隨機抽樣省錢但會漏掉稀有錯誤（最想看的那條）。
- 尾部抽樣：等 trace 完成後再決定留不留，錯的 / 慢的全留。

### 痛點分析

- **能不能抽樣？** 必須，全存不現實。
- **抽樣風險** → 隨機抽樣會漏掉關鍵錯誤鏈。
- **取捨** → 成本 vs 「別漏掉最想看的那條」。
- **真正的痛**：正常 trace 可以大量丟，但「出錯 / 異常慢」的那條恰恰最該留，逼出「尾部抽樣」這種先看再決定的策略。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s63 天量 span、尾部抽樣(錯/慢全留),放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：span 收下組裝 trace,存
├─ 大方塊：span 收集 · trace 組裝 · 存
└─ 撐不住的訊號 → 請求×span 天量、全存太貴、隨機抽樣漏關鍵錯誤

【P2 · 尾部抽樣(錯/慢全留) + 組裝】約 10,000 QPS
├─ 定義：按 traceId 組裝,尾部抽樣(等完成再決定,錯/慢全留正常抽)
├─ 大方塊（綁一起）
│   ├─ 按 traceId 暫存組裝(收齊/逾時)
│   ├─ 尾部抽樣判定(error/慢全留)
│   └─ 正常 trace 按比例抽樣丟棄
├─ 為何綁：別漏關鍵又省成本 = 組裝 + 尾部抽樣,一起做
└─ 撐不住的訊號 → span 量再放大、組裝暫存吃緊

【P3 · 組裝分片 + 抽樣決策分散 + 冷存】約 50,000+ QPS
├─ 定義：按 traceId 分片組裝並行,抽樣決策分散,保留 trace 冷存
├─ 大方塊：組裝分片 · 抽樣決策分散 · 冷存
└─ 為何綁：規模化 trace = 分片組裝 + 抽樣 + 冷存,一起做
```
