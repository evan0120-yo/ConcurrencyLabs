# Checkout 下單鏈路 · s16_checkout

> 一句話定位：按下結帳一次要串算價、扣庫存、建單、付款、通知，核心不能錯、周邊可降級。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：搶資源型（扣庫存）、任務補償型（周邊非同步 / 補償）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 下單要扣庫存，直接踩到 `s15_inventorydeduction` 的爭搶。
- **任務補償型** → 積分、推薦、通知等周邊步驟應非同步、可失敗補償，不該卡住核心。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
按「結帳」，一次要串多個步驟
   使用者 ──> [Checkout]
      │
      ├─ 核心：算價 + 扣庫存 + 建單 + 付款   ← 錯一個都不行
      └─ 周邊：積分 + 推薦 + 通知           ← 可慢、可降級

────────── 大促時跟秒殺連動一起爆 ──────────

使用者想要：下單順、別轉圈圈
營運想要：核心一定成，周邊掛了不能擋下單
```

### 分段解說

- Checkout 是「編排者」，把好幾個系統串成一條龍。
- 關鍵是把「核心」跟「周邊」切開：周邊掛掉不能連累下單。
- 任一步失敗，要嘛回滾、要嘛補償，不能留下半殘訂單。

### 痛點分析

- **核心 vs 周邊** → 必須切開，周邊降級不影響成單。
- **一致性** → 錢與庫存與訂單要對得起來。
- **平均還是尖峰？** 跟大促 / 秒殺連動，會一起爆。
- **真正的痛**：一條鏈路串多個系統，如何在任一環節出錯時仍保住「核心成單」且不留髒資料。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s16 串多系統成一條龍、錢+庫存+訂單要對得起來,編排複雜 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：算價/扣庫存/建單/付款/通知 同步一條龍
├─ 大方塊：核心步驟同步串接 · idem 去重 · 失敗回滾
└─ 撐不住的訊號 → 周邊(通知/推薦)拖慢核心、大促與秒殺連動一起爆

【P2 · 訂單 pending + 周邊非同步】約 10,000 QPS
├─ 定義：核心成單後回 pending,周邊(積分/推薦/通知)進 MQ 非同步
├─ 大方塊（綁一起）
│   ├─ 核心(算價/扣庫存/建單/付款) 保成
│   ├─ [MQ] 周邊事件非同步、可降級
│   └─ 核心失敗 saga 補償(解扣/退款/取消單)
├─ 為何綁：核心/周邊一切開,靠 MQ + saga 保「核心成、周邊掛不連累」
└─ 撐不住的訊號 → order/payment/promo/inventory 耦合成瓶頸

【P3 · 拆分 order / payment / promo / inventory】約 50,000 QPS
├─ 定義：鏈路各環節拆成獨立服務,各自擴展
├─ 大方塊：服務拆分 · 服務間 saga 編排 · 各自獨立擴容
└─ 為何綁：一拆服務,跨服務一致性靠分散式 saga,綁在一起

【P4 · 核心下單最小化 + 周邊全降級】約 100,000+ QPS
├─ 定義：極端尖峰只保「扣庫存+建單+付款」最小核心,其餘全降級
├─ 大方塊：核心路徑瘦身 · 全周邊可降級開關 · 資源優先保核心
└─ 為何綁：保成單 = 核心最小化 + 降級開關 + 資源傾斜,一起做
```
