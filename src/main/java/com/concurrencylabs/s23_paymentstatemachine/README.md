# 交易狀態機系統 · s23_paymentstatemachine

> 一句話定位：一筆交易在建立→付款→成功/失敗/退款之間流轉，靠冪等 key 與狀態機保證不錯亂。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：支付 / 金融
- 壓力模型：錢包帳本型（真錢）、任務補償型（callback / 查單 / 補償）

---

## Block 2：為什麼被列上這些

- **錢包帳本型** → 交易關乎真錢，每個狀態轉移都要正確、可追溯。
- **任務補償型** → 付款結果靠第三方 callback、主動查單、逾時補償收斂，天生要冪等。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
一筆交易的一生
   建立訂單 ──> 待付款 ──> (付款中) ──> 成功 / 失敗
                                   └─> 退款

   收斂靠三種力量：
   ├─ 第三方 callback（可能遲到 / 重送 / 遺失）
   ├─ 主動查單（我去問第三方現在到底怎樣）
   └─ 逾時補償（太久沒結果就走補救）

────────── 同一筆交易的多個訊息可能亂序到達 ──────────
```

### 分段解說

- 這題的本體是「狀態機」：合法的轉移只有幾條，其他都要擋。
- 各種訊息（callback / 查單 / 逾時）可能亂序、重複，狀態機要能容忍。
- 核心轉移要最小、最穩，周邊功能全部解耦出去。

### 痛點分析

- **能不能重複 / 亂序？** 訊息會，但狀態不能被帶錯。
- **一致性** → 涉及真錢，狀態必須正確。
- **收斂機制** → callback 不可靠，要有查單與逾時兜底。
- **真正的痛**：用一個穩固的狀態機，去消化「不可靠、會亂序重複」的外部訊息，且全程不錯帳。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s23 真錢 + 狀態機消化不可靠亂序訊息,不錯帳壓力高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：交易狀態機同步流轉,冪等 key 守合法轉移
├─ 大方塊：狀態機守門 · txnId 冪等 · 每轉移對應記帳
└─ 撐不住的訊號 → callback 亂序/重送/遺失、查單與逾時收斂壓力

【P2 · 訊號非同步收斂 + 查單/逾時補償】約 10,000 QPS
├─ 定義：callback/查單/逾時三力進 MQ 非同步收斂,狀態機吸收亂序
├─ 大方塊（綁一起）
│   ├─ [MQ] 訊號事件非同步處理
│   ├─ 主動查單排程 + 逾時補償
│   └─ 亂序/重複訊號冪等吸收(不倒退不錯帳)
├─ 為何綁：不可靠訊號 → 非同步 + 查單 + 冪等吸收,三件一起收斂
└─ 撐不住的訊號 → 交易量大、單一狀態庫熱點

【P3 · 按 txnId 分片狀態機】約 50,000 QPS
├─ 定義：交易按 txnId 分片,各片獨立狀態流轉,同交易順序保證
├─ 大方塊：txnId 分片路由 · 片內順序保證 · 分片獨立記帳
└─ 為何綁：一分片,同交易訊號要收斂到同片才保順序不錯亂

【P4 · 核心狀態機最小化 + 周邊解耦】約 100,000+ QPS
├─ 定義：核心只留最穩的狀態轉移+記帳,周邊(通知/風控/對帳)全解耦
├─ 大方塊：核心轉移瘦身 · 周邊事件外拋 · 資源優先保核心
└─ 為何綁：保不錯帳 = 核心最小化 + 周邊解耦,一起做
```
