# 熔斷 / 降級平台 · s68_circuitbreakerdegradation

> 一句話定位：當下游變慢 / 出錯時自動熔斷、隔離、降級，把故障關在局部，保住核心流量。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Cloud / 平台治理
- 壓力模型：平台防爆型（核心）

---

## Block 2：為什麼被列上這些

- **平台防爆型** → 這是防止「局部故障蔓延成全站雪崩」的核心防禦機制。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
服務 A 依賴服務 B，B 突然變很慢
   │
   ├─ 沒有熔斷：A 的請求全卡在等 B → A 的執行緒被耗光 → A 也掛
   │            → 依賴 A 的 C 也跟著掛 → 雪崩
   └─ 有熔斷：偵測到 B 不行 → 直接快速失敗 / 給降級結果
              └─ 保住 A 的資源，故障被關在 B 這層

────────── 一個小服務掛掉，差點拖垮全站 ──────────

平台想要：某個依賴壞掉，只損失那塊功能，核心交易照常
```

### 分段解說

- 雪崩的機制：慢 → 資源耗盡 → 自己也掛 → 蔓延。
- 熔斷是「快速失敗」：與其一起等死，不如立刻放棄該依賴。
- 降級是「給次等但可用的結果」，保住核心。

### 痛點分析

- **會不會蔓延？** 會，這正是它要防的。
- **熔斷取捨** → 犧牲局部功能，換全站活著。
- **降級策略** → 要事先定義好「壞了給什麼」。
- **真正的痛**：故障會沿依賴鏈蔓延成雪崩，要靠熔斷把它「快速失敗」關在局部，並用降級守住核心。

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s68 防雪崩、熔斷+隔離+降級,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：呼叫下游基本 timeout/失敗處理
├─ 大方塊：呼叫 · timeout · 錯誤處理
└─ 撐不住的訊號 → 下游變慢資源耗盡、故障沿依賴鏈雪崩

【P2 · 熔斷 + 隔離 bulkhead + 降級策略】約 10,000 QPS
├─ 定義：per依賴熔斷器,隔離池限併發,快速失敗走降級
├─ 大方塊（綁一起）
│   ├─ 熔斷器(CLOSED/OPEN/HALF_OPEN)
│   ├─ bulkhead 隔離池(一依賴不吃光資源)
│   └─ 降級策略(壞了給次等可用結果)
├─ 為何綁：防雪崩 = 熔斷 + 隔離 + 降級,一整套
└─ 撐不住的訊號 → 依賴數多、熔斷狀態需跨實例協調

【P3 · 熔斷狀態分片協調 + 自適應閾值 + 依賴拓撲】約 50,000+ QPS
├─ 定義：熔斷狀態跨實例協調,閾值自適應,依賴拓撲感知降級
├─ 大方塊：狀態分片協調 · 自適應閾值 · 依賴拓撲感知
└─ 為何綁：規模化防禦 = 協調 + 自適應 + 拓撲,一起做
```
