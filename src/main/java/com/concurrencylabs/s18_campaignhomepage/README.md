# 首頁活動頁 · s18_campaignhomepage

> 一句話定位：大促首頁要聚合一堆模組，大家一樣的部分可預生成，個人化部分延遲載入。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：讀爆型（核心）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 大促首頁是全站流量入口，被海量讀取；且一次要聚合多個來源。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
大促首頁，一次要拼很多區塊
   使用者 ──> [首頁]
      ├─ banner / 商品列 / 排行 → 大家都一樣 ──> 可預生成、靜態化
      └─ 個人化推薦             → 因人而異   ──> 延遲載入

────────── 任一區塊的來源慢，就拖累整頁打開 ──────────
```

### 分段解說

- 首頁是「聚合頁」，一個頁面背後是很多資料來源。
- 共用區塊對所有人相同，可以整份預先生成。
- 個人化區塊晚一點出現沒關係，先讓主體秒開。

### 痛點分析

- **讀爆還寫爆？** 讀爆。
- **聚合風險** → 任一來源慢會拖累整頁，要能容忍局部缺失。
- **共用 vs 個人化** → 拆開，共用預生成、個人化延遲。
- **真正的痛**：一個頁面依賴多個後端，如何讓「某個區塊慢 / 掛」不會拖垮整個首頁開啟。

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s18 聚合讀爆、共用可預生成/個人化可延遲,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：API 聚合多模組,共用區塊快取
├─ 大方塊：多來源聚合 · [Redis] 共用區塊快取 · 個人化即時查
└─ 撐不住的訊號 → 任一來源慢拖累整頁、全站入口讀爆

【P2 · BFF cache 聚合結果】約 10,000 QPS
├─ 定義：BFF 層快取整頁聚合結果,局部失敗降級
├─ 大方塊（綁一起）
│   ├─ BFF 聚合並快取整份結果
│   ├─ 個人化區塊延遲載入(主體先回)
│   └─ 局部來源逾時 → 降級占位(不阻整頁)
├─ 為何綁：聚合快取 + 延遲載入 + 降級是「一頁多來源」的一整套
└─ 撐不住的訊號 → 尖峰整頁生成仍重

【P3 · 頁面資料預生成 + 首頁靜態化】約 50,000+ QPS
├─ 定義：共用主體整份預生成靜態化,個人化區塊延遲補
├─ 大方塊：共用區塊預生成快照 · 首頁靜態化 · 個人化延遲載入
└─ 為何綁：大家一樣的主體先靜態化、因人而異的延遲,一起做
```
