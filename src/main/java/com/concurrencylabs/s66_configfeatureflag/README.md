# Config Center / Feature Flag · s66_configfeatureflag

> 一句話定位：集中管理設定與功能開關，變更要能快速下發、灰度生效，且別讓所有服務同時來打。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Cloud / 平台治理
- 壓力模型：讀爆型（大量服務拉 config）、平台防爆型（避免同時打爆 config 中心）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 大量服務實例都要讀 config，且變更要能推到每個實例。
- **平台防爆型** → 若所有服務同時來拉 / 重連，會把 config 中心打爆（驚群）。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
成百上千個服務實例都依賴中央設定 / 開關
   實例們 ──讀──> [Config 中心]
   ▼
   本地快取一份 + 監聽變更
   └─ 改了某開關 ──> 推給所有實例（灰度：先 1%、再全量）

────────── config 中心重啟，所有實例同時重連 ──────────

平台想要：別讓「驚群效應」把 config 中心自己打死
```

### 分段解說

- Config 要讀得快（本地快取），又要變更能即時推。
- Feature Flag 要能灰度：先小比例、再全量，出事可秒回退。
- 最大風險是「驚群」：所有服務同時來拉會打爆中心。

### 痛點分析

- **讀爆還寫爆？** 讀爆（大量實例拉），寫少（偶爾改）。
- **能不能灰度？** 開關要能按比例 / 按租戶生效。
- **驚群風險** → 避免所有實例同時打中心。
- **真正的痛**：config 是全站依賴的單點，變更要能快速安全下發，同時要防「所有服務同時來打」把中心自己弄垮。
