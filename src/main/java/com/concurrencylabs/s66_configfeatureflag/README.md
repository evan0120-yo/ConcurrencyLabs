# Config Center / Feature Flag · s66_configfeatureflag

> 一句話定位：集中管理設定與功能開關，變更要能快速下發、灰度生效，且別讓所有服務同時來打。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：Cloud / 平台治理
- 壓力模型：讀爆型（大量服務拉 config）、平台防爆型（避免同時打爆 config 中心）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 大量服務實例都要讀 config，且變更要能推到每個實例。
- **平台防爆型** → 若所有服務同時來拉 / 重連，會把 config 中心打爆（驚群）。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
成百上千個服務實例都依賴中央設定 / 開關
   實例們 ──讀──> [Config 中心]
   ▼
   本地快取一份 + 監聽變更
   └─ 改了某開關 ──> 推給所有實例（灰度：先 1%、再全量）

────────── config 中心重啟，所有實例同時重連 ──────────

平台想要：別讓「驚群效應」把 config 中心自己打死
```

### 分段解說

- Config 要讀得快（本地快取），又要變更能即時推。
- Feature Flag 要能灰度：先小比例、再全量，出事可秒回退。
- 最大風險是「驚群」：所有服務同時來拉會打爆中心。

### 痛點分析

- **讀爆還寫爆？** 讀爆（大量實例拉），寫少（偶爾改）。
- **能不能灰度？** 開關要能按比例 / 按租戶生效。
- **驚群風險** → 避免所有實例同時打中心。
- **真正的痛**：config 是全站依賴的單點，變更要能快速安全下發，同時要防「所有服務同時來打」把中心自己弄垮。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s66 讀爆 + 灰度 + 驚群防護,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：實例拉 config,變更推送
├─ 大方塊：config 儲存 · 拉取 · 變更通知
└─ 撐不住的訊號 → 大量實例同時拉、config 中心重啟驚群

【P2 · 本地快取 + 灰度下發 + 驚群防護】約 10,000 QPS
├─ 定義：實例本地快取+監聽變更,灰度下發秒回退,驚群退避
├─ 大方塊（綁一起）
│   ├─ 本地快取 + 監聽(不每次打中心)
│   ├─ 灰度下發(1%→全量) + 秒回退
│   └─ 驚群防護(隨機退避/多層)
├─ 為何綁：全站單點安全下發 = 本地快取 + 灰度 + 驚群防護,一起做
└─ 撐不住的訊號 → 實例數再放大、變更推送扇出大

【P3 · 多層快取分發 + 推送分片 + 就近節點】約 50,000+ QPS
├─ 定義：config 多層快取分發,變更推送分片,就近邊緣節點
├─ 大方塊：多層快取分發 · 推送分片 · 就近節點
└─ 為何綁：規模化下發 = 多層快取 + 推送分片,一起做
```
