# 優惠券搶券系統 · s14_couponclaim

> 一句話定位：大促整點放券，遠超券量的人同時搶，一人限領、不能超發。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：搶資源型（核心）、讀爆型（開搶前預熱）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 券量固定、需求遠大於供給，是搶資源三兄弟之一（另兩個是 `s13_flashsale`、`s02_redpacketrain`）。
- **讀爆型** → 開搶前使用者狂刷頁面等倒數，讀壓力先爆。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
大促整點放券，10 萬人搶 1 萬張
   │
   ▼
[券池] 1 萬張
   ├─ (搶到) ──> 券入帳，且「這人不能再領第二張」
   └─ (領完) ──> 秒回「已搶光」

────────── 同一秒湧入、且要判資格 ──────────

使用者想要：手速快就搶到、別讓我重複失敗
營運想要：券數量固定、一人限領、不能超發
```

### 分段解說

- 跟秒殺很像，但多了「一人限領」的個人維度判斷。
- 有時還要判「資格」（新客券 / 會員券），開搶前要先備好。
- 超發的是行銷預算，等於實質成本。

### 痛點分析

- **能不能超發？** 不行，超發 = 行銷成本失控。
- **一人限領** → 要防同一人重複領。
- **平均還是尖峰？** 極端尖峰。
- **真正的痛**：在極端尖峰下同時守住「總量不超發」與「單人不超領」兩個維度。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s14 要同時守「總量不超發 + 單人不超領」兩維度,正確性壓力高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：Redis 扣券量 + 用戶 set 防重領,售完快速失敗
├─ 大方塊：[Redis] 原子扣券量 · 單人 set 防重領 · [DB] 發券
└─ 撐不住的訊號 → 開搶前讀爆、開搶瞬間寫爆,單券池熱點

【P2 · MQ 非同步發券】約 10,000 QPS
├─ 定義：搶到先在 Redis 占位,發券入帳走 MQ 非同步
├─ 大方塊（綁一起）
│   ├─ [Redis] 扣量 + 防重占位(同步、快)
│   ├─ [MQ] 發券入帳非同步
│   └─ 占位↔發券對帳(占了就一定發)
├─ 為何綁：占位/發券一拆,靠對帳保「占了必發、發了不超」
└─ 撐不住的訊號 → 券池 + 資格判定成為瓶頸

【P3 · 券池分 shard + 用戶資格預熱】約 50,000 QPS
├─ 定義：券池分 shard 打散,用戶資格開搶前預熱到快取
├─ 大方塊：券池分 shard · 跨片扣量 · 資格預熱(免臨場查)
└─ 為何綁：分片後,總量控制 + 單人防重 + 資格都要收斂到片/預熱

【P4 · 大促券分批釋放 + 熱門券獨立 cluster】約 100,000+ QPS
├─ 定義：分批釋放平滑洪峰,熱門券獨立叢集
├─ 大方塊：分批釋放策略 · 熱門券獨立部署 · 冷券共用池
└─ 為何綁：熱冷分治 = 分批 + 獨立叢集,一起做
```
