# 優惠券搶券系統 · s14_couponclaim

> 一句話定位：大促整點放券，遠超券量的人同時搶，一人限領、不能超發。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：搶資源型（核心）、讀爆型（開搶前預熱）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 券量固定、需求遠大於供給，是搶資源三兄弟之一（另兩個是 `s13_flashsale`、`s02_redpacketrain`）。
- **讀爆型** → 開搶前使用者狂刷頁面等倒數，讀壓力先爆。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
大促整點放券，10 萬人搶 1 萬張
   │
   ▼
[券池] 1 萬張
   ├─ (搶到) ──> 券入帳，且「這人不能再領第二張」
   └─ (領完) ──> 秒回「已搶光」

────────── 同一秒湧入、且要判資格 ──────────

使用者想要：手速快就搶到、別讓我重複失敗
營運想要：券數量固定、一人限領、不能超發
```

### 分段解說

- 跟秒殺很像，但多了「一人限領」的個人維度判斷。
- 有時還要判「資格」（新客券 / 會員券），開搶前要先備好。
- 超發的是行銷預算，等於實質成本。

### 痛點分析

- **能不能超發？** 不行，超發 = 行銷成本失控。
- **一人限領** → 要防同一人重複領。
- **平均還是尖峰？** 極端尖峰。
- **真正的痛**：在極端尖峰下同時守住「總量不超發」與「單人不超領」兩個維度。
