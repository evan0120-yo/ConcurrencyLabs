# 庫存扣減中心 · s15_inventorydeduction

> 一句話定位：多個下單來源同時扣同一商品庫存，不能超賣、不能扣成負數，失敗要回補。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：搶資源型（核心）、任務補償型（失敗回補）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 同一商品庫存被多來源併發爭搶，是搶資源的底層元件。
- **任務補償型** → 下單失敗 / 取消時，被扣的庫存要能可靠回補。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
多個下單來源同時要扣同一商品庫存
   訂單 A ──扣 2──> [庫存中心] <──扣 1── 訂單 B
   ▼
   ├─ (夠)   ──> 扣成功
   └─ (不夠) ──> 擋下，且不能扣成負數

────────── 下單失敗 / 取消 ──────────

   └──> 要把先前扣掉的庫存補回來（否則庫存憑空消失）
```

### 分段解說

- 庫存是被許多情境共用的核心資源（秒殺、一般下單都打它）。
- 難點一：併發扣同一 key 不能超賣。
- 難點二：扣了之後若下單沒成，要能補回來。

### 痛點分析

- **能不能超賣 / 扣負？** 都不行，庫存等於錢。
- **失敗要不要回補？** 要，否則庫存會憑空蒸發。
- **有沒有熱點？** 爆款商品是熱 key。
- **真正的痛**：高併發扣減的正確性，加上「扣了但單沒成」時的回補一致性。

---

## 容量平台演進地圖（P1 → P4）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s15 庫存等於錢,不超賣不扣負 + 失敗回補,正確性壓力高 → 共 4 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：下單同步扣庫存(原子條件更新),失敗回補
├─ 大方塊：原子條件扣減(不超賣) · orderNo 冪等 · 回補
└─ 撐不住的訊號 → 爆款商品單 key 熱點、同步扣減拖慢下單

【P2 · Redis 預扣 + DB 最終實扣】約 10,000 QPS
├─ 定義：Redis 原子預扣扛併發,DB 非同步最終實扣對帳
├─ 大方塊（綁一起）
│   ├─ [Redis] 預扣(快、扛併發)
│   ├─ [DB] 最終實扣(批量落地)
│   └─ 預扣↔實扣對帳 + 回補一致
├─ 為何綁：預扣/實扣一拆,靠對帳保 Redis 與 DB 庫存最終一致
└─ 撐不住的訊號 → 單一爆款 key 仍是預扣熱點

【P3 · 分倉 / 分片庫存 + 回補補償】約 50,000 QPS
├─ 定義：庫存按倉/分片拆,各片獨立扣減,跨片回補補償
├─ 大方塊：分倉/分片路由 · 片內原子扣 · 跨片回補對帳
└─ 為何綁：一分片,冪等鍵/回補配對收斂片內

【P4 · 庫存 token 化,避免全局鎖】約 100,000+ QPS
├─ 定義：庫存預切成 token,取 token = 扣減,徹底避開熱 key 鎖
├─ 大方塊：庫存 token 池 · 取 token 扣減 · token 回收(回補)
└─ 為何綁：token 化 = 資料結構 + 扣減 + 回收整套換掉
```
