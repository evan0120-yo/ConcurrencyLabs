# 庫存扣減中心 · s15_inventorydeduction

> 一句話定位：多個下單來源同時扣同一商品庫存，不能超賣、不能扣成負數，失敗要回補。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：搶資源型（核心）、任務補償型（失敗回補）

---

## Block 2：為什麼被列上這些

- **搶資源型** → 同一商品庫存被多來源併發爭搶，是搶資源的底層元件。
- **任務補償型** → 下單失敗 / 取消時，被扣的庫存要能可靠回補。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
多個下單來源同時要扣同一商品庫存
   訂單 A ──扣 2──> [庫存中心] <──扣 1── 訂單 B
   ▼
   ├─ (夠)   ──> 扣成功
   └─ (不夠) ──> 擋下，且不能扣成負數

────────── 下單失敗 / 取消 ──────────

   └──> 要把先前扣掉的庫存補回來（否則庫存憑空消失）
```

### 分段解說

- 庫存是被許多情境共用的核心資源（秒殺、一般下單都打它）。
- 難點一：併發扣同一 key 不能超賣。
- 難點二：扣了之後若下單沒成，要能補回來。

### 痛點分析

- **能不能超賣 / 扣負？** 都不行，庫存等於錢。
- **失敗要不要回補？** 要，否則庫存會憑空蒸發。
- **有沒有熱點？** 爆款商品是熱 key。
- **真正的痛**：高併發扣減的正確性，加上「扣了但單沒成」時的回補一致性。
