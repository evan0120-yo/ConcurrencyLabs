# 購物車系統 · s20_shoppingcart

> 一句話定位：使用者高頻加 / 改 / 刪購物車、跨裝置同步，展示可弱一致，但結帳前一刻必須校正。

---

## Block 1：這個情境同時符合哪些分類

- 業務 domain：電商 / 促銷
- 壓力模型：讀爆型（展示）、最終一致可容忍（跨裝置同步）

---

## Block 2：為什麼被列上這些

- **讀爆型** → 購物車頁被頻繁查看，讀多於寫。
- **最終一致可容忍** → 跨裝置同步、展示金額允許短暫不一致，結帳前再校正即可。

---

## Block 3：User Story + 痛點分析（不涉及技術）

### User Story

```text
使用者狂加 / 改 / 刪購物車，還會換裝置
   手機加一個 ──> [購物車] <── 電腦刪一個
   ▼
   展示：可弱一致（晚幾秒同步 OK）

────────── 但按下「結帳」的那一刻 ──────────

   └──> 必須即時校正：價格變了？庫存沒了？商品下架了？
```

### 分段解說

- 平常瀏覽 / 編輯購物車，容忍短暫不一致。
- 真正嚴格的點只有一個：結帳前一刻要用最新的價格與庫存校正。
- 高頻小寫入（加減數量）是常態流量。

### 痛點分析

- **能不能弱一致？** 展示可以，結帳不行。
- **關鍵校正點** → 結帳前必須抓最新價格 / 庫存 / 上下架。
- **跨裝置** → 同一使用者多端要收斂到一致。
- **真正的痛**：平常放鬆一致性換效能，但要精準地在「結帳前一刻」把關，避免用舊價 / 賣缺貨。

---

## 容量平台演進地圖（P1 → P3）

> 起點固定假設 QPS 3,000+。每跳一個平台 = 一次架構改造（一組必須一起改的大方塊）。
> s20 展示可弱一致、只在結帳前強校正,放寬 → 共 3 個平台。

```text
【P1 · 基線】約 3,000 QPS
├─ 定義：DB 記購物車 + Redis cache,結帳前校正
├─ 大方塊：[DB] 車持久化 · [Redis] 展示快取 · 結帳校正
└─ 撐不住的訊號 → 高頻加減小寫入、跨裝置同步、展示讀多

【P2 · 短時間合併寫入】約 10,000 QPS
├─ 定義：高頻小寫入短時合併,降低寫放大
├─ 大方塊（綁一起）
│   ├─ 加減數量短時合併批量落地
│   └─ 展示走 cache(弱一致)
├─ 為何綁：合併寫 + 弱一致展示,一起降寫壓
└─ 撐不住的訊號 → 用戶量大、單庫讀寫混雜

【P3 · userId 分區 + 讀寫分離（結帳前強校正）】約 50,000+ QPS
├─ 定義：車按 userId 分區、讀寫分離;結帳前對最新價/庫存/上下架強校正
├─ 大方塊：userId 分區 · 讀寫分離 · 結帳前校正(最新真相把關)
└─ 為何綁：日常放鬆(分區+讀寫分離)、結帳前收緊(強校正),一起做
```
