# SA.md — s39_deviceheartbeatingestion（IoT 裝置心跳與資料上報）

> 事實文件（SASD）· 結構化分析。描述「心跳上報要做什麼」的**業務行為**，平台無關（P1..Pn 共用；各平台怎麼扛量在各自 SD）。
> 特性：幾乎**純寫爆**（射後不理），量由裝置數 × 頻率決定；核心不對稱 —— 大量「正常無聊資料」可丟可延遲，少數「掉線 / 異常」不可丟、要秒級。**分流是關鍵**。

## 1. Context Diagram

```text
[大量裝置] --(心跳: deviceId, ts, metrics)--> (上報入口) --(ack 收下)--> [裝置]（射後不理）

(上報入口) --(緩衝削峰)------> [MQ]
(上報入口) --(正常資料批量入庫)--> [時序庫 / 資料湖]（冷，可延遲）
(上報入口) --(異常事件即時)---> [告警 / metrics]（熱，秒級）
```

## 2. DFD（精簡 box flow）

```text
[心跳湧入] --> [上報入口]（快速 ack，不同步處理）
   ▼
[緩衝削峰] --> [MQ]（整點對齊 / 斷網補傳的洪峰在此被吸收）
   ▼
[分流]
   ├─【正常資料】批量入時序庫 / 資料湖（可丟少數、可延遲幾分鐘）
   └─【異常偵測】掉線 / 數值越界 → 即時告警（秒級，不可丟）
```

## 3. Process Spec（行為基準，decision table）

```text
[接收]
└─ 快速 ack（射後不理）；洪峰進緩衝，不在請求路徑同步處理

[正常資料]（放寬）
├─ 可丟少數（心跳丟幾筆無妨）
└─ 可延遲入庫（幾分鐘可接受）

[異常事件]（不可放寬，核心）
├─ 掉線（超時未回報）/ 數值越界 → 秒級告警
└─ 絕不可被大量正常資料淹沒 / 延遲

[分流]（核心不變量）
└─ 「大量無聊」與「少數關鍵」必須分兩條路：正常走冷批量、異常走熱即時
```

## 4. State Transition（一台裝置的線上狀態）

```text
【ONLINE】──(超時未心跳)──> 【OFFLINE（告警）】──(恢復心跳)──> ONLINE
          數值越界 → 觸發 ANOMALY 事件（不改線上狀態，獨立告警）
```

## 5. Data Dictionary（邏輯欄位；實體 DDL 在各平台 SD）

```text
heartbeat（心跳資料，寫入時序庫 / 湖）
├─ deviceId / ts / metrics（溫度 / 電量 / …）
└─（正常資料，量大、單筆價值低）

device_status（裝置線上狀態）
├─ deviceId(PK) / lastSeenAt / status（ONLINE|OFFLINE）

alert_event（異常事件，即時通道）
├─ eventId(PK) / deviceId / type（OFFLINE|ANOMALY）/ detail / ts
```
